01 · Why std::execution?

On this page

Say you’re writing a C++ network service. You need to read a request asynchronously, process the data, then send back a response. How would you do it?

Callbacks

The most straightforward approach is callbacks, a pattern pretty common while you are using Qt or libuv:

cpp
void fetch_data(std::function<void(std::string)> on_success,
                std::function<void(std::error_code)> on_error);

But when you need to chain multiple async operations, callbacks turn into “callback hell”:

cpp
fetch_data(
  [](std::string data) {
    process(data,
      [](int result) {
        // keep nesting...
      },
      [](std::error_code ec) { /* error handling */ });
  },
  [](std::error_code ec) { /* error handling */ });

What’s wrong with this picture?

This code has poor readability with too many levels of nesting. A more serious problem than visual indentation is that these callbacks are often semantically ambiguous — it’s hard to tell whether they are error handling routines or happy path business logic.

Due to the lack of a unified interface specification and lifecycle convention, a frustrating problem is that you need to bridge between different async interfaces (e.g., async gRPC, Qt, libuv, etc.) and carefully ensure there are no dangling callback objects.

callback is bad, what about std::future?

std::future

cpp
std::future<int> f = std::async(std::launch::async, [] { return 42; });
int result = f.get(); // blocks

std::future is a loosely coupled asynchronous result delivery mechanism. It uses a push design where the producer is unaware of the consumer. This brings disadvantages such as poor back pressure control. To support consumption at any time, the future/promise design relies on heap allocation. Since its design goal only covers result delivery, it completely lacks the cancellation and structured concurrency features that are crucial in asynchronous programming.

Can C++20 coroutines enough for C++ dev communities to develope async

C++20 Coroutines

cpp
generator<int> gen() { /* ... */ }
task<int> async_fn() { co_return 42; }

Coroutines are just a language mechanism — they define how co_await, co_yield, co_return are translated, but they don’t provide a standard asynchronous runtime.

This means every library (ASIO, libunifex, cppcoro) defines its own task type, and they’re incompatible. Pick one library and you’re locked into its ecosystem. What’s more, coroutines don’t address execution context scheduling — “which thread does this code run on?” Coroutines can’t answer that.

What’s worse, in my opinion C++ coroutines further break the memory safety of C++. C++ already has a poor reputation for memory safety compared to newer languages, but C++20 coroutines make this problem even more serious. You may have heard about lifetime issues with C++20 coroutine lambdas. For example, misusing C++ coroutine lambda capture can silently cause use-after-free. 1

Coroutine promises are easily misimplemented or misused, leading to use-after-free. 2 And these problems are difficult to detect even with sanitizers and static analysis.

The C++26 std::execution Solution

So is there a solution that lets you compose async operations, doesn’t block threads, supports cancellation, and provides standard execution context switching?

That’s exactly what P2300 (std::execution) is for. It introduces a standardized asynchronous operation model built around the Sender / Receiver model.

The Big Picture

Instead of immeditately executing, std::execution builds a description of the work:

Scheduler (execution resource) → Sender (description of async work) → Receiver (result handling)
  • Sender: a description of an async operation that hasn’t started yet (lazy, composable)
  • Receiver: a collection of three completion signal handlers
  • Scheduler: an abstract execution resource (thread pool, GPU, io_uring…)
  • Operation State: the actual state object created after connecting a sender + receiver

Why it matters

Unlike the loose relationship of future/promise, in the sender/receiver model the producer knows at compile time how it will be consumed. This is important for lifecycle management.

  1. Lazy Evaluation & Composability — Like C++20 Ranges, you build pipelines using operator|. No work is done, and no threads are spawned until you explicitly call start().
  2. Structured Concurrency — Child operations are guaranteed to complete before their parent scope ends. No more silent task leaks or dangling callbacks.
  3. Type Safety — compile-time validation via completion_signatures
  4. Customizability — custom behavior via tag_invoke and domain mechanisms
  5. Cancellation Support — unified stop token mechanism

  F --> G["operation_state"]

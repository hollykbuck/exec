The Pain Points of Asynchronous Programming

C++ has long lacked a unified asynchronous programming model. Developers face a variety of incompatible choices:

1. Callbacks

cpp
void fetch_data(std::function<void(std::string)> on_success,
                std::function<void(std::error_code)> on_error);
// Callback hell: nested, scattered error handling, hard to compose
fetch_data(
  [](std::string data) {
    process(data,
      [](int result) {
        // continue nesting...
      },
      [](std::error_code ec) { /* error handling */ });
  },
  [](std::error_code ec) { /* error handling */ });
  • Problem: Too much nesting, scattered error handling paths, not composable, difficult resource management.

2. std::future / std::promise

cpp
std::future<int> f = std::async(std::launch::async, [] { return 42; });
int result = f.get(); // blocking
  • Problem: std::future is not composable (no .then()), .get() blocks, single-use, cannot be cancelled, lacks structured concurrency.

3. Coroutines (C++20 Coroutines)

cpp
generator<int> gen() { /* ... */ }
task<int> async_fn() { co_return 42; }
  • Problem: C++20 only defines the language mechanism for coroutines, without a standardized asynchronous runtime. Each library (ASIO, libunifex, cppcoro) defines its own task type, which are incompatible. Coroutines themselves do not address the scheduling of execution contexts.

The P2300 Solution

P2300 (“std::execution”) introduces a standardized asynchronous operation model for C++26, with the core idea:

Unified Abstraction: Sender / Receiver Model

Scheduler (execution resource) → Sender (description of async work) → Receiver (result handling)
  • Sender: A description of an asynchronous operation that may not have started yet (lazy, composable)
  • Receiver: Aggregation of three completion signal handlers
  • Scheduler: Abstract execution resource (thread pool, GPU, io_uring…)
  • Operation State: The actual state object created after connecting a sender + receiver

Key Design Principles

  1. Lazy Evaluation: Building a pipeline does no work; only start() actually drives execution
  2. Composability: Senders are combined via operator|, similar to Ranges
  3. Structured Concurrency: Ensures all child operations complete before the parent, preventing task leaks
  4. Type Safety: Compile-time validation of types via completion_signatures
  5. Customizability: Supports custom behavior via tag_invoke and domain mechanisms
  6. Cancellation Support: Unified stop token mechanism

The Life of a Pipeline

graph TD
  A["scheduler"] --> B["schedule()"]
  B --> C["sender"]
  C --> D["operator| / then()"]
  D --> E["sender adaptor"]
  E --> F["connect(receiver)"]
  F --> G["operation_state"]
  G --> H["start()"]
  H --> I["(async execution)"]
  I --> J["set_value / set_error / set_stopped"]
  J --> K["receiver's three handlers"]

This is the foundational framework for all subsequent tutorials.

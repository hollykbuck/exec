02 · Three Core Abstractions — receiver, sender, scheduler, operation_state Overview

Standard Section: [exec.async.ops]
stdexec Reference: include/stdexec/execution.hpp, include/stdexec/concepts.hpp
Related Tests: tests/test_basics.cpp · tests/test_receiver.cpp

Overview

std::execution is built on four core abstractions. This section provides their high-level definitions and relationships.

1. Scheduler — Abstraction of Execution Resources

Standard definition (exec.tex:310):

A scheduler is an abstraction of an execution resource with a uniform, generic interface for scheduling work onto that resource.

cpp
// Standard concept (simplified)
template <class S>
concept scheduler =
  requires(S s, receiver auto r) {
    { schedule(s) } -> sender;
    { get_forward_progress_guarantee(s) } -> forward_progress_guarantee;
  };

The scheduler’s responsibilities are:

  • Represent an execution resource (thread pool, GPU, io_uring, current thread…)
  • Generate a sender via schedule() that will execute on that resource
  • Provide get_forward_progress_guarantee() to query concurrency guarantees (concurrent/parallel/weakly parallel)

Common schedulers in stdexec:

cpp
exec::static_thread_pool ctx{8};
scheduler auto sch = ctx.get_scheduler();    // thread pool scheduler

exec::inline_scheduler{}                      // executes immediately on the current thread
nvexec::stream_scheduler{}                    // GPU stream scheduler

2. Sender — Factory/Description of Async Operations

Standard definition (exec.tex:290):

A sender is a factory for one or more asynchronous operations.

A sender is a lazy, composable description of asynchronous work. It does not represent an operation in progress, but describes “what will happen when connected and started.”

cpp
// Standard concept (simplified)
template <class S>
concept sender = /* ... */;

template <class S, class E>
concept sender_in = /* sender can determine completion signatures in environment E */;

Three categories of senders:

CategoryDescriptionExample
Sender FactoryCreates a sender from non-sender inputschedule(sch), just(42), read_env
Sender AdaptorTakes sender(s) and returns a new senderthen(sndr, f), when_all(s1, s2), let_value
Sender ConsumerConsumes a sender, returns non-sendersync_wait(sndr), spawn(sndr)

Key Characteristics

  • Lazy: Building a pipeline does no work
  • Pipeable: sndr | then(f) | upon_error(h)
  • Completion Signatures: Compile-time description of possible completion signal types
  • Each sender has an environment: queryable via get_env(sndr)

3. Receiver — Consumer of Completion Signals

Standard definition (exec.tex:241):

A receiver is an aggregation of three handlers for the three asynchronous completion dispositions.

cpp
template <class R>
concept receiver =
  requires(R r) {
    { get_env(r) } -> queryable;
  } /* + destructible */;

Each receiver must implement three completion functions:

cpp
struct my_receiver {
  using receiver_concept = stdexec::receiver_tag;

  // Success: receives a set of values
  void set_value(int x, double y) & noexcept { /* handle success */ }

  // Failure: receives an exception
  void set_error(std::exception_ptr e) & noexcept { /* handle error */ }

  // Cancellation: no arguments
  void set_stopped() & noexcept { /* handle cancellation */ }

  // Provide environment queries
  auto get_env() const & noexcept {
    return stdexec::prop{stdexec::get_stop_token, stdexec::never_stop_token{}};
  }
};

Three variants of receivers:

ConceptRequirement
receiverOnly requires get_env(r)
receiver_of<R, Completions>Requires ability to handle a specified set of completion signatures
inlinable_receiver<R>Allows inline completion on the start() call stack

4. Operation State — Running Async Operation

Standard definition (exec.tex:230):

An asynchronous operation has associated state known as its operation state.

cpp
template <class O>
concept operation_state =
  std::destructible<O> &&
  requires(O& o) {
    { start(o) } noexcept;
  };

Operation state lifecycle:

graph TD
  A["sender + receiver"]
  A --> B["connect()"]
  B --> C["operation_state"]
  C --> D["Contains copies of sender and receiver"]
  C --> E["start()"]
  E --> F["[async execution]"]
  F --> G["Uncancellable until completion"]
  F --> H["set_value / set_error / set_stopped"]
  H --> I["operation_state invalidated"]

Important lifecycle rules (exec.tex:270-287):

  • The lifetime of an async operation begins with start() and ends when a completion function is called
  • start() must be called after connect() and at most once
  • During execution of a completion function, the operation_state may be destroyed
  • Accessing operation_state after it has been destroyed is undefined behavior

Relationship of the Four Abstractions

graph TD
  A["Scheduler (where)"]
  A --> B["schedule()"]
  B --> C["Sender (what)"]
  C --> D["connect(receiver)"]
  D --> E["Operation State (actual work)"]
  E --> F["start()"]
  F --> G["[async execution]"]
  G --> H["completion function"]
  H --> I["Receiver (result handling)"]

Analogy:

AbstractionAnalogy
SchedulerKitchen (execution resource)
schedule()“Cook here” request
SenderRecipe (lazy description)
connect(receiver)Prepare ingredients and chef
start()Turn on the stove
set_value/set_error/set_stoppedDish served/burnt/fire off

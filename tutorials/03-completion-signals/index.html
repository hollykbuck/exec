03 · Three Completion Signals — set_value / set_error / set_stopped

Standard Section: [exec.set.value] [exec.set.error] [exec.set.stopped]
stdexec Reference: include/stdexec/__detail/__senders.hpp
Related Tests: tests/test_basics.cpp

How many ways can an async operation end?

Think about real life. You order food delivery. Only three things can happen:

  1. The food arrives — you eat (success)
  2. The restaurant is out — you get a refund (failure)
  3. You cancel the order — nothing happens (cancellation)

Async operations are the same. No matter how complex, the outcome always falls into one of these three categories.

The Three Completion Signals

std::execution defines three functions that correspond to these three outcomes:

Completion FunctionMeaningParametersExample
set_value(rcvr, args...)Successful completionAny number of result valuesHTTP response, computation result
set_error(rcvr, err)Failed completionOne error valueNetwork timeout, file not found
set_stopped(rcvr)Cancelled completionNoneUser cancellation, scope destruction

Each completion function takes a receiver as its first parameter:

cpp
namespace std::execution {
  inline constexpr unspecified set_value = /* see [exec.set.value] */;
  inline constexpr unspecified set_error = /* see [exec.set.error] */;
  inline constexpr unspecified set_stopped = /* see [exec.set.stopped] */;
}

Why Exactly Three?

You might ask: what’s the difference between an error and cancellation? Aren’t they both “didn’t get the result”?

The difference is intent.

  • set_error means something went wrong — the network is down, the file doesn’t exist, parsing failed. It’s an意外 that the caller might want to retry or log
  • set_stopped means someone asked to stop — not a bug, but an intentional decision to halt. The caller typically does nothing

  A --> C["Failure (carries error)"]
  A --> D["Cancellation (empty)"]

These three signals form a complete classification of async operation outcomes. Every async operation must end in one of them.

How Receivers Receive Signals

Every Receiver must be able to handle these three signals. Here’s noop_receiver from stdexec (examples/scope.cpp):

cpp
struct noop_receiver {
  using receiver_concept = receiver_tag;

  template <class... _As>
  void set_value(_As&&...) noexcept {}

  void set_error(std::exception_ptr) noexcept {}

  void set_stopped() noexcept {}

  auto get_env() const & noexcept {
    return stdexec::prop{get_stop_token, stdexec::never_stop_token{}};
  }
};

In the code above, all three signal handlers are empty — “noop” means “no operation.” In real code, you’d put actual logic here: pass the result to the next stage, log the error, or clean up on cancellation.

The exact signature of each handler is determined by the sender’s completion_signatures, which we’ll cover in Chapter 12.

Senders That Send Signals

Some senders fire a completion signal immediately:

cpp
// Immediately sends set_value(42)
sender auto s1 = stdexec::just(42);

// Immediately sends set_error()
sender auto s2 = stdexec::just_error(std::make_exception_ptr(std::runtime_error{"oops"}));

// Immediately sends set_stopped()
sender auto s3 = stdexec::just_stopped();

In the code above, just, just_error, and just_stopped are the simplest Sender Factories. They don’t wait for anything — they produce their completion signal right away.

Other senders wait until an async operation completes:

cpp
exec::static_thread_pool ctx{8};
auto sch = ctx.get_scheduler();

// Sends set_value(...) after executing on the thread pool
sender auto s = schedule(sch) | then([] { return 42; });

Here, schedule(sch) creates a Sender that runs on the thread pool. When execution finishes, it calls set_value with the result. then wraps the value 42 into a new set_value and passes it downstream.

Signal Transformation in Pipelines

Sender Adaptors can transform signals as they arrive:

cpp
sender auto s = just(42)                               // set_value(42)
              | then([](int x) { return x + 1; })      // set_value(43)
              | upon_error([](std::exception_ptr e) {   // catches set_error
                    return -1;
                })
              | upon_stopped([] {                       // catches set_stopped
                    return 0;
                });

Let’s trace what happens in this pipeline:

  1. just(42) fires set_value(42)
  2. then(f) receives set_value(42), computes 42 + 1, fires set_value(43)
  3. upon_error(h) — if a set_error arrives upstream, it catches it and converts it to set_value(-1)
  4. upon_stopped(h) — if a set_stopped arrives, it catches it and converts it to set_value(0)

You might ask: why convert errors and cancellation into normal values? Because downstream might only care about success. These adaptors let you normalize all completion signals into a single path.

The three completion signals are the only data channel in a sender pipeline. Any information — successful values, errors, cancellation — must be conveyed through one of these three. Understanding this is the key to std::execution.

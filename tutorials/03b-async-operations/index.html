03b · Formal Model of Asynchronous Operations

Beyond the Signals

Chapter 03 introduced the three completion signals, Chapter 02 the four abstractions. But the standard defines a formal model underneath all of it — the vocabulary that gives sender/receiver its precise semantics.

Execution Resource

An execution resource manages a set of execution agents (threads, GPU cores, IO completion ports). It is either valid or invalid.

Execution Resource
  ├── Valid   -> can execute work
  └── Invalid -> destroyed, all derived objects are unusable

Examples: the currently active thread, a system thread pool, a CUDA stream.

Asynchronous Operation

An asynchronous operation is a unit of program execution with five properties:

  • Created explicitly (connecting a sender + receiver)
  • Started at most once (calling start() on the operation state)
  • Completes exactly once in one of three dispositions
  • Can migrate — may run on a different resource than where it started
  • Can spawn children — child operations complete before the parent
cpp
// Lifecycle:
//   connect(sender, receiver)  -> creates async operation (opstate)
//   start(opstate)             -> async lifetime begins
//   ... work executes ...
//   set_{value,error,stopped}(rcvr, ...) -> async lifetime ends

The async lifetime begins when start() executes and ends when a completion function executes. Accessing the operation state after its lifetime ends is undefined behavior.

The Three Dispositions

DispositionCompletionResult Datums
Successset_value(rcvr, vs...)Any number
Failureset_error(rcvr, err)Exactly one
Cancellationset_stopped(rcvr)None

Every async operation ends in exactly one of these three.

Operation State

Every async operation has associated operation state. This is the object returned by connect(). It owns:

  • The receiver (which contains the completion handlers)
  • Any internal buffers the sender needs

The operation state’s lifetime must exceed the async lifetime. Destroying the operation state before the completion function executes is undefined behavior.

cpp
// Correct:
auto op = connect(sndr, rcvr);
start(op);            // async lifetime begins
// ... time passes ...
// set_value(rcvr, 42) executes -> async lifetime ends
// now we can destroy op

// Undefined behavior:
auto op = connect(sndr, rcvr);
start(op);
// destroy op here -> op invalid while async operation still active

The operation state becomes invalid during the execution of the completion function. This is intentional — it allows the receiver’s completion handler to destroy the operation state itself.

Environment

Every async operation has an environment — a queryable object that represents the caller’s execution-time properties. The environment is equal to the receiver’s environment (obtained via get_env(rcvr)).

cpp
// Inside a receiver:
auto get_env() const noexcept {
  return prop{get_scheduler, my_sched,
             get_stop_token, my_stop_source.get_token()};
}

The environment flows from parent to child through the pipeline.

Sender

A sender is a factory for async operations. When you connect(sndr, rcvr), you create an operation — the sender tells the system what to do, the receiver tells it what to do with the result.

A sender is:

  • A factory for one or more async operations
  • Started when connected + started
  • Valid or invalid — becomes invalid when its parent sender becomes invalid

Scheduler

A scheduler abstracts an execution resource. Its job is to produce senders whose operations execute on that resource.

cpp
sender auto s = schedule(my_scheduler);  // get a sender
// Connecting this sender creates an operation that runs on my_scheduler

A scheduler:

  • Is a factory for senders
  • Becomes invalid when its execution resource becomes invalid
  • All derived objects (schedule senders, operation states) become invalid too

Completion Scheduler

An async operation can have different schedulers for each completion disposition:

cpp
// Value completion may run on one scheduler,
// error completion on another,
// stopped completion on yet another.

A schedule sender is required to have a value completion scheduler equal to the scheduler that produced it. This is how adaptors know where to transfer execution.

Completion Signature

A completion signature is a function type describing a completion operation:

cpp
set_value_t(int, float)   // set_value(rcvr, int, float)
set_error_t(error_code)   // set_error(rcvr, error_code)
set_stopped_t()           // set_stopped(rcvr)

Together, a sender type and an environment type determine the set of possible completion signatures. The receiver’s type does not affect signatures — only the receiver’s environment type does.

A non-dependent sender is one whose completion signatures are knowable without any environment.

Three Categories of Sender Algorithms

CategoryTakesReturnsExample
FactoryNon-sendersA senderjust(42)
AdaptorOne or more senders + argsA parent senderthen(sndr, f), when_all(s1, s2)
ConsumerOne or more sendersNon-sendersync_wait(sndr)

An async operation created by a parent sender (adaptor) is the parent of the child operations created by the child senders. Parent operations complete after all started child operations complete.

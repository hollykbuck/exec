04 · Hello World Pipeline — schedule | then | sync_wait

Standard Section: [exec.sched] [exec.sync.wait] stdexec Reference: examples/hello_world.cpp Related Tests: tests/test_basics.cpp

Your First stdexec Program

cpp
// Corresponding test: test_basics.cpp → "schedule + then + sync_wait basic pipeline"
#include <iostream>
#include <stdexec/execution.hpp>
#include <exec/static_thread_pool.hpp>

using namespace stdexec;

int main() {
  exec::static_thread_pool ctx{8};           // ① Create a thread pool
  scheduler auto sch = ctx.get_scheduler();  // ② Get a scheduler

  // ③ Build the pipeline
  sender auto begin  = schedule(sch);                              // "execute on thread pool"
  sender auto hi     = then(begin, [] { return 13; });             // "then compute 13"
  sender auto add_42 = then(hi, [](int arg) { return arg + 42; }); // "then add 42"

  // ④ Block and wait for the result
  auto [i] = sync_wait(std::move(add_42)).value();
  std::cout << i << std::endl;  // 55
}

Line-by-Line Explanation

① static_thread_pool ctx{8}: Creates a thread pool managing 8 threads. This is an execution resource.

② ctx.get_scheduler(): Gets a scheduler from the thread pool. A scheduler is a “handle” to the thread pool, used to schedule work on it.

③ Pipeline Construction:

graph LR
  A["schedule(sch)"] --> B["sender&lt;void&gt;<br/>means 'do something on the thread pool'"]
  B --> C["then([] { return 13; })"]
  C --> D["sender&lt;int&gt;<br/>means 'then compute 13'"]
  D --> E["then([](int a) { return a + 42; })"]
  E --> F["sender&lt;int&gt;<br/>means 'then add 42'"]

Note: No code is executing on the thread pool at this point. The pipeline is just a lazy description.

④ sync_wait: A consumer. It internally creates a receiver → connect(sender, receiver) → start() → blocks until completion → returns the result.

Pipe Operator Syntax

The above pipeline is equivalent to:

cpp
// Corresponding test: test_pipe.cpp → "pipe syntax equals function call syntax"
// Nested function call (not recommended)
auto result = sync_wait(
  then(
    then(
      schedule(sch),
      [] { return 13; }
    ),
    [](int a) { return a + 42; }
  )
);

// Pipe syntax (recommended)
auto result = sync_wait(
  schedule(sch)
    | then([] { return 13; })
    | then([](int a) { return a + 42; })
);

operator| makes the code read naturally from left to right, similar to Unix pipes.

Return Value: std::optional<std::tuple<...>>

sync_wait returns std::optional<std::tuple<Ts...>>:

set_value(42)std::optional{std::tuple{42}}
set_error(e)std::nullopt (throws exception)
set_stoppedstd::nullopt

Key Takeaways

  1. Pipeline construction is an O(1) lazy operation that does no work
  2. sync_wait is a synchronization boundary that pulls the async world back into the sync world
  3. All errors and cancellations are propagated through std::optional
  4. Pipe syntax and nested function call syntax are equivalent; operator| is recommended

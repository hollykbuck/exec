04 · Hello World Pipeline — schedule | then | sync_wait

Standard Section: [exec.sched] [exec.sync.wait] stdexec Reference: examples/hello_world.cpp Related Tests: tests/test_basics.cpp

The previous chapters covered concepts. Now let’s write our first real std::execution program.

Your First stdexec Program

cpp
#include <iostream>
#include <stdexec/execution.hpp>
#include <exec/static_thread_pool.hpp>

using namespace stdexec;

int main() {
  exec::static_thread_pool ctx{8};           // ① Create a thread pool
  scheduler auto sch = ctx.get_scheduler();  // ② Get a scheduler

  // ③ Build the pipeline
  sender auto begin  = schedule(sch);                              // "execute on thread pool"
  sender auto hi     = then(begin, [] { return 13; });             // "then compute 13"
  sender auto add_42 = then(hi, [](int arg) { return arg + 42; }); // "then add 42"

  // ④ Block and wait for the result
  auto [i] = sync_wait(std::move(add_42)).value();
  std::cout << i << std::endl;  // 55
}

This program does four things. Let’s go through them one by one.

① static_thread_pool ctx{8}: Creates a thread pool managing 8 threads. This is an “execution resource.”

② ctx.get_scheduler(): Gets a Scheduler from the thread pool. The Scheduler is a handle to the pool — you use it to schedule work onto it.

③ Pipeline construction:

graph LR
  A["schedule(sch)"] --> B["sender&lt;void&gt;<br/>'do something on the thread pool'"]
  B --> C["then([] { return 13; })"]
  C --> D["sender&lt;int&gt;<br/>'then compute 13'"]
  D --> E["then([](int a) { return a + 42; })"]
  E --> F["sender&lt;int&gt;<br/>'then add 42'"]

Note: No code is executing on the thread pool right now. The pipeline is just a lazy description, like a blueprint. The actual work happens in the next step.

④ sync_wait: This is a Sender Consumer. It does three things internally:

  1. Creates a Receiver
  2. Calls connect(sender, receiver) to get an Operation State
  3. Calls start() to begin execution, then blocks until completion, and returns the result

You might ask: how does sync_wait know which thread pool to use? Because the Scheduler was captured inside the Sender when schedule(sch) was called. It propagates through the pipeline, so when sync_wait kicks things off, the Sender knows exactly where to run.

Pipe Syntax

The pipeline above is equivalent to nested function calls:

cpp
// Nested function call (not recommended)
auto result = sync_wait(
  then(
    then(
      schedule(sch),
      [] { return 13; }
    ),
    [](int a) { return a + 42; }
  )
);

// Pipe syntax (recommended)
auto result = sync_wait(
  schedule(sch)
    | then([] { return 13; })
    | then([](int a) { return a + 42; })
);

Both forms are completely equivalent. Pipe syntax reads left to right, which is more intuitive — like Unix pipes. We’ll dig into how operator| works in a later chapter.

sync_wait Return Value

sync_wait returns std::optional<std::tuple<Ts...>>. The three completion signals map to three outcomes:

Completion SignalReturn Value
set_value(42)std::optional{std::tuple{42}}
set_error(e)std::nullopt (rethrows exception)
set_stoppedstd::nullopt
cpp
auto [i] = sync_wait(just(42)).value();                              // i = 42
auto [x, y] = sync_wait(when_all(just(1), just(2.0))).value();       // x=1, y=2.0
auto opt = sync_wait(just_error(std::exception_ptr{}));
if (!opt) { /* error or cancellation occurred */ }

In the code above, when_all(just(1), just(2.0)) fires set_value(1, 2.0), so sync_wait returns std::optional{std::tuple{1, 2.0}}. After unpacking, x=1, y=2.0.

Querying the Environment

Senders in a pipeline can also query the current execution environment:

cpp
sync_wait(when_all(
  just(42),
  get_scheduler(),       // query the scheduler from the receiver's environment
  get_stop_token()       // query the stop token
));

In the code above, get_scheduler() and get_stop_token() aren’t ordinary Sender Factories — they read information from the Receiver’s environment. This is especially useful when crossing execution contexts.

Key Takeaways

  1. Pipeline construction is an O(1) lazy operation that does no work
  2. sync_wait is a synchronization boundary that pulls the async world back into the sync world
  3. All errors and cancellations are propagated through std::optional
  4. Pipe syntax and nested function call syntax are equivalent; pipe syntax is recommended

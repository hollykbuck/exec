05 · Thread Pools and Execution Contexts

On this page

Related Tests: tests/test_thread_pool.cpp, exec/inline_scheduler.hpp

The Hello World in the previous chapter used a thread pool. Now let’s take a closer look — what exactly is an “execution context” in std::execution, what options are available, and how do you switch between them?

Execution Contexts

An execution context is “where the actual work happens.” stdexec provides several built-in execution contexts:

ContextHeaderBehavior
static_thread_poolexec/static_thread_pool.hppFixed-size thread pool
inline_schedulerexec/inline_scheduler.hppExecutes immediately on the current thread
trampoline_schedulerexec/trampoline_scheduler.hppExecutes after stack unwinding (prevents stack overflow)
run_loopstdexec/execution.hppManually controlled FIFO event loop
parallel_schedulerstdexec/__detail/Standard parallel scheduler

Why isn’t there a simple std::thread scheduler? Because static_thread_pool is the general-purpose scheduler you’ll use most of the time, and inline_scheduler is there for testing and synchronous scenarios.

static_thread_pool in Detail

cpp
exec::static_thread_pool pool{4};   // Create 4 threads
auto sch = pool.get_scheduler();    // Get the scheduler

// All work scheduled via sch executes on the thread pool
auto result = sync_wait(
  schedule(sch) | then([] {
    std::cout << "Running on thread pool\n";
    return 42;
  })
);

In the code above, pool{4} creates 4 threads, and get_scheduler() returns a lightweight handle. When sync_wait kicks things off, the Sender created by schedule(sch) schedules the lambda onto the thread pool. You don’t need to manage thread creation or destruction manually.

Lifecycle

cpp
exec::static_thread_pool pool{4};
{
  auto sch = pool.get_scheduler();
  auto snd = schedule(sch) | then(work);
  sync_wait(snd);  // completes before pool is destroyed
}
// pool is destroyed here, automatically waits for all queued tasks to finish

This code demonstrates proper lifecycle management: make sure all async operations finish before the pool is destroyed. static_thread_pool’s destructor automatically waits for all queued tasks to complete, rather than brutally interrupting them.

Execution Context Switching

What if I want some work to run on one thread pool for I/O, and other work on a different thread pool for computation?

cpp
exec::static_thread_pool io_pool{2};    // simulate I/O thread
exec::static_thread_pool cpu_pool{4};   // simulate compute thread

auto result = sync_wait(
  starts_on(io_pool.get_scheduler(),    // ① Start on I/O thread
    just(1)
    | then([](int x) {                  // ② Execute on I/O thread
        return x + 1;
      })
    | continues_on(cpu_pool.get_scheduler()) // ③ Switch to compute thread
    | then([](int x) {                  // ④ Execute on compute thread
        return x * 2;
      })
  )
);
// result == 4

This code shows a typical scenario: I/O operations run on an I/O thread pool, computation switches to a compute thread pool. starts_on and continues_on make this switch possible.

The difference between starts_on and continues_on:

AdaptorEffect
starts_on(sch, sndr)Starts sndr on sch
continues_on(sch, sndr)Switches to sch after sndr completes
on(sch, sndr)Equivalent to starts_on(sch, continues_on(sch, sndr))

inline_scheduler — Current Thread Execution

cpp
exec::inline_scheduler inline_sch{};

sync_wait(
  schedule(inline_sch)
    | then([] {
        // This lambda executes on the calling thread of sync_wait
        return 42;
      })
);

In the code above, inline_scheduler doesn’t create any new threads. The lambda runs synchronously on the thread that called sync_wait.

inline_scheduler is suitable for:

  • Single-threaded testing
  • Synchronous operations that need to run on the current thread
  • As a minimal reference implementation of the scheduler interface

trampoline_scheduler — Preventing Stack Overflow

cpp
exec::trampoline_scheduler tramp_sch{};

// Use trampoline for deep recursion to prevent stack overflow
sender auto deep_recursion(int depth) -> sender auto {
  if (depth == 0) return just(0);
  return schedule(tramp_sch)
       | then([=] { return deep_recursion(depth - 1); })
       | let_value([](sender auto s) { return s; });
}

In the code above, if deep_recursion called itself directly, each recursive step would consume stack space — deep enough and you get a stack overflow. trampoline_scheduler makes each step execute in the current thread’s event loop, avoiding deeply nested call stacks.

Design Philosophy

The separation of execution context from scheduler is a core design of std::execution:

graph TD
  A["Execution Context (static_thread_pool)"]
  A --> B["get_scheduler()"]
  B --> C["Scheduler<br/>(lightweight handle, copyable)"]
  C --> D["schedule()"]
  D --> E["Sender (lazy description)"]

This separation three benefits:

  1. Execution context and scheduler are decoupled: multiple schedulers can reference the same resource
  2. Scheduler is a lightweight copyable object: it can be safely passed through pipelines
  3. Context switching is expressed at compile time: starts_on / continues_on express switching intent at the type level

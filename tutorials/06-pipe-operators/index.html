06 · Pipe Operators — How operator| Composes Senders

Standard Section: [exec.adapt.obj] stdexec Reference: include/stdexec/execution.hpp Related Tests: tests/test_pipe.cpp

The Problem

Given multiple sender adaptors, how do you compose them elegantly?

cpp
// Corresponding test: test_pipe.cpp
// Nested function call — reads from inside out, counterintuitive
auto r = upon_error(
           then(
             upon_stopped(
               let_value(
                 schedule(sch),
                 [](auto...) { return just(42); }
               ),
               [] { return -1; }
             ),
             [](int x) { return x + 1; }
           ),
           [](auto e) { return 0; }
         );

Solution: operator| pipe operator.

cpp
// Corresponding test: test_pipe.cpp → "operator| chains sender adaptors"
// Pipe — reads from left to right, natural and fluent
auto r = schedule(sch)
       | let_value([](auto...) { return just(42); })
       | upon_stopped([] { return -1; })
       | then([](int x) { return x + 1; })
       | upon_error([](auto e) { return 0; });

Standard Definition

The standard [exec.adapt.obj] defines:

cpp
// Left-associative binary operator
sender auto operator|(sender auto sndr, sender_adaptor_closure auto closure);
  • Left operand is a sender (or type convertible to sender)
  • Right operand is a sender adaptor closure object (a function object invocable with .)
  • Equivalent to closure(sndr), i.e., calling the closure object with the sender

Two Forms of Sender Adaptors

Most adaptors support two call forms:

cpp
// Corresponding test: test_pipe.cpp → "pipe syntax equals function call syntax"
// Form 1: Function call
auto s1 = then(schedule(sch), f);

// Form 2: Partial application (currying), returns a closure object
auto s2 = schedule(sch) | then(f);

// Both are equivalent: verified via sync_wait result
// sync_wait(then(schedule(sch), f)) == sync_wait(schedule(sch) | then(f))

Implementation of Partial Application

cpp
// Corresponding test: test_pipe.cpp
// Two overloads of then
namespace stdexec {
  // Full call
  sender auto then(sender auto sndr, std::invocable auto f);

  // Partial application — returns a closure object
  sender_adaptor_closure auto then(std::invocable auto f) {
    return closure{[](sender auto&& sndr, auto&& f) {
      return then(std::forward<decltype(sndr)>(sndr), std::forward<decltype(f)>(f));
    }, std::forward<decltype(f)>(f)};
  }
}

// Usage
schedule(sch) | then([] { return 42; })
// Equivalent to then(schedule(sch), [] { return 42; })
// Internally expands to: operator|(schedule(sch), then([] { return 42; }))
// → then([] { return 42; })(schedule(sch))
// → then(schedule(sch), [] { return 42; })

Composing Closure Objects

Multiple closure objects can be pre-composed into a composite closure:

cpp
// Corresponding test: test_pipe.cpp → "closure objects can be pre-composed"
// Predefine a set of operations
auto pipeline = then([](int x) { return x + 1; })
              | then([](int x) { return x * 2; })
              | upon_error([](auto e) { return -1; });

// Apply to a sender
auto result = sync_wait(just(41) | pipeline);
// Equivalent to: sync_wait(upon_error(then(then(just(41), +1), *2), error_handler))

Consumer Pipe Syntax

Sender consumers can also be invoked via the pipe syntax:

cpp
// Corresponding test: test_pipe.cpp → "consumer wraps pipeline result"
just(42)
  | then([](int x) { return x + 1; })
  | sync_wait;                   // consumer, not adaptor

// Equivalent to
sync_wait(just(42) | then([](int x) { return x + 1; }));

Design Motivation for the Pipe Operator

The Ranges library introduced operator| for range adaptors, and P2300 follows the same design:

  1. Readability: Linear left-to-right reading
  2. Composability: Closure objects can be stored, passed around, and composed
  3. Consistency: Consistent pipe style with the Ranges library
  4. Laziness: Closure objects do no work, they only build the pipeline

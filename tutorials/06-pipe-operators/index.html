06 · Pipe Operators — How operator| Composes Senders

Standard Section: [exec.adapt.obj] stdexec Reference: include/stdexec/execution.hpp Related Tests: tests/test_pipe.cpp

The Problem

In the previous chapters, you’ve seen just(42) | then(f) | sync_wait. But have you ever wondered: what would the code look like without operator|?

cpp
// Nested function call — reads from inside out, counterintuitive
auto r = upon_error(
           then(
             upon_stopped(
               let_value(
                 schedule(sch),
                 [](auto...) { return just(42); }
               ),
               [] { return -1; }
             ),
             [](int x) { return x + 1; }
           ),
           [](auto e) { return 0; }
         );

In the code above, to understand what the pipeline does, you have to start from the innermost schedule(sch) and read outward — layer by layer. The order you read the code is the opposite of the order it executes.

Rewritten with operator|:

cpp
// Pipe syntax — reads left to right, natural and fluent
auto r = schedule(sch)
       | let_value([](auto...) { return just(42); })
       | upon_stopped([] { return -1; })
       | then([](int x) { return x + 1; })
       | upon_error([](auto e) { return 0; });

Now the order you read is the order of execution: schedule first, then handle values, handle cancellation, do computation, handle errors. Same philosophy as Unix pipes — data flows from left to right.

Standard Definition

The standard ([exec.adapt.obj]) defines:

cpp
// Left-associative binary operator
sender auto operator|(sender auto sndr, sender_adaptor_closure auto closure);
  • Left operand is a Sender (or a type convertible to Sender)
  • Right operand is a Sender Adaptor Closure object (an invocable function object)
  • Equivalent to closure(sndr) — calling the closure object with the sender

Two Forms of Sender Adaptors

Most adaptors support two call forms:

cpp
// Form 1: Function call
auto s1 = then(schedule(sch), f);

// Form 2: Partial application (currying), returns a closure object
auto s2 = schedule(sch) | then(f);

// Both are equivalent
// sync_wait(then(schedule(sch), f)) == sync_wait(schedule(sch) | then(f))

You might ask: how does partial application work? Roughly like this:

cpp
namespace stdexec {
  // Full call — takes both a sender and a function
  sender auto then(sender auto sndr, std::invocable auto f);

  // Partial application — returns a closure object that "remembers" f
  sender_adaptor_closure auto then(std::invocable auto f) {
    return closure{[](sender auto&& sndr, auto&& f) {
      return then(std::forward<decltype(sndr)>(sndr), std::forward<decltype(f)>(f));
    }, std::forward<decltype(f)>(f)};
  }
}

// Usage
schedule(sch) | then([] { return 42; })
// Equivalent to then(schedule(sch), [] { return 42; })
// Internally expands to: operator|(schedule(sch), then([] { return 42; }))
// → then([] { return 42; })(schedule(sch))
// → then(schedule(sch), [] { return 42; })

In the code above, then(f) returns a closure object that “remembers” f. When operator| passes a Sender to it, the closure combines both and calls the full then(sndr, f).

Pre-composing Pipelines

Multiple closure objects can be pre-composed into a composite closure:

cpp
// Pre-define a set of operations
auto pipeline = then([](int x) { return x + 1; })
              | then([](int x) { return x * 2; })
              | upon_error([](auto e) { return -1; });

// Apply to a sender
auto result = sync_wait(just(41) | pipeline);
// Equivalent to: sync_wait(upon_error(then(then(just(41), +1), *2), error_handler))

In the code above, pipeline is a pre-composed closure object. You can store it, pass it around, and reuse it — like defining a “processing chain” that can be applied to any sender at any time.

Consumer Pipe Syntax

Sender consumers can also be invoked via pipe syntax:

cpp
just(42)
  | then([](int x) { return x + 1; })
  | sync_wait;                   // consumer, not adaptor

// Equivalent to
sync_wait(just(42) | then([](int x) { return x + 1; }));

In the code above, sync_wait sits at the end of the pipeline, consuming the final sender and returning the result.

Why Pipe Operators?

The Ranges library was the first to introduce operator| for range adaptors in C++20. P2300 follows the same design:

  1. Readability: linear left-to-right reading, matches how we think
  2. Composability: closure objects can be stored, passed around, and pre-composed
  3. Consistency: consistent pipe style with the Ranges library
  4. Laziness: closure objects do no work, they only build the pipeline

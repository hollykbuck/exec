Two Async Styles

std::execution gives you two styles of async programming:

  1. Sender pipelines — composed with operator|, great for fixed, high-performance paths
  2. Coroutines — use co_await, great for complex control flow (loops, branches, recursion)

You might ask: do I have to pick one?

The answer is: no. std::execution lets you seamlessly mix both.

graph LR
  A["Sender"] <--> B["Awaitable"]
  • Sender is awaitable: you can co_await sender inside stdexec::task<T>
  • Awaitable is sender: any awaitable can be used as a sender in a pipeline

co_await sender in Coroutines

cpp
#include <stdexec/execution.hpp>

using namespace stdexec;

// This coroutine takes two senders, co_awaits them, and adds the results
template <sender S1, sender S2>
auto async_add(S1 s1, S2 s2) -> stdexec::task<int> {
  int a = co_await static_cast<S1&&>(s1);  // await the first sender
  int b = co_await static_cast<S2&&>(s2);  // await the second sender
  co_return a + b;
}

int main() {
  // task itself is a sender, so it can be consumed by sync_wait
  auto [i] = sync_wait(async_add(just(40), just(2))).value();
  std::cout << i << '\n';  // 42
}

There are three key points in the code above:

  1. task<T> itself is a Sender (it models the sender concept)
  2. Inside task<T>, co_await sender implicitly converts the sender into an awaitable
  3. task<T> can be used anywhere a sender is expected (sync_wait, then, when_all…)

Handling Cancellation

How do you handle cancellation in a coroutine? Use stopped_as_optional:

cpp
template <sender S1, sender S2>
auto async_add_opt(S1 s1, S2 s2) -> stdexec::task<std::optional<int>> {
  co_return co_await stopped_as_optional(async_add(s1, s2));
}

// If async_add is cancelled, returns nullopt
auto opt = sync_wait(async_add_opt(just(40), just(2))).value();
if (opt) {
  std::cout << opt.value() << '\n';
}

In the code above, stopped_as_optional converts the set_stopped signal into std::nullopt. You don’t need to write separate error handling for cancellation — just check if the optional has a value.

Querying the Environment in Coroutines

cpp
// tasks have an associated stop token
auto async_stop_token() -> stdexec::task<std::optional<stdexec::inplace_stop_token>> {
  co_return co_await stopped_as_optional(get_stop_token());
}

A Common Pitfall

A frequent misunderstanding when first using task<T> is assuming that co_await schedule(pool) will switch the coroutine’s execution to the pool thread, just like sync_wait(schedule(pool) | then(...)) does with plain sender pipelines. It won’t.

cpp
exec::static_thread_pool pool{1};

// ✓ Plain sender pipeline switches threads
std::thread::id work_id;
sync_wait(schedule(pool.get_scheduler())
  | then([&] { work_id = this_thread::get_id(); }));
assert(work_id != this_thread::get_id());  // passes

// ✗ Coroutine does NOT switch threads
auto coro = [&]() -> task<bool> {
  co_await schedule(pool.get_scheduler());
  co_return this_thread::get_id() != work_id;
};
auto [switched] = sync_wait(coro()).value();
assert(switched);  // FAILS — same thread!

To actually switch execution context with coroutines, use continues_on / starts_on / on at the pipeline level — wrap the entire task, not the individual co_await:

cpp
// ✓ Run the whole coroutine on the pool thread
sync_wait(continues_on(coro(), pool.get_scheduler()));

// ✓ Or start the coroutine on the pool
sync_wait(starts_on(pool.get_scheduler(), coro()));

For finer-grained control inside a coroutine, use continues_on to wrap a subsection of work, but understand the affine tax — the completion will bounce back to the task’s scheduler.

Best Practices

  1. Use sender pipelines for static, performance-critical async workflows
  2. Use task<T> coroutines for dynamic, control-flow-complex async workflows
  3. Mix both — co_await sender inside coroutines, use task<T> as a sender in pipelines
cpp
// Embed coroutine logic inside a pipeline
auto pipeline(int x) -> stdexec::task<int> {
  auto y = co_await some_complex_logic(x);
  co_return y;
}

sync_wait(
  just(42)
    | then([](int x) { return x + 1; })
    | let_value([](int x) { return pipeline(x); })  // task is a sender
);

The code above shows a typical mixed scenario: simple computation uses then, complex logic uses the pipeline coroutine, and let_value bridges them seamlessly.

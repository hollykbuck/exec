Bidirectional Interop Between Senders and Coroutines

std::execution provides bidirectional interop with C++20 coroutines:

graph LR
  A["Sender"] <--> B["Awaitable"]
  • Sender is awaitable: You can use co_await sender inside stdexec::task<T>
  • Awaitable is sender: Any awaitable can be used as a sender in a pipeline

co_await sender in Coroutines

cpp
// Corresponding test: test_coroutine.cpp → "task coroutine can await senders"
#include <stdexec/execution.hpp>

using namespace stdexec;

// Both sender and task are lazy,
// so this takes sender parameters and co_awaits them
template <sender S1, sender S2>
auto async_add(S1 s1, S2 s2) -> stdexec::task<int> {
  int a = co_await static_cast<S1&&>(s1);  // await the first sender
  int b = co_await static_cast<S2&&>(s2);  // await the second sender
  co_return a + b;
}

int main() {
  // task is itself a sender, so it can be used with sync_wait
  auto [i] = sync_wait(async_add(just(40), just(2))).value();
  std::cout << i << '\n';  // 42
}

Key Points

  1. task<T> is a sender (models the sender concept)
  2. co_await sender inside task<T> implicitly converts the sender to an awaitable
  3. task<T> itself can be used anywhere a sender is expected (sync_wait, then, when_all…)

Handling Cancellation

Use stopped_as_optional to convert cancellation into std::nullopt:

cpp
// Corresponding test: test_coroutine.cpp → "stopped_as_optional in coroutine"
template <sender S1, sender S2>
auto async_add_opt(S1 s1, S2 s2) -> stdexec::task<std::optional<int>> {
  co_return co_await stopped_as_optional(async_add(s1, s2));
}

// If async_add is cancelled, returns nullopt
auto opt = sync_wait(async_add_opt(just(40), just(2))).value();
if (opt) {
  std::cout << opt.value() << '\n';
}

Querying the Environment in Coroutines

cpp
// Corresponding test: test_stop_token.cpp → "get_stop_token from sync_wait environment"
// tasks have an associated stop token
auto async_stop_token() -> stdexec::task<std::optional<stdexec::inplace_stop_token>> {
  co_return co_await stopped_as_optional(get_stop_token());
}

Coroutines vs Sender Pipelines

AspectSender PipelineCoroutines
Composition`operator`
Control FlowLinear (one adaptor per step)Arbitrary (loops, branches, recursion)

PerformanceZero-overhead abstractionCoroutine frame overhead
Use CaseFixed pipelines, high performanceComplex control flow, state machines

Best Practices

  1. Use sender pipelines to express static, performance-critical async workflows
  2. Use task<T> coroutines to express dynamic, control-flow-complex async workflows
  3. Mix both — co_await sender inside coroutines, use task<T> as a sender in pipelines
cpp
// Corresponding test: test_coroutine.cpp → "task is itself a sender usable with sync_wait"
// Mixed example: embedding coroutine logic in a pipeline
auto pipeline(int x) -> stdexec::task<int> {
  auto y = co_await some_complex_logic(x);
  co_return y;
}

sync_wait(
  just(42)
    | then([](int x) { return x + 1; })
    | let_value([](int x) { return pipeline(x); })  // task is a sender
);

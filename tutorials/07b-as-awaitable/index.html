07b · as_awaitable and with_awaitable_senders — How co_await Sender Works Inside

The Missing Piece

This chapter explains the internals of coroutine integration. You can skip it without affecting subsequent reading.

Chapter 07 showed you that you can co_await a sender inside task<T>. But how does it actually work?

When you co_await a sender, two things happen: first the coroutine suspends, then after the sender completes the receiver is invoked, obtains the value, and resumes.

Coroutines only understand awaitables, not senders, so a bridge is needed.

Two exposition classes and one CPO solve this problem:

  1. with_awaitable_senders<Promise> — a CRTP base class for promise types that makes any sender awaitable
  2. sender-awaitable<Sndr, Promise> — the bridge that connects sender/receiver to the coroutine machinery
  3. as_awaitable — the CPO that dispatches: “is this thing a sender, an awaitable, or something else?”

with_awaitable_senders — The Promise Base

task<T>’s promise type needs to inherit from with_awaitable_senders<task::promise_type>:

cpp
namespace std::execution {
  template <class Promise>
  struct with_awaitable_senders {
    template <class OtherPromise>
    void set_continuation(coroutine_handle<OtherPromise> h) noexcept;

    coroutine_handle<> continuation() const noexcept;

    coroutine_handle<> unhandled_stopped() noexcept {
      return stopped-handler(continuation.address());
    }

    template <class Value>
    auto await_transform(Value&& value);
  };
}

set_continuation — called when this coroutine is co_await-ed by another coroutine. It saves the caller’s coroutine handle and propagates the caller’s unhandled_stopped handler.

unhandled_stopped — invoked when a co_await sender completes with set_stopped. By default it calls terminate(). If a parent coroutine has a handler, it chains upward. This is how cancellation propagates through coroutine frames.

await_transform — the hook that C++ coroutines call for every co_await expr. It delegates to as_awaitable(expr, *this):

cpp
template <class Value>
auto await_transform(Value&& value) {
  return as_awaitable(std::forward<Value>(value), static_cast<Promise&>(*this));
}

as_awaitable — The Dispatcher

as_awaitable(expr, promise) follows a priority chain:

  1. If expr.as_awaitable(promise) is well-formed — use the object’s own custom awaitable conversion
  2. If expr is a sender_in with single value — apply transform_sender, then use adapt-for-await-completion, then wrap in sender-awaitable
  3. If expr is already an awaitable — return it as-is (the coroutine handles it natively)
  4. As a fallback for single-value senders — sender-awaitable{adapt-for-await-completion(transform_sender(expr)), promise}

The adapt-for-await-completion step calls get_await_completion_adaptor(get_env(s)) if available — a customization hook that lets schedulers inject adapter logic at the boundary between sender and coroutine.

sender-awaitable — The Bridge

This is the workhorse. It is the awaiter type that co_await expr interacts with:

cpp
template <class Sndr, class Promise>
class sender-awaitable {
  using value-type = single-sender-value-type<Sndr, env_of_t<Promise>>;
  using result-type = conditional_t<is_void_v<value-type>, unit, value-type>;

  struct awaitable-receiver;

  variant<monostate, result-type, exception_ptr> result;
  connect_result_t<Sndr, awaitable-receiver> state;

public:
  sender-awaitable(Sndr&& sndr, Promise& p);

  static constexpr bool await_ready() noexcept { return false; }
  void await_suspend(coroutine_handle<Promise>) noexcept { start(state); }
  value-type await_resume();
};

The three awaiter methods:

await_ready() — always false. The coroutine always suspends.

await_suspend(h) — calls start(state). This fires the sender. The coroutine frame is suspended until the sender completes.

await_resume() — called when the coroutine resumes. Checks the result variant:

cpp
if (result.index() == 2)
  rethrow_exception(get<2>(result));
if constexpr (!is_void_v<value-type>)
  return forward<value-type>(get<1>(result));

The awaitable-receiver

This is the receiver type inside sender-awaitable. It stores:

  • A pointer to the result variant (where completion data goes)
  • The continuation coroutine handle (what to resume on completion)

Its three completion handlers:

cpp
void set_value(awaitable-receiver& rcvr, auto&&... vs) noexcept {
  try {
    rcvr.result-ptr->template emplace<1>(vs...);
  } catch(...) {
    rcvr.result-ptr->template emplace<2>(current_exception());
  }
  MAKE-NOEXCEPT(rcvr.continuation.resume());
}

void set_error(awaitable-receiver& rcvr, auto err) noexcept {
  try {
    rcvr.result-ptr->template emplace<2>(AS-EXCEPT-PTR(err));
  } catch(...) {
    rcvr.result-ptr->template emplace<2>(current_exception());
  }
  MAKE-NOEXCEPT(rcvr.continuation.resume());
}

void set_stopped(awaitable-receiver& rcvr) noexcept {
  MAKE-NOEXCEPT(
    static_cast<coroutine_handle<>>(
      rcvr.continuation.promise().unhandled_stopped()).resume());
}

The pattern is:

  • set_value — store the result in the variant (index 1), resume the coroutine
  • set_error — store the exception_ptr in the variant (index 2), resume the coroutine
  • set_stopped — don’t resume the coroutine at all; instead call the promise’s unhandled_stopped() to propagate cancellation upward

Importantly, the environment of awaitable-receiver forwards all forwarding queries to the coroutine’s promise. This means get_scheduler(), get_stop_token(), etc. work transparently inside co_await.

The awaitable-sender Concept

A sender S is awaitable in a coroutine with promise P if:

cpp
template <class Sndr, class Promise>
concept awaitable-sender =
  single-sender<Sndr, env_of_t<Promise>> &&
  sender-to<Sndr, sender-awaitable<Sndr, Promise>::awaitable-receiver> &&
  requires (Promise& p) {
    { p.unhandled_stopped() } -> convertible_to<coroutine_handle<>>;
  };

Three requirements:

  1. The sender must have exactly one value completion signature (single-sender)
  2. The sender must be connectable to the internal awaitable-receiver
  3. The promise type must expose unhandled_stopped() returning a coroutine_handle<>

This is why task<T> can only co_await senders with exactly one value type. If a sender can complete with multiple value types, wrap it in into_variant or stopped_as_optional first.

Cancellation Propagation Chain

graph TD
    A["co_await sender inside task&lt;A&gt;"] --> B["Sender completes with set_stopped"]
    B --> C["awaitable-receiver::set_stopped() called"]
    C --> D["promise.unhandled_stopped() returns parent handle"]
    D --> E["Resumes parent handle (task&lt;B&gt;)"]
    E --> F["task&lt;B&gt;'s unhandled_stopped() chains upward"]
    F --> G["If no handler in chain: terminate()"]

The set_continuation() method is what builds this chain:

cpp
template <class OtherPromise>
void set_continuation(coroutine_handle<OtherPromise> h) noexcept {
  continuation = h;
  if constexpr (requires(OtherPromise& other) { other.unhandled_stopped(); }) {
    stopped-handler = [](void* p) noexcept -> coroutine_handle<> {
      return coroutine_handle<OtherPromise>::from_address(p)
        .promise().unhandled_stopped();
    };
  } else {
    stopped-handler = &default-unhandled-stopped;
  }
}

When coroutine B co_awaits coroutine A’s task, B’s set_continuation is called with A’s promise handle. If A completes with set_stopped, the chain walks up to B’s unhandled_stopped, and so on.

Summary

ComponentRole
with_awaitable_senders<P>Promise base class; provides await_transform and unhandled_stopped
as_awaitable(expr, p)CPO that routes expr to the right awaitable wrapper
sender-awaitable<S, P>Awaiter that connects a sender, suspends, and resumes on completion
awaitable-receiverStores result + continuation handle; three handler forms

Every co_await sender inside task<T> goes through this exact path. The elegance is that it uses the sender model’s own connect + start + completion signals — no special coroutine-specific dispatch layer needed.

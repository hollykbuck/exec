The task Class Template

task<T, Environment> is a sender that doubles as a coroutine return type. Chapter 07 showed usage; this chapter examines the type design.

cpp
namespace std::execution {
  template <class T = void, class Environment = env<>>
  class task {
  public:
    using sender_concept = sender_tag;
    using allocator_type     = see-below;
    using start_scheduler_type = see-below;
    using stop_source_type   = see-below;
    using stop_token_type    = decltype(declval<stop_source_type>().get_token());
    using error_types        = see-below;

    class promise_type;

    template <class Self, class... Env>
      static consteval auto get_completion_signatures();

    template <receiver Rcvr>
      state<Rcvr> connect(Rcvr&& rcvr) &&;

  private:
    coroutine_handle<promise_type> handle;
  };
}

The Environment Parameter

task<T, Environment> takes an Environment type that customizes every aspect of the coroutine:

Nested TypeDefaultSource
allocator_typeallocator<byte>Environment::allocator_type
start_scheduler_typetask_schedulerEnvironment::start_scheduler_type
stop_source_typeinplace_stop_sourceEnvironment::stop_source_type
error_typescompletion_signatures<set_error_t(exception_ptr)>Environment::error_types

This means you can customize what allocator, scheduler, stop source, and error types a task uses by providing a custom Environment:

cpp
struct my_task_env {
  using allocator_type = my_custom_allocator;
  using start_scheduler_type = my_scheduler;
  using stop_source_type = never_stop_source;
  using error_types = completion_signatures<set_error_t(MyError)>;
};

task<int, my_task_env> my_coro() {
  co_return 42;
}

promise_type

The promise type bridges C++ coroutine mechanics to the sender model:

cpp
class promise_type {
public:
  task get_return_object() noexcept;
  static constexpr suspend_always initial_suspend() noexcept { return {}; }
  auto final_suspend() noexcept;

  void unhandled_exception();
  coroutine_handle<> unhandled_stopped() noexcept;

  void return_void();                  // if void T
  void return_value(V&& value);        // if non-void T

  template <sender Sender>
  auto await_transform(Sender&& sndr);

  unspecified get_env() const noexcept;

  void* operator new(size_t size);
  template <class Alloc, class... Args>
  void* operator new(size_t size, allocator_arg_t, Alloc alloc, Args&&...);
  template <class This, class Alloc, class... Args>
  void* operator new(size_t size, const This&, allocator_arg_t, Alloc alloc, Args&&...);
  void operator delete(void* pointer, size_t size) noexcept;
};

initial_suspend — Always Suspended

initial_suspend() returns suspend_always{}. This means the coroutine does not start executing when called — it suspends immediately. The caller gets a task object with a suspended coroutine frame. Execution only begins when connect + start are called.

final_suspend — The Completion Event

final_suspend() returns an awaitable that, when awaited, destroys the coroutine frame and calls the completion function on the receiver:

cpp
// Inside final_suspend's awaiter:
auto st = STATE(*this);  // reference to task::state
st.handle.destroy();     // destroy coroutine frame

if (bool(st.error))
  set_error(move(st.rcvr), move(st.error));
else if constexpr (is_void_v<T>)
  set_value(move(st.rcvr));
else
  set_value(move(st.rcvr), *move(st.result));

This is the moment the task completes as a sender — the coroutine frame is freed and the receiver’s completion handler fires.

await_transform — The co_await Hook

cpp
template <sender Sender>
auto await_transform(Sender&& sndr) {
  if constexpr (same_as<inline_scheduler, start_scheduler_type>)
    return as_awaitable(forward<Sender>(sndr), *this);
  else
    return as_awaitable(affine(forward<Sender>(sndr)), *this);
}

If the task uses inline_scheduler, co_await sender passes through directly. For any other scheduler, it wraps the sender in affine first — ensuring the awaited operation’s completion runs on the task’s scheduler.

Allocation

promise_type provides custom operator new/delete for the coroutine frame. The allocator is obtained from:

  1. The receiver’s environment (via get_allocator(get_env(rcvr)))
  2. Falls back to allocator_type() (default allocator<byte>)

state — The Operation State

task::state<Rcvr> is the operation state returned by connect():

cpp
template <receiver Rcvr>
class task::state {
  coroutine_handle<promise_type> handle;
  remove_cvref_t<Rcvr>           rcvr;
  optional<stop_source_type>     source;
  own-env-t                      own-env;
  Environment                    environment;
  optional<T>                    result;    // if T is non-void
  exception_ptr                  error;
};

start() — The Launch

cpp
void start() & noexcept {
  // 1. Associate promise with state, rcvr, and scheduler
  STATE(prom) = *this;
  RCVR(prom)  = rcvr;
  SCHED(prom) = start_scheduler_type(get_start_scheduler(get_env(rcvr)))
                or start_scheduler_type();

  // 2. Resume the coroutine
  handle.resume();
}

This is where the three key associations are made:

  • STATE(prom) — the promise can now access the operation state
  • RCVR(prom) — the promise knows which receiver to complete
  • SCHED(prom) — the promise knows which scheduler to use

get_stop_token() — Stop Token Bridging

The stop token is bridged from the receiver’s environment into the coroutine:

cpp
stop_token_type get-stop-token() {
  if constexpr (same_as<stop_token_type,
                        decltype(get_stop_token(get_env(rcvr)))>)
    return get_stop_token(get_env(rcvr));
  else {
    if (!source)
      // Initialize source to mirror the receiver's stop state
      source.emplace(/* mirrors get_stop_token(env) */);
    return source->get_token();
  }
}

If the receiver’s stop token type matches the task’s, it’s used directly. Otherwise, a local inplace_stop_source mirrors the receiver’s stop state.

task_scheduler

task_scheduler wraps any scheduler and adds:

  1. Type-erased storage via shared_ptr<parallel_scheduler_backend>
  2. ts-domain that converts bulk_chunked/bulk_unchunked into backend-specific calls
cpp
template <scheduler Sch, class Allocator = allocator<void>>
explicit task_scheduler(Sch&& sch, Allocator alloc = {});

When task_scheduler is the start_scheduler_type, the task coroutine’s await_transform automatically wraps co_await sender in affine(). This means:

cpp
task<int> my_coro() {
  // co_await sender is automatically wrapped in affine(),
  // ensuring completion runs on this task's scheduler
  int x = co_await some_sender();
  co_return x + 1;
}

The ts-domain is a default_domain subclass that intercepts bulk_chunked and bulk_unchunked algorithms, routing them through the backend’s virtual interface.

07d · affine — Completing on the Receiver's Scheduler

On this page

The Problem

When you co_await sender inside a task<T>, the awaited sender might complete on a different execution resource than the task’s scheduler. Without an explicit continues_on, the coroutine resumes on whatever thread the sender happened to finish on.

affine solves this: it adapts a sender so that it completes on the receiver’s start scheduler.

How It Works

cpp
namespace std::execution {
  inline constexpr unspecified affine = /* pipeable sender adaptor */;
}

// Usage:
sender auto s = some_sender() | affine();
// s completes on the receiver's start scheduler

affine is a pipeable sender adaptor. Its transform_sender implementation:

cpp
auto& [_, _, child] = sndr;

if constexpr (requires { forward_like<Sndr>(child).affine(); }) {
  // The sender has a built-in affine() member — use it directly
  return forward_like<Sndr>(child).affine();
} else {
  // Fallback: wrap in continues_on with the start scheduler
  return continues_on(forward_like<Sndr>(child),
                      UNSTOPPABLE-SCHEDULER(get_start_scheduler(ev)));
}

Two paths:

  1. Member affine() — Senders that already guarantee completion on the start scheduler (like just, just_error, just_stopped, read_env, write_env) can provide a zero-overhead affine() that just returns *this.

  2. Fallback — Uses continues_on(child, UNSTOPPABLE-SCHEDULER(get_start_scheduler(ev))). The UNSTOPPABLE-SCHEDULER wrapper prevents cancellation from interfering with the context switch.

Why unstoppable-scheduler?

affine wraps the scheduler in UNSTOPPABLE-SCHEDULER, which is equivalent to unstoppable(schedule(sch)). This means even if the operation is cancelled, the transfer to the task’s scheduler still happens. Without this, a set_stopped during the transfer would leave the coroutine in an inconsistent state on the wrong thread.

Where affine Is Used

task::promise_type::await_transform automatically wraps every co_await sender in affine when the task’s start_scheduler_type is not inline_scheduler:

cpp
template <sender Sender>
auto await_transform(Sender&& sndr) {
  if constexpr (same_as<inline_scheduler, start_scheduler_type>)
    return as_awaitable(forward<Sender>(sndr), *this);
  else
    return as_awaitable(affine(forward<Sender>(sndr)), *this);
}

This means every co_await inside a non-inline task automatically returns to the task’s scheduler before the coroutine resumes.

When You Need It

Outside of task<T>, you rarely need affine explicitly — continues_on or on serve the same purpose with explicit scheduler selection. Inside a coroutine, it’s the mechanism that keeps execution on the right resource without manual continues_on calls at every await point.

Pitfall: affine Prevents co_await schedule(pool) from Switching Threads

A frequent surprise: co_await schedule(pool_sch) inside task<T> does not switch the coroutine to the pool thread, even though sync_wait(schedule(pool_sch) | then(...)) with a plain sender pipeline does.

The reason is affine. await_transform wraps the sender in affine(), which generates finally(schedule(pool_sch), unstoppable(schedule(task_scheduler))). The finally runs schedule(pool_sch) to completion — which does execute on the pool thread — but then runs the cleanup sender (unstoppable(schedule(task_scheduler))), which schedules the coroutine’s resumption back onto the task’s scheduler. When sync_wait drives the task, that scheduler is its internal run_loop, which runs on the caller thread. The net result: the awaited sender executed on the pool, but the coroutine’s continuation resumes on the caller.

co_await schedule(pool_sch)
  → affine() → finally(schedule(pool_sch), unstoppable(schedule(task_scheduler)))
  → schedule(pool_sch)  — runs on pool thread  ✓
  → unstoppable(schedule(task_scheduler))  — schedules back to run_loop
  → run_loop resumes coroutine on caller thread  ← ✗

If you need the coroutine to actually execute on a different scheduler, use continues_on(coro(), pool_sch) or starts_on(pool_sch, coro()) at the pipeline level, wrapping the entire task.

08 · Receiver Concept

Standard section: [exec.recv] stdexec reference: include/stdexec/concepts.hpp Related tests: tests/test_receiver.cpp

Definition

A Receiver is the result consumer of an asynchronous operation. Standard definition (exec.tex:241):

A receiver is an aggregation of three handlers for the three asynchronous completion dispositions.

cpp
namespace std::execution {
  struct receiver_tag {};

  template<class R>
  concept receiver =
    std::destructible<R> &&
    requires(const std::remove_reference_t<R>& r) {
      { get_env(r) } -> queryable;
    };

  template<class R, class Completions>
  concept receiver_of = /* see below */;

  template<class R, class ChildOp>
  concept inlinable_receiver = /* see below */;
}

receiver Concept

Basic requirements:

  1. Destructible (std::destructible)
  2. Must provide environment queries (get_env(r) returns a queryable object)

Any type satisfying these conditions is a receiver. Note that a receiver does not directly require implementing set_value/set_error/set_stopped — these requirements are expressed in the receiver_of concept.

receiver_of Concept

cpp
// Simplified description
template<class R, class Completions>
concept receiver_of =
  receiver<R> &&
  // R must be able to handle all completion signatures in Completions
  requires(R&& r, /*...*/) {
    set_value(std::forward<R>(r), /*...*/);
    set_error(std::forward<R>(r), /*...*/);
    set_stopped(std::forward<R>(r));
  };

When a sender calls connect(sndr, rcvr), it requires receiver_of<R, completion_signatures_of_t<Sndr, Env>>.

Writing a Receiver

cpp
// Corresponding test: test_receiver.cpp → "just creates sender delivering an int"
struct my_receiver {
  using receiver_concept = stdexec::receiver_tag;  // optional, used for enable_sender

  // handle set_value — template parameters determined by sender's completion signatures
  void set_value(int x) & noexcept {
    result = x;
  }

  // handle set_error
  void set_error(std::exception_ptr e) & noexcept {
    error = std::move(e);
  }

  // handle set_stopped
  void set_stopped() & noexcept {
    cancelled = true;
  }

  // provide environment
  auto get_env() const & noexcept {
    // return an empty queryable (providing no special environment properties)
    return stdexec::empty_env{};
    // or provide custom properties:
    // return stdexec::prop{get_stop_token, never_stop_token{}};
  }

  int result = 0;
  std::exception_ptr error;
  bool cancelled = false;
};

Three Receiver Dispatch Forms

set_value/set_error/set_stopped on a receiver have three valid invocation forms:

FormRequirement
set_value(r, args...) through lvalue rrequires r.set_value(args...)
set_value(std::move(r), args...) through rvalue rrequires std::move(r).set_value(args...)
set_value(static_cast<R&&>(r), args...) through any value categoryrequires static_cast<R&&>(r).set_value(args...)

In practice, senders typically call via static_cast<R&&>(r), meaning your receiver needs to correctly handle move semantics.

inlinable_receiver Concept

cpp
template<class R, class ChildOp>
concept inlinable_receiver =
  receiver<R> &&
  requires (ChildOp& op) {
    { op.start() } noexcept;
  };

inlinable_receiver allows a sender to inline the execution of completion functions on the start() call stack. This is a performance optimization — when a sender can complete synchronously immediately, no asynchronous scheduling is needed.

Receiver Tag Check

stdexec uses receiver_tag to enable compile-time specialization:

cpp
// if the type declares receiver_concept
struct my_receiver {
  using receiver_concept = stdexec::receiver_tag;
  // ...
};

This lets enable_sender specialization know this is a receiver, allowing sender algorithms to correctly match it.

Receiver Environment Propagation

The receiver’s environment is the key mechanism for property propagation through the sender pipeline:

graph TD
  A["Parent Sender"] --> B["connect()"]
  B --> C["Operation State [owns Receiver]"]
  C --> D["get_env(rcvr)"]
  D --> E["Environment"]
  E --> F["get_scheduler(env) → where to execute"]
  E --> G["get_stop_token(env) → whether cancelled"]
  E --> H["get_allocator(env) → how to allocate memory"]

This is how sync_wait knows to provide a run_loop::scheduler to the receiver’s environment.

08 · Receiver Concept

Overview

What is a Receiver? In one sentence: it’s the result consumer of an asynchronous operation.

In traditional async programming, you register three separate callbacks for each operation — success, failure, cancellation:

cpp
// Old way: three independent callbacks, each managing its own lifecycle
async_read(file, buf, size,
  [](int bytes_read)       { /* success */ },
  [](std::exception_ptr e) { /* failure */ },
  []                       { /* cancelled */ }
);

P2300 takes a different approach — it packs all three callbacks into a single object, with the framework managing the lifecycle:

cpp
// P2300: three callbacks bundled into a Receiver
struct file_receiver {
  void set_value(int bytes_read) noexcept        { /* success */ }
  void set_error(std::exception_ptr e) noexcept  { /* failure */ }
  void set_stopped() noexcept                    { /* cancelled */ }
  auto get_env() const noexcept                  { return empty_env{}; }
};

In the code above, file_receiver’s three completion functions replace three independent callbacks. This bundling design allows the framework to check at compile time whether a Receiver can handle all the completion signals a Sender might produce — if the Receiver is missing a completion function, the code simply won’t compile. You won’t discover the mismatch at runtime.

The standard’s constraint on Receiver is defined in [exec.recv]:

A receiver is an aggregation of three handlers for the three asynchronous completion dispositions.

cpp
namespace std::execution {
  template<class R>
  concept receiver =
    std::destructible<R> &&
    requires(const std::remove_reference_t<R>& r) {
      { get_env(r) } -> queryable;
    };

  template<class R, class Completions>
  concept receiver_of = /* see below */;
}

In the code above, the receiver concept only requires two things: destructible and able to provide environment queries. Note that it does not directly require implementing set_value / set_error / set_stopped — those requirements are expressed in the receiver_of concept, which binds a Receiver to specific completion signatures.

Basic Usage

The Simplest Receiver

If you’re certain the Sender will only produce a success, you can define just set_value:

cpp
struct basic_receiver {
  void set_value(int x) noexcept {
    std::cout << "result: " << x << '\n';
  }
  auto get_env() const noexcept {
    return stdexec::empty_env{};
  }
};

// Usage
auto op = stdexec::connect(stdexec::just(42), basic_receiver{});
stdexec::start(op);  // prints: result: 42

In the code above, just(42) is a Sender that only produces set_value(int), so basic_receiver only needs to handle the success case. P2300’s concept mechanism only checks the paths that will actually be taken — it doesn’t force you to write out every possible branch.

Handling Errors

If the Sender might fail, add set_error:

cpp
struct failable_receiver {
  void set_value(int x) noexcept {
    std::cout << "result: " << x << '\n';
  }
  void set_error(std::exception_ptr e) noexcept {
    try {
      std::rethrow_exception(e);
    } catch (const std::exception& ex) {
      std::cerr << "error: " << ex.what() << '\n';
    }
  }
  auto get_env() const noexcept {
    return stdexec::empty_env{};
  }
};

In the code above, set_error receives a std::exception_ptr, which you can rethrow with rethrow_exception to recover the original exception. Note that set_error must also be noexcept — errors are passed via exception pointers, not thrown directly.

Handling Cancellation

If the Sender supports cancellation, you also need set_stopped:

cpp
struct full_receiver {
  void set_value(int x) noexcept { result = x; }
  void set_error(std::exception_ptr e) noexcept { error = std::move(e); }
  void set_stopped() noexcept { cancelled = true; }
  auto get_env() const noexcept { return stdexec::empty_env{}; }

  int result = 0;
  std::exception_ptr error;
  bool cancelled = false;
};

In the code above, we have a complete Receiver implementation — all three completion signals are handled. set_stopped takes no arguments; it simply means “this operation was cancelled.” The specific reason for cancellation is obtained through the stop token (covered in later chapters).

Note that exactly one of the three completion signals will be called. A Sender will never send both success and failure, nor will it notify cancellation after succeeding.

Connecting Receiver to Sender

With a Receiver defined, a Sender can use the receiver_of concept to check compatibility at compile time:

cpp
template<class R, class Completions>
concept receiver_of =
  receiver<R> &&
  requires(R&& r, /* args matching Completions ... */) {
    set_value(std::forward<R>(r), /* ... */);
    set_error(std::forward<R>(r), /* ... */);
    set_stopped(std::forward<R>(r));
  };

In other words, when you call connect(sndr, rcvr), the compiler checks receiver_of<R, completion_signatures_of_t<Sndr, Env>>. If the Receiver can’t handle one of the completion signals the Sender might emit, compilation fails — this is P2300’s compile-time type safety mechanism.

cpp
// Compiles: Receiver can handle int values
auto op1 = connect(just(42), int_receiver{});

// Compile error: Receiver lacks set_stopped
// auto op2 = connect(cancellable_sender, int_receiver{});

In the code above, you can see compile-time checking in action. You don’t need to write static_assert yourself — the connect() concept constraint does the work for you.

Three Dispatch Forms

set_value / set_error / set_stopped on a Receiver have three valid invocation forms:

cpp
// Form 1: called through lvalue
set_value(r, args...);               // requires r.set_value(args...)

// Form 2: called through rvalue
set_value(std::move(r), args...);    // requires std::move(r).set_value(args...)

// Form 3: called through static_cast (most common in practice)
set_value(static_cast<R&&>(r), args...);
// requires static_cast<R&&>(r).set_value(args...)

In the code above, form three is the most common calling convention used by Senders. This means your Receiver must correctly handle move semantics. A common pitfall: accessing a Receiver’s members after it has been std::move’d — the members may have been moved away and are now in an undefined state.

Note that receiver_of checks that all three forms are valid, so you don’t need to manually specify value categories when defining a Receiver — just ensure both & and && versions are correct.

Environment Propagation

A Receiver has another important responsibility — providing environment information to the Sender via get_env(). This is the core mechanism for property propagation in P2300:

cpp
// The environment returned by get_env determines:
//   ① which thread the scheduler executes on
//   ② whether the stop token has been triggered
//   ③ how the allocator should allocate memory

auto env = get_env(my_receiver);

auto scheduler  = get_scheduler(env);      // → which thread
auto stop_token = get_stop_token(env);     // → whether cancelled
auto allocator  = get_allocator(env);      // → how to allocate

In the code above, you can see the three core queries on the environment. This information propagates through the entire Sender pipeline — a parent Sender passes get_env(rcvr) down to child Senders, and each child decides its behavior based on the result.

The simplest environment implementation just returns stdexec::empty_env{}, providing no special properties:

cpp
auto get_env() const noexcept {
  // No scheduler, stop token, or allocator — use defaults
  return stdexec::empty_env{};
}

// You can also inject custom properties into the environment:
// return stdexec::prop{get_stop_token, stdexec::never_stop_token{}};

In the code above, if you return empty_env, the framework uses defaults — execute on the current thread, ignore cancellation, use the default allocator.

inlinable_receiver

cpp
template<class R, class ChildOp>
concept inlinable_receiver =
  receiver<R> &&
  requires (ChildOp& op) {
    { op.start() } noexcept;
  };

inlinable_receiver is a performance optimization marker. When a Sender can complete synchronously, inlinable_receiver allows the framework to call the completion function directly on the start() call stack, skipping asynchronous scheduling. This is particularly useful with synchronous Senders like inline_scheduler or just().

receiver_tag

If you want to use stdexec’s enable_sender mechanism, declare receiver_concept:

cpp
struct my_receiver {
  using receiver_concept = stdexec::receiver_tag;
  // ...
};

In the code above, stdexec can now recognize your type as a Receiver at compile time, allowing Sender algorithms to match it correctly.

Important Notes

  1. The receiver concept does not require completion functions — those requirements come from receiver_of, which checks against specific completion signatures
  2. Exactly one completion signal will be called — a single operation will not both succeed and fail
  3. Don’t use a Receiver after it’s been moved — Senders typically call via static_cast<R&&>, so members may have been moved away
  4. get_env() should return a lightweight object — typically returned by value, just a few bytes in size

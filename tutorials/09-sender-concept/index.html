09 · Sender Concept

Standard section: [exec.snd.concepts] [exec.snd.general] stdexec reference: include/stdexec/concepts.hpp Related tests: tests/test_basics.cpp · tests/test_receiver.cpp

Definition

Standard definition (exec.tex:290):

A sender is a factory for one or more asynchronous operations.

cpp
namespace std::execution {
  struct sender_tag {};

  template<class Sndr>
  inline constexpr bool enable_sender = /* see below */;

  template<class Sndr>
  concept sender =
    enable_sender<std::remove_cvref_t<Sndr>> &&
    requires(const std::remove_cvref_t<Sndr>& sndr) {
      { get_env(sndr) } -> queryable;
    };

  template<class Sndr, class... Env>
  concept sender_in =
    sender<Sndr> &&
    requires (Sndr&& sndr, Env&&... env) {
      { get_completion_signatures<Sndr, Env...>() }
          -> valid-completion-signatures;
    };
}

enable_sender

For a type to be a sender, enable_sender must be explicitly enabled. The standard provides several ways:

cpp
// Way 1: specialize enable_sender
template<>
inline constexpr bool enable_sender<my_sender> = true;

// Way 2: declare sender_concept member type (stdexec convention)
struct my_sender {
  using sender_concept = stdexec::sender_tag;
  // ...
};

// Way 3: inherit from sender_adaptor_closure (for adaptor closures)
struct my_adaptor : stdexec::sender_adaptor_closure<my_adaptor> {
  // ...
};

sender_in — Environment-aware Sender

A sender’s completion signatures may depend on the environment:

cpp
// Environment-independent sender (most common)
using CS = completion_signatures_of_t<just<42>>;
// CS = completion_signatures<set_value_t(int)>

// Environment-dependent sender
using CS = completion_signatures_of_t<read_env_t, empty_env>;
// CS = completion_signatures<set_value_t(/*...*/)>

sender_in<Sndr, Env...> checks whether the sender can determine valid completion signatures given the environment Env.

tag_of_t

Every sender has an associated tag type identifying which kind of sender algorithm it is:

cpp
template<sender Sndr>
using tag_of_t = /* ... */;

tag_of_t<just<42>>        → just_t
tag_of_t<then_t, Sndr, F> → then_t
tag_of_t<schedule_t, Sch> → schedule_t

This tag is used in the domain mechanism to identify which domain should handle the sender’s transformation.

Sender Classification

From the perspective of whether completion signatures depend on the environment:

CategoryDescriptionExample
Non-dependent senderCompletion signatures determinable without environmentjust(42), then(sndr, f)
Dependent senderCompletion signatures depend on environmentread_env(get_scheduler); schedule(sch) is dependent (though its completion signatures are unaffected by environment, making it a special case of non-dependent)

More precisely, the dependent_sender concept:

cpp
template<class Sndr>
concept dependent_sender =
  sender<Sndr> &&
  !sender_in<Sndr>;

A sender is dependent if it needs environment information to determine its completion signatures. For example, a sender returned by write_env(sndr, prop) may be dependent.

Three Categories of Sender

The standard formally defines three sender algorithms in exec.tex:380-398:

cpp
// Sender Factory — create sender from non-sender
sender auto just(auto... args);
sender auto schedule(scheduler auto sch);
sender auto read_env(query auto q);

// Sender Adaptor — take sender(s) and return new sender
sender auto then(sender auto sndr, invocable auto f);
sender auto when_all(sender auto... sndrs);

// Sender Consumer — consume sender, return non-sender
auto sync_wait(sender auto sndr) -> optional<tuple<...>>;
void spawn(sender auto sndr);

Sender Properties and Environment

Every sender has attributes, queryable via get_env(sndr):

cpp
template<sender Sndr>
using env_of_t = decltype(get_env(declval<Sndr>()));

// query the sender's completion scheduler
get_completion_scheduler<set_value_t>(get_env(sndr));

Minimal Sender Implementation

Here is a complete minimal sender (in the style of examples/algorithms/then.hpp):

cpp
struct my_sender {
  using sender_concept = stdexec::sender_tag;

  // declare completion signatures
  using completion_signatures =
    stdexec::completion_signatures<
      stdexec::set_value_t(int)
    >;

  // connect method — create operation_state
  template<stdexec::receiver R>
  auto connect(R r) && -> my_op<R> {
    return my_op<R>{std::move(r)};
  }

  // optional: provide sender properties
  auto get_env() const noexcept {
    return stdexec::empty_env{};
  }
};

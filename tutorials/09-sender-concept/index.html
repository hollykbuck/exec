09 · Sender Concept

Overview

What is a Sender? In one sentence: it’s a factory for asynchronous operations.

With traditional async APIs, each call executes immediately:

cpp
thread_pool.submit(work);   // called and executed — no composability

A Sender is different — calling just(42) only produces a job description. No code runs yet. Execution only begins after you connect a Receiver and call start():

cpp
sender auto work = stdexec::just(42)         // ① create job description (doesn't run)
                 | stdexec::then([](int x) { // ② chain next step (doesn't run)
                     return x * 2;
                   });

auto [result] = stdexec::sync_wait(work).value();  // ③ execution begins here

In the code above, you can see the core design philosophy of Sender: description is separated from execution. The first three lines construct a pipeline — like drawing up blueprints without lifting a hammer. Only the final sync_wait triggers actual execution. This design lets you compose async operations as freely as building blocks.

The standard’s definition of Sender is found in [exec.snd.concepts]:

A sender is a factory for one or more asynchronous operations.

cpp
namespace std::execution {
  template<class Sndr>
  inline constexpr bool enable_sender = /* see below */;

  template<class Sndr>
  concept sender =
    enable_sender<std::remove_cvref_t<Sndr>> &&
    requires(const std::remove_cvref_t<Sndr>& sndr) {
      { get_env(sndr) } -> queryable;
    };
}

In the code above, the sender concept requires a type to have enable_sender enabled and to be able to provide environment queries. Note that sender itself does not require connect — that requirement lives in the sender_to concept, checked together with the Receiver.

enable_sender

For a type to be a Sender, you must explicitly enable enable_sender. The standard offers several ways:

cpp
// Way 1: specialize enable_sender (most direct)
template<>
inline constexpr bool std::execution::enable_sender<my_sender> = true;

// Way 2: declare sender_concept member type (stdexec recommended)
struct my_sender {
  using sender_concept = stdexec::sender_tag;
  // ...
};

// Way 3: inherit from sender_adaptor_closure (for adaptor closures)
struct my_adaptor : stdexec::sender_adaptor_closure<my_adaptor> {
  // ...
};

In the code above, way two is the most common approach in stdexec — declaring sender_concept causes stdexec’s trait system to automatically specialize enable_sender for you. Way three is used for implementing the pipe operator | adaptors, which will be covered in later chapters.

How a Sender Is “Activated”

A Sender is lazy by nature. It requires two stages before execution:

cpp
// ① connect: bind Sender and Receiver, create operation_state
auto op = stdexec::connect(some_sender, some_receiver);

// ② start: launch the async operation (must be noexcept, at most once)
stdexec::start(op);

// After the async operation completes, one of the Receiver's
// set_value / set_error / set_stopped will be called

In the code above, you can see the two-stage activation model of Sender. connect() may throw (e.g. memory allocation failure), but start() must be noexcept — all potentially-failing work should be done during connect().

Three Roles of Sender

P2300 classifies Senders into three categories by function:

Factory

Factories create new async operations from non-Sender arguments:

cpp
sender auto s1 = stdexec::just(42);                    // immediately produce value 42
sender auto s2 = stdexec::just_error(ex_ptr);           // immediately produce error
sender auto s3 = stdexec::schedule(sch);                // no-op on a scheduler
sender auto s4 = stdexec::read_env(get_scheduler);      // read scheduler from env

In the code above, just(42) is the simplest factory — when start() is called, it synchronously sends set_value(42). schedule(sch) sends an empty set_value() on the scheduler’s execution agent.

Adaptor

Adaptors take one or more Senders and return a new Sender:

cpp
sender auto s5 = stdexec::then(s1, [](int x) {        // value transformation
  return x * 2;
});
sender auto s6 = stdexec::upon_error(s2, [](auto e) { // error recovery
  return 0;
});
sender auto s7 = stdexec::when_all(s1, s3);           // concurrent wait

In the code above, then passes the child Sender’s result to the callback; upon_error recovers from failure; when_all waits for all child Senders to complete and merges results.

Consumer

Consumers consume a Sender and return a non-Sender type:

cpp
auto result = stdexec::sync_wait(s5);      // synchronous wait, returns optional<tuple<...>>
stdexec::spawn(s5);                        // fire-and-forget, returns immediately

In the code above, sync_wait blocks the current thread until the Sender completes and returns the result. spawn launches the Sender asynchronously and returns immediately — suitable for “fire and forget” scenarios.

Environment Dependency

Most Senders have completion signatures that are known at compile time, without needing runtime environment information:

cpp
using CS = stdexec::completion_signatures_of_t<stdexec::just<42>>;
// CS = completion_signatures<set_value_t(int)>
// No environment needed to know: succeeds with int, never fails, never cancels

However, some Senders require environment context to determine their completion signatures:

cpp
// read_env(get_scheduler) needs the environment to know what scheduler type
// it will return — its completion signatures depend on the caller's environment

The standard uses the sender_in concept to distinguish these two cases:

cpp
template<class Sndr, class... Env>
concept sender_in =
  sender<Sndr> &&
  requires (Sndr&& sndr, Env&&... env) {
    { get_completion_signatures<Sndr, Env...>() }
        -> valid-completion-signatures;
  };

In the code above, if sender_in<Sndr, Env> holds, it means that given environment Env, you can determine Sndr’s completion signatures. A Sender that satisfies sender_in without any environment is a non-dependent Sender (the most common category).

Implementing a Minimal Sender

Here is a complete minimal Sender implementation — it can be consumed normally by sync_wait:

cpp
struct my_sender {
  using sender_concept = stdexec::sender_tag;

  // Declare completion signatures: only produces set_value(int)
  using completion_signatures =
    stdexec::completion_signatures<
      stdexec::set_value_t(int)
    >;

  // connect: create operation_state
  template<stdexec::receiver R>
  auto connect(R r) && -> my_op<R> {
    return my_op<R>{std::move(r)};
  }

  // get_env: provide Sender attributes
  auto get_env() const noexcept {
    return stdexec::empty_env{};
  }
};

// operation_state: actually executes the async logic
template<stdexec::receiver R>
struct my_op {
  R rcvr_;

  void start() & noexcept {
    // Synchronous completion: directly call set_value
    stdexec::set_value(static_cast<R&&>(rcvr_), 42);
  }
};

In the code above, we’ve defined a minimal Sender. Its connect creates an operation_state, and start synchronously calls set_value(rcvr_, 42). Simple as it is, it contains all three core components of a Sender: completion_signatures (type identity), connect (bind to Receiver), and get_env (environment query).

Sender Attributes and tag_of_t

Every Sender also has an associated tag type identifying which Sender algorithm it belongs to:

cpp
template<sender Sndr>
using tag_of_t = /* ... */;

tag_of_t<stdexec::just<42>>              // → just_t
tag_of_t<stdexec::then_t, Sndr, F>       // → then_t
tag_of_t<stdexec::schedule_t, Sch>       // → schedule_t

This tag is used in the domain mechanism — the framework uses tag_of_t to decide which domain should handle a Sender’s transformation. Ordinary users generally don’t need to worry about it, but understanding it helps when learning about the transform_sender customization point later.

Important Notes

  1. Sender ≠ async operation — a Sender is just a factory. The same Sender can be connect-ed multiple times, each producing a brand-new operation_state
  2. Senders are lazy — without connect + start, no code ever runs
  3. enable_sender must be explicitly enabled — defining a random class does not make it a Sender; you must specialize enable_sender or declare sender_concept
  4. Completion signatures are determined at compile time — this means sync_wait’s return type is deduced at compile time with zero RTTI overhead

10 · Scheduler Concept

Overview

What is a Scheduler? In one sentence: it’s an abstraction of an execution resource — thread pool, GPU stream, event loop — all accessed through the same schedule() interface.

In traditional async programming, every execution resource has a different API:

cpp
// Three resources, three different interfaces
thread_pool.submit(work);       // thread pool
cuda_stream.launch(kernel);     // GPU
event_loop.post(callback);      // event loop

P2300 takes a different approach: regardless of the underlying resource, you always call schedule(sch), which returns a Sender:

cpp
// Unified interface: schedule(sch) returns a Sender
sender auto s1 = schedule(thread_pool_scheduler);
sender auto s2 = schedule(gpu_scheduler);
sender auto s3 = schedule(event_loop_scheduler);
// All three used identically — differences encapsulated inside the scheduler

In the code above, you can see the core value of Scheduler: unified abstraction. Upper-layer code doesn’t care whether the backend is CPU threads, GPU, or an IO event loop — just get a Scheduler, call schedule(sch) to get a Sender, then express your business logic as a Sender pipeline.

The standard’s definition of Scheduler is in [exec.sched]:

A scheduler is an abstraction of an execution resource with a uniform, generic interface for scheduling work onto that resource.

cpp
namespace std::execution {
  template<class Sch>
  concept scheduler =
    std::destructible<Sch> &&
    std::copy_constructible<Sch> &&
    std::equality_comparable<Sch> &&
    requires(Sch&& sch, const Sch& csch) {
      { schedule(static_cast<Sch&&>(sch)) } -> sender;
      { csch.query(get_forward_progress_guarantee) }
        -> same_as<forward_progress_guarantee>;
    };
}

In the code above, the scheduler concept requires three things: value semantics (destructible, copyable, comparable), schedule(sch) returns a Sender, and support for querying the concurrency guarantee.

What schedule() Does

schedule(sch) returns a Sender that does three things:

  1. Acquires an execution agent on the resource associated with the scheduler
  2. Calls set_value(rcvr) (no arguments) on that execution agent
  3. After completion, the value completion scheduler is sch itself
cpp
scheduler auto sch = pool.get_scheduler();

// "Do nothing" on the thread pool — just switch execution context
sender auto s = schedule(sch);

// Practical usage: execute business logic on a thread pool thread
sender auto work = schedule(sch)
                 | then([] {
                     // This code runs on a thread pool thread
                     return heavy_computation();
                   });

In the code above, schedule(sch) itself does no meaningful work — it only ensures the empty set_value() is sent on the correct execution resource. The actual business logic goes in the subsequent then.

Three Concurrency Guarantees

Every Scheduler must declare its concurrency characteristics:

cpp
enum class forward_progress_guarantee {
  concurrent,       // may execute concurrently, but may block each other
  parallel,         // true parallel execution (multi-core)
  weakly_parallel   // weak parallelism (e.g. GPU warp)
};

Concurrency guarantees for different schedulers:

SchedulerGuaranteeNotes
inline_schedulerconcurrentExecutes synchronously on the current thread
static_thread_poolparallelMultiple threads running truly in parallel
nvexec::stream_schedulerweakly_parallelGPU stream execution

In the table above, concurrent is the most basic guarantee — multiple tasks may be executing at the same time, but tasks on the same thread must queue sequentially. parallel is a stronger guarantee — it means true parallel access through lock-free data structures is possible. weakly_parallel applies to highly constrained execution environments like GPUs.

Built-in Schedulers

inline_scheduler

inline_scheduler is the simplest scheduler — its start() calls set_value directly on the current call stack:

cpp
exec::inline_scheduler sch;

// The Sender returned by schedule(sch) completes synchronously in start()
auto op = connect(schedule(sch), my_receiver{});
start(op);  // set_value(rcvr) is called synchronously on this call stack
            // When start() returns, the Receiver has already received the result

In the code above, inline_scheduler is fully synchronous. It’s mainly used for testing, or as a “placeholder scheduler” in scenarios that don’t need asynchronous dispatch.

static_thread_pool

static_thread_pool creates a fixed-size thread pool:

cpp
exec::static_thread_pool pool{4};  // 4 worker threads
auto sch = pool.get_scheduler();

sender auto work = schedule(sch)
                 | then([] {
                     // Executes on one of the thread pool's threads
                     return some_work();
                   });

// Flow of schedule(sch):
// 1. start() submits the work to the thread pool queue, then returns immediately
// 2. An idle thread in the pool picks up the work
// 3. That thread calls set_value(rcvr)

In the code above, start() returns immediately and the actual work completes on a background thread. You don’t need to manually manage thread creation and destruction — static_thread_pool cleans up automatically in its destructor.

run_loop

run_loop is a single-threaded event loop — you must manually invoke loop.run() to drive it:

cpp
stdexec::run_loop loop;
auto sch = loop.get_scheduler();

// Submit work to the run_loop queue (doesn't execute immediately)
sender auto work = schedule(sch)
                 | then([] { return delayed_task(); });
auto op = connect(work, my_receiver{});
start(op);
// Now the work is in the queue, but hasn't executed yet

// Run the event loop — processes queued work
loop.run();  // returns after all work is done

In the code above, run_loop is suitable for scenarios requiring precise control over execution timing — for example, simulating a single-threaded environment in tests, or using a main loop to drive all async operations in an embedded system.

Comparison

The core difference between the three schedulers — when set_value gets called after start():

cpp
// inline_scheduler:    set_value called before start() returns
// static_thread_pool:  start() returns immediately, set_value called on another thread
// run_loop:            start() returns immediately, set_value called inside loop.run()

The comments above summarize the timing differences. Understanding these differences is essential for using each scheduler correctly.

Implementing a Custom Scheduler

Implementing a custom scheduler requires two parts: the scheduler class itself, and the Sender returned by schedule().

Step 1: Define the Scheduler Class

cpp
struct my_scheduler {
  // Declare concurrency guarantee
  friend auto tag_invoke(
    stdexec::tag_t<stdexec::get_forward_progress_guarantee>,
    const my_scheduler&) noexcept -> stdexec::forward_progress_guarantee {
    return stdexec::forward_progress_guarantee::concurrent;
  }

  // Value semantics: support equality comparison
  bool operator==(const my_scheduler&) const = default;
};

In the code above, get_forward_progress_guarantee is the scheduler’s “ID card” — the framework uses it to assess the scheduler’s concurrency abilities. We chose concurrent because our scheduler executes synchronously on the current thread.

Step 2: Implement the schedule Sender

cpp
struct my_schedule_sender {
  using sender_concept = stdexec::sender_tag;

  // Declare completion signatures
  using completion_signatures =
    stdexec::completion_signatures<stdexec::set_value_t()>;

  template<stdexec::receiver R>
  struct operation {
    R rcvr_;
    void start() & noexcept {
      // Call set_value on this execution resource
      stdexec::set_value(static_cast<R&&>(rcvr_));
    }
  };

  template<stdexec::receiver R>
  auto connect(R r) && -> operation<R> {
    return {std::move(r)};
  }
};

In the code above, the Sender returned by schedule has a completion signature of set_value_t() — an empty-argument success signal. This is the convention for all schedule() Senders: they only switch the execution context and carry no business data.

Step 3: Implement schedule via tag_invoke

cpp
my_schedule_sender tag_invoke(
  stdexec::tag_t<stdexec::schedule>, const my_scheduler&) {
  return {};
}

In the code above, schedule is a CPO (Customization Point Object), invoked through the tag_invoke mechanism. As long as you define tag_invoke(schedule_t, my_scheduler), schedule(my_scheduler) will work correctly.

Scheduler Responsibility Boundaries

A scheduler does exactly one thing: provides the Sender returned by schedule(sch), and guarantees that after start(), set_value will be called on the correct execution resource.

A scheduler is not responsible for:

  • Switching execution contexts — that’s the job of starts_on / continues_on adaptors
  • Managing operation_state lifecycle — that’s the job of the connect / start protocol
  • Deciding how completion signals propagate — that’s the job of Sender adaptors
cpp
// Wrong: Scheduler is responsible for context switching
// Right: continues_on uses the Scheduler to switch contexts

// scheduler provides the execution resource
// continues_on decides "at which point to switch to that resource"
sender auto on_pool = continues_on(work, pool.get_scheduler());

In the code above, scheduler and adaptor each have their own role — the scheduler provides the resource, and the adaptor decides when to use it.

Important Notes

  1. forward_progress_guarantee is a compile-time constant — it affects the framework’s optimization decisions and static checks, not runtime state
  2. Schedulers should be lightweight value types — copying a Scheduler should be cheap, typically just a pointer to the execution resource
  3. schedule(sch) always sends set_value() — this is a standard convention; don’t expect a schedule Sender to send other completion signals
  4. Schedulers don’t handle cancellation — cancellation is handled by the stop token mechanism, covered in later chapters

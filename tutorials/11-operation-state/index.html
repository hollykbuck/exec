11 · Operation State Lifecycle

Overview

What is an Operation State? In one sentence: it’s the actual async operation entity created by connecting a Sender and a Receiver.

As we’ve seen, a Sender is a “job description” — it tells you what to do, but doesn’t do it itself. Only when a Sender is bound to a Receiver through connect is a real async operation object produced. That object is the Operation State:

cpp
sender auto work = just(42) | then([](int x) { return x * 2; });

// connect creates operation_state — resources allocated, state stored
auto op = stdexec::connect(work, my_receiver{});

// start launches the async operation
stdexec::start(op);

In the code above, connect is the resource allocation point and start is the logic launch point. Separating the two is a key P2300 design decision — all potentially-failing work (like memory allocation) happens during connect, and start must be noexcept.

The standard’s definition of Operation State is in [exec.opstate]:

cpp
namespace std::execution {
  template<class O>
  concept operation_state =
    std::destructible<O> &&
    std::is_object_v<O> &&
    requires(O& o) {
      { start(o) } noexcept;
    };
}

In the code above, the operation_state concept is remarkably concise — destructible, an object type, start() must be noexcept. Nothing more.

The Four Lifecycle Stages

Annotated with line comments, an Operation State’s lifecycle has exactly four stages:

cpp
// ① connect: create operation_state (may throw)
auto op = connect(sender, receiver);

// ② start: launch the async operation (at most once, must be noexcept)
start(op);

// ③ async execution in progress...
//    → success: set_value(receiver, result)
//    → failure: set_error(receiver, exception_ptr)
//    → cancellation: set_stopped(receiver)

// ④ after completion, op is immediately invalid. This is undefined behavior:
// start(op);  // UB! op is already dead

In the code above, the key point is that after start() is called, op’s fate depends entirely on when the async operation completes. If it completes synchronously (like just()), op is already invalid by the time start() returns.

Three Strict Rules

Rule 1: start() may be called at most once

cpp
auto op = connect(my_sender, my_receiver);
start(op);   // OK
start(op);   // undefined behavior!

In the code above, you can see a common mistake — calling start() twice. An operation_state is one-shot by design: an async operation either succeeds, fails, or is cancelled. There is no “replay.” If you need retry, you must create a new operation_state.

Rule 2: op may be destroyed during completion function execution

cpp
struct my_operation {
  R rcvr_;

  void start() & noexcept {
    // Inside set_value, rcvr_'s owner may free this
    set_value(std::move(rcvr_), 42);
    // At this point, `this` may already be freed!
  }
};

In the code above, set_value may internally trigger the Receiver’s destruction logic (e.g. sync_wait releases the operation_state upon receiving the result). This means after set_value returns, this may already be invalid — you must not access any member variables.

This is one of the most important safety rules in P2300: the completion function is a firebreak — once you cross it, there’s no turning back.

Rule 3: op is immediately invalid after completion

cpp
auto op = connect(snd, rcvr);
start(op);

// If start() completed synchronously, op is already invalid
// do_something(op);  // undefined behavior!

In the code above, synchronously-completing Senders (like just) call the completion function before start() returns, meaning op is already in an invalid state when start() returns. Accessing a completed operation_state is undefined behavior.

Internal Structure

A typical operation_state holds copies of both Sender and Receiver:

cpp
template<class S, class R>
struct my_operation {
  S sndr_;           // copy of the Sender (job description)
  R rcvr_;           // copy of the Receiver (result consumer)

  my_operation(S s, R r)
    : sndr_(std::move(s))
    , rcvr_(std::move(r))
  {}

  void start() & noexcept {
    // Execute business logic
    // On completion, call set_value / set_error / set_stopped
    try {
      auto result = sndr_.invoke();
      stdexec::set_value(std::move(rcvr_), result);
    } catch (...) {
      stdexec::set_error(std::move(rcvr_), std::current_exception());
    }
  }
};

In the code above, the operation_state owns copies of the Sender and Receiver. Note that even if the business logic inside start() throws, it’s converted to a set_error call — an exception will never escape from start(), because it’s marked noexcept.

The connect Return Type

cpp
template<class Sndr, class Rcvr>
  requires sender_to<Sndr, Rcvr>
using connect_result_t = decltype(connect(declval<Sndr>(), declval<Rcvr>()));

// for just(42) + my_receiver:
auto op = stdexec::connect(stdexec::just(42), my_receiver{});
// connect_result_t = some internal operation class

In the code above, connect_result_t is a compile-time deduced type — you typically don’t need to write it out manually because auto suffices. It’s mainly used in template metaprogramming scenarios, like reconstructing a new operation_state inside a retry algorithm.

The Retry Pattern: Managing Re-creation with optional

Since an operation_state is one-shot, retrying means destroy first, then reconstruct. The stdexec retry algorithm uses std::optional to implement this pattern:

cpp
// The wrong retry — directly calling start again
auto op = connect(snd, rcvr);
start(op);
if (need_retry) {
  start(op);  // UB! op is already invalid
}

In the code above, you can see the intuitive mistake — “it failed, so I’ll just start again.” Nope — start can only be called once. The correct approach:

cpp
template<class S, class R>
struct retry_op {
  S s_;
  R r_;
  // Use optional to manage the nested operation_state
  std::optional<connect_result_t<S&, retry_receiver<S, R>>> inner_op_;

  retry_op(S s, R r)
    : s_(std::move(s))
    , r_(std::move(r))
    , inner_op_(std::in_place, connect(s_, retry_receiver{this}))
  {}

  void start() & noexcept {
    start(*inner_op_);  // launch the nested operation
  }

  void retry() noexcept {
    inner_op_.reset();                                  // destroy the old
    inner_op_.emplace(connect(s_, retry_receiver{this})); // reconstruct
    start(*inner_op_);                                  // relaunch
  }
};

In the code above, the std::optional reset + emplace pattern enables safe reconstruction of the operation_state. This pattern is very useful in scenarios requiring retry, looping, or any situation where you need to “connect the same Sender multiple times.”

Why start() Must Be noexcept

The noexcept requirement on start() is not arbitrary — there are three design reasons behind it:

cpp
// connect phase: may throw, dynamic memory allocation allowed
auto op = connect(snd, rcvr);  // may throw std::bad_alloc

// start phase: must be noexcept, only the launch action
start(op);  // must not throw

In the code above, you can see the design intent: allocation and launch are separated. connect handles all potentially-failing work (memory allocation, parameter validation), and start does only the pure “launch” action. If the async operation encounters a runtime error, it’s delivered via set_error, not thrown.

cpp
void start() & noexcept {
  try {
    auto result = do_work();
    set_value(std::move(rcvr_), result);
  } catch (...) {
    set_error(std::move(rcvr_), std::current_exception());
  }
}

In the code above, start() may use try-catch internally to ensure exceptions never escape its call boundary. Business logic exceptions are uniformly converted to set_error delivery.

Key Design Decisions

DecisionReason
start() is noexceptResource allocation done in connect(); start() only launches
start() may only be called onceEnforces a one-shot lifecycle; prevents reuse of a completed op
op may be destroyed during completionAllows the Receiver to release resources immediately upon receiving the result
connect() may throwThe connection phase may involve memory allocation and other fallible work

The table above summarizes the four core design decisions of Operation State. Understanding these rules means understanding P2300’s resource management philosophy — separate allocation from launch, one-shot usage, firebreak pattern.

Important Notes

  1. Don’t use op after start() — even if start() returns, it may have synchronously completed, leaving op invalid
  2. Don’t write code after set_value — the completion function may destroy this; once you cross the completion function call, no member access is safe
  3. std::optional is the standard tool for reconstructing operation_state — use reset + emplace for retry or looping
  4. For type-erased operation_state, use any_sender — if you need runtime polymorphism, use stdexec’s type erasure utilities

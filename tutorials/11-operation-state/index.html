11 · Operation State Lifecycle

Standard section: [exec.opstate] [exec.connect] stdexec reference: examples/then.cpp Related tests: tests/test_basics.cpp, examples/retry.cpp

Definition

An operation_state is the actual asynchronous operation entity created by connecting a sender and a receiver.

Standard definition (exec.tex:230,290):

Connecting a sender and a receiver creates an asynchronous operation.

cpp
namespace std::execution {
  struct operation_state_tag {};

  template<class O>
  concept operation_state =
    std::destructible<O> &&
    std::is_object_v<O> &&
    requires(O& o) {
      { start(o) } noexcept;
    };
}

Key requirement: start() must be noexcept.

Lifecycle Rules

The standard (exec.tex:270-287) specifies these strict rules:

graph TD
  A["Sender + Receiver"]
  A --> B["connect()"]
  B --> C["Operation State ← lifecycle starts<br/>owns sender + receiver"]
  C --> D["start() ← at most once, noexcept"]
  D --> E["[asynchronous execution]"]
  E --> F["set_value / set_error / set_stopped"]
  F --> G["Operation State invalid ← lifecycle ends"]
  G --> H["destruction"]

Rule 1: start() may be called at most once

cpp
auto op = connect(my_sender, my_receiver);
start(op);   // OK
start(op);   // undefined behavior!

Rule 2: The operation_state may be destroyed while a completion function is executing

cpp
struct my_op {
  std::shared_ptr<some_resource> resource;
  R rcvr_;

  void start() & noexcept {
    // inside the async operation's completion callback:
    set_value(std::move(rcvr_), 42);
    // at this point `this` may already be invalid (if rcvr_'s owner released the op)
  }
};

Rule 3: After the operation completes, the operation_state is immediately invalid

cpp
// error: accessing op after completion
auto op = connect(snd, rcvr);
start(op);
// if start() completes synchronously, op is already invalid
// do_something(op);  // UB

Internal Structure of an Operation State

A typical operation_state contains:

cpp
template<class S, class R>
struct my_operation {
  S sndr_;           // copy of the sender
  R rcvr_;           // copy of the receiver

  my_operation(S s, R r)
    : sndr_(std::move(s))
    , rcvr_(std::move(r))
  {}

  void start() & noexcept {
    // execute business logic
    // on completion, call set_value / set_error / set_stopped
    try {
      auto result = sndr_.invoke();
      stdexec::set_value(std::move(rcvr_), result);
    } catch (...) {
      stdexec::set_error(std::move(rcvr_), std::current_exception());
    }
  }
};

connect Types

cpp
template<class Sndr, class Rcvr>
  requires sender_to<Sndr, Rcvr>
using connect_result_t = decltype(connect(declval<Sndr>(), declval<Rcvr>()));

// for just(42) + my_receiver:
auto op = stdexec::connect(just(42), my_receiver{});
// connect_result_t = some internal operation class

Lifecycle Management: std::optional Pattern

The retry algorithm (examples/algorithms/retry.hpp) shows an important pattern — using std::optional to manage the operation state lifecycle:

cpp
template<class S, class R>
struct op {
  S s_;
  R r_;
  // use optional to store operation state, allowing reconstruction on retry
  std::optional<connect_result_t<S&, retry_receiver<S, R>>> o_;

  op(S s, R r)
    : s_(std::move(s))
    , r_(std::move(r))
    , o_(std::in_place, connect(s_, retry_receiver{this}))
  {}

  void start() & noexcept {
    start(*o_);  // start nested operation state
  }

  void retry() noexcept {
    o_.emplace(connect(s_, retry_receiver{this}));
    start(*o_);  // restart
  }
};

Importance of noexcept

start() must be noexcept. This means:

  1. Operation state creation (connect) must do all potentially throwing work
  2. start() only performs the “launch” action, no “allocation”
  3. If the async operation encounters an error, it is delivered via set_error, not thrown from start()
cpp
void start() & noexcept {
  try {
    auto result = do_work();       // this is fine
    set_value(std::move(rcvr_), result);
  } catch (...) {
    set_error(std::move(rcvr_), std::current_exception());
  }
}

12 · Completion Signatures

Overview

What are Completion Signatures? In one sentence: they are a Sender’s compile-time type identity card.

Every Sender has an “identity card” that states three things: what type it returns on success, what errors it might produce, and whether it can be cancelled. This identity card is established at compile time — no runtime information needed:

cpp
// just(42)'s "identity card"
using sigs = stdexec::completion_signatures_of_t<
  decltype(stdexec::just(42))
>;
// sigs = completion_signatures<set_value_t(int)>
// Reads: succeeds with int, never fails, never cancels

// just_error(...)'s "identity card"
using err_sigs = stdexec::completion_signatures_of_t<
  decltype(stdexec::just_error(std::exception_ptr{}))
>;
// err_sigs = completion_signatures<set_error_t(std::exception_ptr)>
// Reads: never succeeds, fails with exception_ptr, never cancels

In the code above, you can see the core purpose of Completion Signatures: knowing all possible outcomes of a Sender at compile time. This lets connect() check whether the Receiver matches, and lets sync_wait deduce the correct return type — all without RTTI.

The standard’s definition of Completion Signature is in [exec.cmplsig]:

A completion signature is a function type that describes a completion operation.

cpp
// Three forms of completion signature
set_value_t(int, double)           // success: returns int and double
set_error_t(std::exception_ptr)    // failure: returns exception_ptr
set_stopped_t()                     // cancellation: no arguments

In the code above, a completion signature is essentially a function type — set_value_t(int, double) reads as “on success, sends an int and a double.” Multiple completion signatures are grouped together in the completion_signatures template:

cpp
namespace std::execution {
  template<completion-signature... Fns>
  struct completion_signatures {};
}

Why Compile-Time Type Description

You might wonder: why put completion information at compile time? Wouldn’t a runtime query suffice?

cpp
// sync_wait's return type is deduced entirely at compile time
sender auto work = stdexec::just(42) | stdexec::then([](int x) {
  return std::to_string(x);
});

// The compiler knows work's completion signature is set_value_t(string)
// So sync_wait's return type is optional<tuple<string>> — at compile time
auto result = stdexec::sync_wait(work);  // type already determined, zero overhead

In the code above, sync_wait needs no runtime type information. The compiler deduces that work completes with string, and the return type is optional<tuple<string>>. If you used runtime types (like std::any), you’d pay for boxing and unboxing on every call — with enormous performance overhead.

sync_wait’s Deduction Chain

Let’s trace how sync_wait deduces its return type step by step:

cpp
// Step 1: get the completion signatures
using Sigs = stdexec::completion_signatures_of_t<decltype(work)>;
// Sigs = completion_signatures<set_value_t(string)>

// Step 2: extract the value types
using ValueTypes = stdexec::value_types_of_t<decltype(work)>;
// ValueTypes = variant<tuple<string>>

// Step 3: sync_wait uses ValueTypes to determine the return type
// Return type = optional<tuple<string>>

In the code above, you can see three compile-time deduction steps. Each step is done via template metaprogramming at compile time, and the generated runtime code has zero type erasure overhead.

connect’s Type Check

Completion Signatures also power connect’s compile-time safety check:

cpp
// just(42)'s completion signature: set_value_t(int)
// Receiver must have set_value(int), or compilation fails

// Compiles: Receiver can handle int
auto op1 = connect(just(42), int_receiver{});

// Compile error: Receiver lacks set_error
// auto op2 = connect(just_error(...), int_receiver{});

In the code above, the compiler checks receiver_of<R, completion_signatures_of_t<Sndr>> at the connect call site. If the Receiver is missing any required completion function, compilation fails immediately — the error won’t be deferred to runtime.

Querying Completion Signatures

cpp
template<class Sndr, class... Env>
  requires sender_in<Sndr, Env...>
using completion_signatures_of_t =
  decltype(get_completion_signatures<Sndr, Env...>());

// Usage
using CS = stdexec::completion_signatures_of_t<
  decltype(stdexec::just(42))
>;
// CS = completion_signatures<set_value_t(int)>

Extracting specific types from completion signatures:

cpp
// Extract value types of all value completion signatures (as Variant<Tuple<...>>)
template<class Sndr, class Env = env<>,
         template<class...> class Tuple = decayed-tuple,
         template<class...> class Variant = variant-or-empty>
using value_types_of_t = /* ... */;

// Extract all error types
template<class Sndr, class Env = env<>,
         template<class...> class Variant = variant-or-empty>
using error_types_of_t = /* ... */;

// Whether set_stopped may be sent
template<class Sndr, class Env = env<>>
constexpr bool sends_stopped = /* ... */;

Here are some concrete examples:

cpp
// just(42, 3.14) produces two values: int and double
using Just = decltype(stdexec::just(42, 3.14));
// value_types_of_t<Just> = variant<tuple<int, double>>
// error_types_of_t<Just> = variant<>            (never fails)
// sends_stopped<Just>    = false                 (never cancels)

// upon_error can "swallow" the error, turning a failure path into success
using Failable = decltype(
  stdexec::just_error(std::exception_ptr{})
  | stdexec::upon_error([](auto) { return 0; })
);
// value_types_of_t<Failable> = variant<tuple<int>>   (error recovered)
// error_types_of_t<Failable> = variant<>              (no longer fails)
// sends_stopped<Failable>    = false

In the code above, you can see how upon_error transforms the completion signatures — it converts set_error_t(exception_ptr) into set_value_t(int), “repairing” the error path into a success path. This is exactly the power of Completion Signatures as compile-time type identity cards: adaptors can rewrite their child Sender’s completion signatures.

Declaring Completion Signatures for Custom Senders

Custom Senders can declare completion signatures in two ways.

Way 1: Static Member Type

Suitable for simple Senders whose completion signatures don’t depend on the environment:

cpp
struct simple_sender {
  using sender_concept = stdexec::sender_tag;

  // Fixed completion signatures — determined at compile time
  using completion_signatures =
    stdexec::completion_signatures<
      stdexec::set_value_t(int),
      stdexec::set_error_t(std::exception_ptr)
    >;
};

In the code above, simple_sender’s completion signatures are fixed — success returns int, failure returns exception_ptr. This is the simplest declaration method, suitable for most Sender factories and adaptors like just, then, etc.

Way 2: get_completion_signatures Function Template

Suitable for complex scenarios where completion signatures depend on the environment or the Sender’s value category:

cpp
struct env_dependent_sender {
  using sender_concept = stdexec::sender_tag;

  // Completion signatures depend on Self's value category and the passed Env
  template <class Self, class... Env>
  static consteval auto get_completion_signatures()
    -> stdexec::completion_signatures</* ... */>
  {
    return {};
  }
};

In the code above, get_completion_signatures is a consteval function — it executes at compile time, dynamically computing completion signatures based on template parameter Self (the Sender’s type and value category) and Env (the environment type).

The retry algorithm uses this approach because its completion signatures depend on the child Sender’s completion signatures:

cpp
// retry's get_completion_signatures (simplified)
template <class Self, class... Env>
static consteval auto get_completion_signatures()
  -> stdexec::__transform_completion_signatures_t<
    stdexec::completion_signatures_of_t<S&, Env...>,
    stdexec::completion_signatures<stdexec::set_error_t(std::exception_ptr)>,
    _value,
    _error>
{
  return {};
}

In the code above, retry first obtains the child Sender’s completion signatures via completion_signatures_of_t<S&, Env...>, then transforms them — converting all error paths to set_error_t(exception_ptr). This demonstrates the flexibility of Completion Signatures as type-level functions.

transform_completion_signatures

stdexec provides __transform_completion_signatures_t for transforming completion signatures:

cpp
template<class Sigs,                    // input completion signatures
         class Default = completion_signatures<>,
         template<class...> class ValueFn,   // value signature transform
         template<class...> class ErrorFn,   // error signature transform
         bool SendStopped = /* ... */>       // whether to pass through stopped
using __transform_completion_signatures_t = /* ... */;

This utility is extremely useful when writing Sender adaptors. For example, the then adaptor needs to transform the child Sender’s set_value_t(int) into set_value_t(string) (because the callback converts int to string):

cpp
// then adaptor's completion signature transform
// Input:    completion_signatures<set_value_t(int)>
// ValueFn:  replace original value types with callback's return type
// ErrorFn:  pass through unchanged
// Output:   completion_signatures<set_value_t(string)>

The comments above illustrate the basic logic of completion signature transformation — adaptors don’t alter error or cancellation paths, only the value path’s types. This compile-time type transformation is the core of P2300’s type-safe composition.

Completion Signatures’ Four Roles

RoleDescription
Compile-time validationconnect() checks receiver_of<R, Sigs> — is the Receiver compatible?
Type deductionsync_wait deduces its return type via value_types_of_t
Algorithm adaptationwhen_all computes its own completion signatures from child Senders’ signatures
Domain transformationtransform_sender selects an algorithm implementation based on completion signatures

Important Notes

  1. completion_signatures_of_t may require environment parameters — environment-dependent Senders must be provided with Env to query their completion signatures
  2. sends_stopped is a compile-time constant — use it in if constexpr branches, not runtime checks
  3. Completion signatures are the Sender’s “contract” — the declared completion signatures must match the Sender’s actual runtime behavior, or it’s undefined behavior
  4. consteval functions guarantee compile-time evaluation — get_completion_signatures will never generate runtime code
  5. Custom Senders must declare correct completion signatures — if the signatures are incomplete, sync_wait will deduce wrong return types, and connect may compile with the wrong Receiver

Completion Signatures are the foundation of P2300’s type system. In subsequent chapters, each time we learn a new Sender adaptor (then, when_all, let_value, etc.), we’ll see how it transforms the child Sender’s completion signatures.

12b · Completion Signature Metaprogramming — gather-signatures and Friends

Standard Section: [exec.cmplsig]
Draft Reference: exec.tex lines 6423–6569

Beyond the Basics

Chapter 12 covered what completion signatures are and how to declare them. This chapter covers the metaprogramming machinery the standard itself uses to manipulate them.

The completion-signature Concept

Every entry in a completion_signatures list must satisfy completion-signature:

cpp
template <class Fn>
concept completion-signature = see-below;

A type Fn satisfies it iff it is a function type with one of three forms:

FormMeaning
set_value_t(Vs...)Value completion, Vs are object or reference types
set_error_t(Err)Error completion, Err is an object or reference type
set_stopped_t()Stopped completion, no arguments

This is checked at compile time. You cannot put random function types into completion_signatures.

valid-completion-signatures

A concept that checks whether a type is a proper specialization of completion_signatures — the standard uses it to constrain template arguments:

cpp
template <class T>
concept valid-completion-signatures = /* ... */;

gather-signatures — The Workhorse

This is the exposition-only utility that every type computation in the standard depends on:

cpp
template <class Tag,
          valid-completion-signatures Completions,
          template<class...> class Tuple,
          template<class...> class Variant>
using gather-signatures = see-below;

It extracts all completion signatures matching Tag from Completions, wraps each set of arguments in Tuple, and then wraps all of those in Variant:

gather-signatures<set_value_t,
                  completion_signatures<set_value_t(int),
                                        set_value_t(float, double),
                                        set_error_t(exception_ptr)>,
                  tuple, variant>

// Step 1: filter by Tag -> [set_value_t(int), set_value_t(float, double)]
// Step 2: wrap args in tuple -> [tuple<int>, tuple<float, double>]
// Step 3: wrap in variant -> variant<tuple<int>, tuple<float, double>>

The META-APPLY Puzzle

The standard uses a wrapper called META-APPLY to make gather-signatures work with non-variadic templates:

cpp
template <bool>
struct indirect-meta-apply {
  template <template<class...> class T, class... As>
  using meta-apply = T<As...>;
};

template <class...>
concept always-true = true;

// META-APPLY(T, As...) =
//   typename indirect-meta-apply<always-true<As...>>::template meta-apply<T, As...>

If T is a variadic template like tuple, META-APPLY(tuple, int, float) gives tuple<int, float> — straightforward. But if T is non-variadic like pair, META-APPLY(pair, int, float) also works, while a direct pair<int, float> would need all template parameters accounted for. The indirect-meta-apply indirection lets this compile.

value_types_of_t

The standard public alias that uses gather-signatures internally:

cpp
template <class Sndr, class Env = env<>,
          template<class...> class Tuple = decayed-tuple,
          template<class...> class Variant = variant-or-empty>
  requires sender_in<Sndr, Env>
using value_types_of_t =
  gather-signatures<set_value_t,
                    completion_signatures_of_t<Sndr, Env>,
                    Tuple, Variant>;

The default decayed-tuple is like tuple but applies decay_t to each element. The default variant-or-empty produces variant<tuple<Ts...>> if there is exactly one signature, or an empty variant<> if there are none.

This is what sync_wait uses to deduce its return type:

cpp
// sync_wait internal:
// value_types_of_t<Sndr> == variant<tuple<int>>
// return type: optional<tuple<int>>

error_types_of_t

Same pattern, but for error signatures:

cpp
template <class Sndr, class Env = env<>,
          template<class...> class Variant = variant-or-empty>
  requires sender_in<Sndr, Env>
using error_types_of_t =
  gather-signatures<set_error_t,
                    completion_signatures_of_t<Sndr, Env>,
                    type_identity_t, Variant>;

Note the Tuple parameter is type_identity_t — error completions always have exactly one argument, so no tuple wrapper is needed.

sends_stopped

A compile-time boolean:

cpp
template <class Sndr, class Env = env<>>
  requires sender_in<Sndr, Env>
constexpr bool sends_stopped =
  !same_as<type-list<>,
           gather-signatures<set_stopped_t,
                             completion_signatures_of_t<Sndr, Env>,
                             type-list, type-list>>;

If gather-signatures returns an empty type-list<>, there are no stopped signatures — sends_stopped is false.

completion_signatures Member Utilities

The completion_signatures class template itself provides two exposition-only helpers:

cpp
template <completion-signature... Fns>
struct completion_signatures {
  template <class Tag>
  static constexpr size_t count-of(Tag) {
    return /* count of Fns whose return type is Tag */;
  }

  template <class Fn>
  static constexpr void for-each(Fn&& fn) {
    (fn(static_cast<Fns*>(nullptr)), ...);
  }
};

count-of counts how many signatures match a tag:

cpp
using Sigs = completion_signatures<
  set_value_t(int),
  set_value_t(float),
  set_error_t(exception_ptr)>;

// Sigs::count-of(set_value) == 2
// Sigs::count-of(set_error) == 1
// Sigs::count-of(set_stopped) == 0

for-each invokes a callable for each signature, passing a null pointer of the corresponding function type. It uses fold expressions over the parameter pack.

Putting gather-signatures to Work

Here is a concrete trace through sync_wait’s return type deduction:

Sender: just(42) | then([](int x) { return std::to_string(x); })

Step 1: completion_signatures_of_t<Sender>
  -> completion_signatures<set_value_t(std::string)>

Step 2: value_types_of_t<Sender>
  -> gather-signatures<set_value_t,
                        completion_signatures<set_value_t(string)>,
                        decayed-tuple, variant-or-empty>
  -> filter by set_value_t -> [set_value_t(string)]
  -> wrap args: decayed-tuple<string> -> tuple<string>
  -> wrap in variant-or-empty: variant<tuple<string>>

Step 3: error_types_of_t<Sender>
  -> gather-signatures<set_error_t,
                        completion_signatures<set_value_t(string)>,
                        type_identity_t, variant-or-empty>
  -> no signatures match set_error_t
  -> empty: variant<>

Step 4: sends_stopped<Sender>
  -> gather-signatures<set_stopped_t, ..., type-list, type-list>
  -> no signatures match set_stopped_t
  -> type-list<>  (empty)
  -> !same_as<type-list<>, type-list<>>  ->  false

Result: sync_wait returns optional<tuple<string>>

transform_completion_signatures

While gather-signatures is for querying, transform_completion_signatures is for rewriting. The standard doesn’t mandate a specific implementation, but both stdexec and the exposition follow this pattern:

cpp
template <class Sigs,                     // input
          class DefaultSigs,              // fallback
          template<class...> class ValueFn,   // per-value transform
          template<class...> class ErrorFn,   // per-error transform
          bool PassStopped = true>
using transform_completion_signatures = /* ... */;

For each set_value_t(Ts...) in Sigs, it produces ValueFn<Ts...>. For each set_error_t(E) in Sigs, it produces ErrorFn<E>. If no signatures match a category, it inserts DefaultSigs. If PassStopped is true, it adds set_stopped_t() to the output.

The then adaptor uses it like this:

cpp
// Input:  completion_signatures<set_value_t(int), set_error_t(exception_ptr)>
// ValueFn: __value -> set_value_t(call_result_t<F, __value>)  [callback applied]
// ErrorFn: __error -> set_error_t(__error)                     [passthrough]
// Output: completion_signatures<set_value_t(string), set_error_t(exception_ptr)>

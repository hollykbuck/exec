13 · Sender Factory: just / just_error / just_stopped

Why Do We Need just?

In P2300’s async model, everything that flows through a pipeline must be a sender. But not all data in your program begins its life asynchronously. You might need to feed a config value, a cached integer, or a hardcoded constant into an async pipeline.

You shouldn’t have to spin up a thread pool just to pass a 42 downstream. That’s where just comes in.

just simply wraps a plain value into a sender. Once started, this sender immediately and synchronously calls set_value to hand the value to the downstream receiver.

If you know JavaScript: just is to sender what Promise.resolve() is to Promise — a way to lift a synchronous value into the async world.

Basic Usage

The just family has three factory functions, one for each completion signal:

cpp
namespace std::execution {
  inline constexpr unspecified just = /* ... */;
  inline constexpr unspecified just_error = /* ... */;
  inline constexpr unspecified just_stopped = /* ... */;
}

just(args...) —— Immediate Success

cpp
sender auto s1 = just();
// Completion signature: set_value_t()

sender auto s2 = just(42);
// Completion signature: set_value_t(int)

sender auto s3 = just(1, 2.0, "hello");
// Completion signature: set_value_t(int, double, const char*)

The code above creates three senders. just() sends an empty “I’m done” signal; just(42) passes the integer 42 downstream; just(1, 2.0, "hello") delivers three values of different types at once.

How It Works Internally

The sender returned by just(v) calls set_value directly inside start(), synchronously:

cpp
void start() & noexcept {
  stdexec::set_value(std::move(rcvr_), args_...);
}

The implementation is about as simple as it gets — no thread scheduling, no queues, just a direct, synchronous handoff to the downstream receiver.

Typical Use Cases

cpp
// As a pipeline starting point
sync_wait(
  just(42)
    | then([](int x) { return x * 2; })
);
// Result: 84

// As a fixed return value inside let_value
sender auto handler = let_value(
  some_sender,
  []() { return just(0); }  // return 0 on success
);

just_error(err) —— Immediate Failure

cpp
// Immediately sends set_error()
sender auto e = just_error(std::make_exception_ptr(std::runtime_error{"oops"}));
// Completion signature: set_error_t(std::exception_ptr)

// Also supports error_code
std::error_code ec = std::make_error_code(std::errc::io_error);
sender auto e2 = just_error(ec);

The code above creates a sender that, when started, notifies the downstream receiver with set_error. This shines in conditional branches:

cpp
sender auto validate(int x) -> sender auto {
  if (x < 0)
    return just_error(std::make_exception_ptr(std::invalid_argument{"negative"}));
  return just(x);
}

This validate function returns a “failed immediately” sender when the input is invalid, or a value-carrying sender when it’s valid. Both branches return senders, keeping the types consistent.

just_stopped() —— Immediate Cancellation

cpp
sender auto s = just_stopped();
// Completion signature: set_stopped_t()

This sender tells the downstream “I’ve been cancelled.” Its most common use is also in conditional branches, expressing “this path is a dead end”:

cpp
sender auto maybe_cancel(bool should_stop) -> sender auto {
  if (should_stop)
    return just_stopped();
  return just(42);
}

Composing in Pipelines

All just factories plug directly into pipelines:

cpp
auto pipeline = just(1)
              | then([](int x) { return x + 1; })
              | upon_error([](auto e) { return -1; })
              | upon_stopped([] { return 0; });

// Or chosen dynamically via let_value
auto result = sync_wait(
  schedule(sch)
    | let_value([] {
        if (rand() % 2 == 0)
          return just(42);
        else
          return just_error(std::make_exception_ptr(std::runtime_error{"bad luck"}));
      })
);

The code above shows let_value dynamically picking between just(42) and just_error(...) at runtime. This kind of branching is the core use case of let_value, covered in depth in chapter 17.

Limitations

just-created senders complete immediately and synchronously — they run on the same thread that calls start(), with no background dispatch. If you need true async execution, combine just with schedule and starts_on (see chapters 15 and 18).

Also, the values passed to just are copied or moved into the sender’s internal storage. You cannot pass a bare reference (though std::ref works), and non-copyable objects must be movable (like std::unique_ptr).

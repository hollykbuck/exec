14 · read_env — Reading Properties from the Environment

Standard section: [exec.read.env] stdexec reference: include/stdexec/__detail/ Related tests: tests/test_run_loop.cpp

Overview

read_env is a sender factory that reads the value of a query object from the receiver’s environment.

cpp
namespace std::execution {
  inline constexpr unspecified read_env{};
}

Basic Usage

cpp
// Read the scheduler from the current receiver environment
sender auto s = read_env(get_scheduler);

// Read the stop token
sender auto s = read_env(get_stop_token);

// Read the allocator
sender auto s = read_env(get_allocator);

get_scheduler() Query

This is the most common usage — querying and obtaining the current receiver environment’s scheduler within a pipeline:

cpp
sync_wait(
  schedule(sch)
    | then([] {
        // Here, the receiver environment contains get_scheduler
        // This query retrieves the scheduler from the current execution context
      })
);

// read_env(get_scheduler) is equivalent to:
sync_wait(get_scheduler());  // in sync_wait's environment, returns run_loop::scheduler

Example from hello_world.cpp:

cpp
// Get the scheduler from the current receiver environment
auto [sched] = sync_wait(get_scheduler()).value();
// sched is of type run_loop::scheduler

// Then use this scheduler in another pipeline
sync_wait(
  let_value(get_scheduler(), [](auto sched) {
    return starts_on(sched, then(just(), [] {
      std::cout << "from run_loop\n";
      return 42;
    }));
  })
);

Complete Signatures of read_env

cpp
// read_env's completion signatures under environment type Env
// depend on the result type of query q in Env
// let T = decltype(q(declval<Env>()))
// then: completion_signatures<set_value_t(T)>

For example, under the environment provided by sync_wait:

cpp
// sync_wait's receiver environment contains:
// - get_scheduler → run_loop::scheduler
// - get_stop_token → never_stop_token

using Sigs = completion_signatures_of_t<decltype(read_env(get_scheduler)), decltype(sync_wait_env)>;
// Sigs = completion_signatures<set_value_t(run_loop::scheduler)>

read_env and the Domain System

read_env(get_domain) plays an important role in the domain system:

cpp
// Query the current environment's domain
auto dom = sync_wait(read_env(get_domain)).value();

Custom Queries

You can create your own query objects to use with read_env:

cpp
// Define a query tag
struct my_config_t {
  template<class Env>
  auto operator()(const Env& env) const -> int {
    return env.query(*this);  // delegate to the environment's query method
  }
};
inline constexpr my_config_t my_config{};

// Create an environment containing this query
auto env_with_config = stdexec::prop{my_config, 42};

// Use read_env to read it
// (requires that the sender's receiver environment includes my_config)

Composition with write_env

write_env(prop, sndr) can inject environment properties into a pipeline, which read_env can then read:

cpp
sync_wait(
  just(42)
    | write_env(prop{my_config, 99})
    | then([](int x) {
        // Here the receiver environment contains my_config = 99
        // can use read_env(my_config)
        return x;
      })
);

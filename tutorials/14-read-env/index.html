14 · read_env — Reading Properties from the Environment

What Is the “Environment”?

In the sender/receiver model, every receiver carries an environment alongside it. This environment acts like a backpack — it holds all the queryable context for the current execution:

  • What scheduler am I running on? (get_scheduler)
  • What’s my current stop token? (get_stop_token)
  • Which allocator should I use? (get_allocator)

With this mechanism, you don’t need to pass schedulers and allocators down through layers of function calls — the receiver carries them for you.

read_env is how you pull information out of this environment.

Why Do We Need read_env?

Suppose inside a then callback you need to know “which thread pool am I on right now?”, or you want to check the stop token to decide whether to bail out early. That information is already in the receiver’s environment — you just need read_env to read it.

Basic Usage

cpp
namespace std::execution {
  inline constexpr unspecified read_env{};
}

And the most common queries:

cpp
sender auto s = read_env(get_scheduler);    // Read the scheduler from the environment
sender auto s = read_env(get_stop_token);   // Read the stop token
sender auto s = read_env(get_allocator);    // Read the allocator

The Most Common Query: get_scheduler()

Retrieving the current scheduler from within a pipeline is the most common use of read_env:

cpp
sync_wait(
  schedule(sch)
    | then([] {
        // The receiver environment now contains get_scheduler
      })
);

After schedule(sch), the receiver’s environment holds the scheduler. If you need to grab it mid-pipeline:

cpp
auto [sched] = sync_wait(get_scheduler()).value();
// sched is run_loop::scheduler

sync_wait(
  let_value(get_scheduler(), [](auto sched) {
    return starts_on(sched, then(just(), [] {
      std::cout << "from run_loop\n";
      return 42;
    }));
  })
);

The code above reads the scheduler from the environment, then passes it to subsequent operations via let_value. The sender returned by read_env emits set_value(the_query_result), so it plugs into a pipeline just like any other sender.

Completion Signatures of read_env

cpp
// read_env's completion signature depends on the query's return type in environment Env
// Let T = decltype(q(declval<Env>()))
// Then: completion_signatures<set_value_t(T)>

Under sync_wait’s environment:

cpp
// sync_wait's receiver environment provides:
// - get_scheduler → run_loop::scheduler
// - get_stop_token → never_stop_token

using Sigs = completion_signatures_of_t<decltype(read_env(get_scheduler)), decltype(sync_wait_env)>;
// Sigs = completion_signatures<set_value_t(run_loop::scheduler)>

read_env and the Domain System

read_env(get_domain) plays a key role in the domain system:

cpp
auto dom = sync_wait(read_env(get_domain)).value();

Custom Queries

You can define your own query objects beyond the standard ones:

cpp
struct my_config_t {
  template<class Env>
  auto operator()(const Env& env) const -> int {
    return env.query(*this);
  }
};
inline constexpr my_config_t my_config{};

auto env_with_config = stdexec::prop{my_config, 42};
// When a receiver's environment contains my_config,
// read_env(my_config) will return 42

Relationship with write_env

Chapter 24 covers write_env, which injects environment properties into a pipeline. read_env reads them back. Together they form an environment variable system for sender pipelines:

cpp
sync_wait(
  just(42)
    | write_env(prop{my_config, 99})  // Write to the environment
    | then([](int x) {
        // read_env(my_config) would return 99 here
        return x;
      })
);

15 · schedule — Creating a Sender from a Scheduler

Why Do We Need schedule?

Every asynchronous operation must answer a fundamental question: “where do I run?”

In a single-threaded world this question doesn’t exist. But modern programs run on thread pools, GPUs, and network IO threads. You need an explicit way to say “this sender pipeline starts on that scheduler’s resources.”

schedule is the answer. If schedulers are hotels, schedule(sch) is checking in at the front desk — once you’ve checked in, subsequent operations (then, etc.) know which room (thread) they’re supposed to be in.

Basic Behavior

cpp
namespace std::execution {
  template<scheduler Sch>
    using schedule_result_t = decltype(schedule(declval<Sch>()));

  inline constexpr schedule_t schedule{};
}

The standard specifies (exec.tex:310-325) that the sender returned by schedule(sch) does three things:

  1. Acquires an execution agent on the resource associated with sch
  2. Calls set_value(rcvr) on that agent (with no arguments)
  3. Sets the value-completion scheduler to sch itself
cpp
// schedule's completion signature carries no data
// completion_signatures_of_t<schedule_result_t<Sch>>
// = completion_signatures<set_value_t()>

The key thing to remember: schedule only expresses “begin execution here” — it delivers no data. To pass data, you need combinators like then, let_value, etc.:

cpp
// Wrong: schedule produces no data
auto bad = schedule(sch) | then([](int x) { /* Compile error! schedule has no arguments */ });

// Correct: use just for data, then starts_on to switch context
auto correct = starts_on(sch, just(42)) | then([](int x) { /* OK */ });

A Mandatory Property of the schedule Sender

The sender returned by schedule has an important guarantee: its value-completion scheduler is the very scheduler that created it:

cpp
sender auto s = schedule(sch);

// get_completion_scheduler<set_value_t>(get_env(s)) == sch
// This lets then and other adaptors know "where to run"

This is why schedule(sch) | then(f) works — the then adaptor queries the environment for get_scheduler, gets back sch, and schedules f on the right thread.

Using schedule in a Pipeline

cpp
// Simple: run a function on a thread pool
schedule(pool.get_scheduler())
  | then([] { std::cout << "in pool\n"; })
  | sync_wait;

// Combined with when_all: run on pool and inline simultaneously
when_all(
  schedule(pool.get_scheduler()) | then(heavy_work),
  schedule(inline_scheduler{}) | then(light_work)
) | sync_wait;

In the code above, heavy_work runs on the thread pool while light_work runs inline (synchronously, on the current thread). They execute concurrently, and when_all waits for both to finish.

schedule vs starts_on

ExpressionEffect
schedule(sch) | then(f)Execute f on sch
starts_on(sch, just(42) | then(f))The entire 42 | f starts on sch
starts_on(sch, sndr)sndr starts entirely on sch

starts_on is more flexible — it accepts any sender, while schedule only creates a no-argument start signal.

Limitations

schedule itself carries no data, only an empty set_value() signal. If you need “execute and return data,” you must chain then or another adaptor after it. Also, schedule doesn’t guarantee immediate execution — it just enqueues work on the scheduler’s resource; actual timing is up to the scheduler.

15 · schedule — Creating a Sender from a Scheduler

Standard section: [exec.schedule] stdexec reference: examples/hello_world.cpp Related tests: tests/test_basics.cpp · tests/test_thread_pool.cpp

Overview

schedule is the most basic sender factory. It creates a sender from a scheduler, representing “begin execution on the resources associated with that scheduler”.

cpp
namespace std::execution {
  template<scheduler Sch>
    using schedule_result_t = decltype(schedule(declval<Sch>()));

  inline constexpr schedule_t schedule{};
}

Behavior

The standard specifies (exec.tex:310-325):

The asynchronous operation of the sender returned by schedule(sch):

  1. Obtains an execution agent on the execution resource associated with sch
  2. Executes set_value(rcvr) (no arguments)
  3. The value completion scheduler is sch itself
cpp
// Completion signatures
// completion_signatures_of_t<schedule_result_t<Sch>>
// = completion_signatures<set_value_t()>
// Note: no arguments! schedule just "starts execution", it doesn't produce data.

Typical Usage

cpp
exec::static_thread_pool pool{4};
auto sch = pool.get_scheduler();

// "Check in" on the thread pool then do work
schedule(sch) | then([] {
  return 42;  // Now on this thread pool
});

schedule only conveys the semantics of “execute here”. To carry data, you need combinators like then, let_value, etc.:

cpp
// Wrong: schedule doesn't produce data
auto bad = schedule(sch) | then([](int x) { /* Compile error! schedule has no arguments */ });

// Correct: use just to provide data first, then starts_on to switch context
auto correct = starts_on(sch, just(42)) | then([](int x) { /* OK */ });

Special Properties of the schedule Sender

The schedule sender has a mandatory property: its value completion scheduler is the scheduler that produced it.

cpp
sender auto s = schedule(sch);

// get_completion_scheduler<set_value_t>(get_env(s)) == sch
// This ensures that after schedule, adaptors like then know "where to execute"

The significance of this property: when you use schedule(sch) | then(f) in a pipeline, the then adaptor knows that f should execute on sch.

Using in a Pipeline

cpp
// Simple: execute a function on the pool
schedule(pool.get_scheduler())
  | then([] { std::cout << "in pool\n"; })
  | sync_wait;

// Combined with when_all: execute on pool and inline simultaneously
when_all(
  schedule(pool.get_scheduler()) | then(heavy_work),
  schedule(inline_scheduler{}) | then(light_work)
) | sync_wait;

schedule vs starts_on

ExpressionEffect
schedule(sch) | then(f)Execute f on sch
starts_on(sch, just(42) | then(f))Execute 42 | f on sch
starts_on(sch, sndr)sndr starts entirely on sch

starts_on is more flexible — it accepts any sender, while schedule can only create a no-argument start signal.

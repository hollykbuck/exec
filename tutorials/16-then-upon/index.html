16 · then / upon_error / upon_stopped — Handling the Three Completion Signals

Standard sections: [exec.then] [exec.upon.error] [exec.upon.stopped] stdexec reference: examples/server_theme/then_upon.cpp Related tests: tests/test_then_upon.cpp

Overview

These three adaptors handle the three completion signals respectively:

AdaptorHandlesSignature Requirement
then(sndr, f)set_valuef(Args...) → T
upon_error(sndr, f)set_errorf(Error) → T
upon_stopped(sndr, f)set_stoppedf() → T

then — Transforming Success Values

cpp
// Corresponding test: test_then_upon.cpp → "then transforms set_value"
// Standard definition
sender auto then(sender auto sndr, std::invocable auto f);

// Usage
auto result = sync_wait(
  just(41)
    | then([](int x) { return x + 1; })
);
// result = optional{tuple{42}}

Semantics of then:

  1. Wait for the previous sender to deliver set_value
  2. Call f with the result values
  3. Deliver set_value(f(args...)) to the downstream
cpp
// If the previous one is set_error / set_stopped, pass through
just_error(ptr)
  | then([](int x) { /* won't execute */ });

Multiple Arguments

cpp
// Corresponding test: test_then_upon.cpp → "then on when_all result"
when_all(just(1), just(2.0))
  | then([](int a, double b) { return a + b; });

upon_error — Handling Errors

cpp
// Corresponding test: test_then_upon.cpp → "upon_error catches set_error and converts to value"
// Error handling: convert set_error to set_value
auto result = sync_wait(
  just_error(std::make_exception_ptr(std::runtime_error{"oops"}))
    | upon_error([](std::exception_ptr e) {
        try { std::rethrow_exception(e); }
        catch (const std::exception& ex) {
          return -1;
        }
      })
);
// result = optional{tuple{-1}}

upon_error consumes set_error and converts it to set_value. This means the downstream will no longer receive set_error.

cpp
using Sigs = completion_signatures_of_t<
  decltype(just_error(ptr) | upon_error([](auto) { return 0; }))
>;
// Sigs = completion_signatures<set_value_t(int)>
// Note: set_error_t is gone!

upon_stopped — Handling Cancellation

cpp
// Corresponding test: test_then_upon.cpp → "upon_stopped converts cancellation to value"
auto result = sync_wait(
  just_stopped()
    | upon_stopped([] { return 0; })
);
// result = optional{tuple{0}}

Priority of Handling the Three Signals

When a sender may emit multiple signals:

cpp
// Corresponding test: test_then_upon.cpp → "chained then, upon_error, upon_stopped normalizes signals"
auto pipeline = some_sender
              | then([](int x) { return x + 1; })             // handle set_value
              | upon_error([](auto e) { return -1; })          // handle set_error
              | upon_stopped([] { return 0; });                // handle set_stopped

// Final completion signature is only: completion_signatures<set_value_t(int)>
// All three signals are normalized to set_value

Chained vs Branched Handling

then/upon_error/upon_stopped chained handling means each adaptor only handles one signal, passing others through.

graph LR
  J["just(42)"] --> V1["set_value(42)"]
  J --> T["then(f)"]
  T --> V2["set_value(f(42))"]
  T --> UE["upon_error(g)"]
  UE --> V3["pass through set_value(f(42))"]
  UE --> US["upon_stopped(h)"]
  US --> V4["pass through set_value(f(42))"]

Other adaptors (like let_value/let_error/let_stopped) provide more flexible signal-to-sender transformations.

Server Theme Example

From examples/server_theme/then_upon.cpp:

cpp
// Corresponding test: test_then_upon.cpp → "multiple then operations compose linearly"
// HTTP request handling: normal processing, error handling, and cancellation handling
schedule(io_sched)
  | then([] { return read_http_request(); })
  | then([](http_request req) { return classify_image(req); })
  | then([](classification result) { return make_http_response(result); })
  | upon_error([](std::exception_ptr e) {
      return make_error_response(500, "Internal Server Error");
    })
  | upon_stopped([] {
      return make_error_response(499, "Request Cancelled");
    });

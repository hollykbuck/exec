16 · then / upon_error / upon_stopped — Handling the Three Completion Signals

Why Three Signals?

Every asynchronous operation ends in exactly one of three ways:

(1) Success — the data arrived. Delivered via set_value. (2) Failure — something went wrong. Delivered via set_error. (3) Cancellation — it was stopped midway. Delivered via set_stopped.

Traditional C++ async approaches — exceptions, error codes, callbacks — tend to mix these three outcomes together. Exceptions can be forgotten, error codes ignored, and cancellation simply doesn’t exist in the standard at all.

C++26 std::execution cleanly separates all three, forcing you to have a plan for each outcome. then, upon_error, and upon_stopped are the adaptors that handle each one.

Overview

AdaptorHandlesCallback Signature
then(sndr, f)set_valuef(Args...) → T
upon_error(sndr, f)set_errorf(Error) → T
upon_stopped(sndr, f)set_stoppedf() → T

then —— Handling Success

cpp
sender auto then(sender auto sndr, std::invocable auto f);

auto result = sync_wait(
  just(41)
    | then([](int x) { return x + 1; })
);
// result = optional{tuple{42}}

In the code above, just(41) produces a sender that, on success, emits set_value(41). then receives the 41, computes + 1, and sends set_value(42) downstream.

then works in three steps:

  1. Wait for the upstream sender to deliver set_value
  2. Call f with the value arguments
  3. Deliver set_value(f(args...)) to the downstream
cpp
// If the upstream sends set_error or set_stopped, then's callback never runs
just_error(ptr)
  | then([](int x) { /* This code never executes */ });

Multiple Arguments

cpp
when_all(just(1), just(2.0))
  | then([](int a, double b) { return a + b; });

upon_error —— Handling Failure

cpp
auto result = sync_wait(
  just_error(std::make_exception_ptr(std::runtime_error{"oops"}))
    | upon_error([](std::exception_ptr e) {
        try { std::rethrow_exception(e); }
        catch (const std::exception& ex) {
          return -1;
        }
      })
);
// result = optional{tuple{-1}}

In the code above, just_error sends set_error. upon_error catches it, recovers from the exception, and returns a default value of -1. The downstream sees set_value(-1) — the error has been “swallowed.”

upon_error converts set_error into set_value, meaning the downstream no longer sees an error at all:

cpp
using Sigs = completion_signatures_of_t<
  decltype(just_error(ptr) | upon_error([](auto) { return 0; }))
>;
// Sigs = completion_signatures<set_value_t(int)>
// Note: set_error_t is gone!

upon_stopped —— Handling Cancellation

cpp
auto result = sync_wait(
  just_stopped()
    | upon_stopped([] { return 0; })
);
// result = optional{tuple{0}}

This turns set_stopped (cancellation) into set_value(0) (success). Useful when “no result” is just fine — e.g., a cache miss is not an error.

Signal Handling Priority

When a sender can emit multiple signals, chaining all three adaptors normalizes everything into set_value:

cpp
auto pipeline = some_sender
              | then([](int x) { return x + 1; })             // handle set_value
              | upon_error([](auto e) { return -1; })          // handle set_error
              | upon_stopped([] { return 0; });                // handle set_stopped

// Final completion signature: completion_signatures<set_value_t(int)>
// All three signals are normalized to set_value

The code above shows the “normalization” effect of chaining. Each adaptor handles one signal and passes the others through. No matter what the upstream sender emits, the pipeline’s output is always set_value(int).

Other adaptors (let_value/let_error/let_stopped, chapter 17) provide more flexible signal-to-sender transformations.

Server Theme Complete Example

From examples/server_theme/then_upon.cpp — HTTP request handling where success, error, and cancellation each have their own handler:

cpp
schedule(io_sched)
  | then([] { return read_http_request(); })
  | then([](http_request req) { return classify_image(req); })
  | then([](classification result) { return make_http_response(result); })
  | upon_error([](std::exception_ptr e) {
      return make_error_response(500, "Internal Server Error");
    })
  | upon_stopped([] {
      return make_error_response(499, "Request Cancelled");
    });

The first three then calls handle the happy path — read request, classify image, build response. If anything throws, upon_error catches it and returns a 500. If the request is cancelled, upon_stopped returns a 499.

Limitations

then, upon_error, and upon_stopped each handle only one signal type; the others pass through untouched. The most common beginner mistake is writing only then and forgetting upon_error and upon_stopped — error and cancellation signals then leak downstream, where they may be silently lost if nothing handles them.

Also, the callbacks return plain values, not senders. If you need to launch a new async operation inside the callback (e.g., calling another service on error), you need let_value/let_error/let_stopped from chapter 17.

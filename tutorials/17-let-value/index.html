17 · let_value / let_error / let_stopped — Dynamically Generating New Senders Based on Results

Standard sections: [exec.let.value] [exec.let.error] [exec.let.stopped] stdexec reference: examples/server_theme/let_value.cpp Related tests: tests/test_let_value.cpp

Overview

The let_value family of adaptors maps a completion signal to a new sender. This is one of the most powerful adaptors — it lets you dynamically decide what to do next based on the result of a previous operation.

AdaptorTrigger ConditionCallback Signature
let_value(sndr, f)set_value(args...)f(Args...) → sender<U>
let_error(sndr, f)set_error(err)f(Error) → sender<U>
let_stopped(sndr, f)set_stopped()f() → sender<U>

let_value — Deciding Next Steps Based on Success Values

cpp
// Corresponding test: test_let_value.cpp → "let_value branches on set_value"
// Basic usage: dynamically choose processing based on request
sender auto handle_request(http_request req) {
  return just(req)
    | let_value([](http_request& req) -> sender auto {
        if (req.method == "GET")
          return handle_get(req);       // returns sender
        else if (req.method == "POST")
          return handle_post(req);      // returns sender
        else
          return just_error(std::make_exception_ptr(
            std::runtime_error{"unsupported method"}));
      });
}

let_value vs then

Key difference:

cpp
// Corresponding test: test_let_value.cpp → "let_value with positive input succeeds"
// then: callback returns a plain value (wrapped in set_value)
then(sndr, [](int x) -> int { return x + 1; })
// Completion signature: set_value_t(int)

// let_value: callback returns a sender (can be asynchronous)
let_value(sndr, [](int x) -> sender auto {
  return just(x) | then([](int y) { return y + 1; });
})
// Completion signature: depends on the returned sender

Rules:

  • then(f): f returns T, equivalent to implicitly wrapping the result as just(f(args...)) after set_value
  • let_value(f): f returns sender<T>, delivers set_value(T) after this sender completes

let_error — Error Recovery

cpp
// Corresponding test: test_let_value.cpp → "let_error recovers from error"
sender auto safe_divide(int a, int b) {
  return just(a, b)
    | let_value([](int a, int b) -> sender auto {
        if (b == 0)
          return just_error(std::make_exception_ptr(
            std::domain_error{"divide by zero"}));
        return just(a / b);
      })
    | let_error([](std::exception_ptr e) -> sender auto {
        // Error recovery: use default value
        return just(0);
      });
}

let_stopped — Cleanup on Cancellation

cpp
// Corresponding test: test_let_value.cpp → "let_stopped handles cancellation"
sender auto with_cleanup(sender auto work) {
  return work
    | let_stopped([] {
        // Perform cleanup operations (returns a sender)
        return just(0);  // or start a new cleanup async operation
      });
}

Server Theme Complete Example

From examples/server_theme/let_value.cpp — Request validation → Processing → Response:

cpp
// Corresponding test: test_let_value.cpp
schedule(io_sched)
  | then([] { return read_request(); })                       // Read request
  | let_value([](request req) -> sender auto {                // Decide based on request
      if (!validate(req))
        return just_error(std::make_exception_ptr(
          std::runtime_error{"bad request"}));
      return process(req);                                   // Return processing sender
    })
  | then([](response resp) { return send_response(resp); })  // Send response
  | let_error([](auto e) -> sender auto {                     // Error recovery
      return send_error_response(e);
    })
  | let_stopped([] -> sender auto {                           // Cancellation handling
      return send_error_response(499, "cancelled");
    });

Signal Transformation in let_value

Signal transformation rules (internally called __transform_completion_signatures_t in stdexec):

graph LR
  A["Input: just(x)"] --> B["let_value(f)"]
  B --> C["where f(x) → just(y)"]
  C --> D["Output: completion_signatures&lt;set_value_t(decltype(y))&gt;"]

Pass-through of error and cancellation signals:

cpp
// let_value passes through set_error and set_stopped (unless caught by map_xxx)
just_error(ptr)
  | let_value([](int x) { return just(x + 1); });
// set_error(ptr) passes through, the set_value branch won't execute

just_stopped()
  | let_value([](int x) { return just(x + 1); });
// set_stopped() passes through

Nested Async Operations

The real power of let_value lies in supporting nested asynchronous operations:

cpp
// Corresponding test: test_let_value.cpp
schedule(io_sched)
  | then([] { return open_file("data.txt"); })
  | let_value([](file_handle f) -> sender auto {
      // Returns a new async operation
      return read_file(f, 4096)
           | continues_on(cpu_sched)
           | then([](buffer buf) { return parse(buf); });
    })
  | then([](parsed_data data) {
      // Continue on cpu_sched
      return process(data);
    });

Internally, the implementation of let_value creates new connect + start calls, expanding the nested sender as part of the outer pipeline.

17 · let_value / let_error / let_stopped — Dynamically Generating New Senders Based on Results

The Limits of then

In the previous chapter we learned about then — it transforms a value on success and passes it along. But then has a fundamental limitation:

then’s callback returns a plain value, not a sender.

This means the most you can do inside the callback is “swap the value.” You can’t say “based on this value, I want to take a different async pipeline.” For example:

  • Is the request a GET? Take the handle_get async pipeline.
  • Is it a POST? Take the handle_post async pipeline.
  • Is it invalid? Return an error immediately.

These needs can’t be met with then, since then’s return type is always the same.

let_value solves this. Its callback returns not a plain value, but a sender — so you can dynamically choose what to do next based on the runtime input.

then is like a fixed conveyor belt — no matter the input, the output type is determined at compile time. let_value is like a smart sorting system — it can decide which conveyor belt comes next, based on the result.

Callback Signatures

AdaptorTrigger ConditionCallback Signature
let_value(sndr, f)set_value(args...)f(Args...) → sender<U>
let_error(sndr, f)set_error(err)f(Error) → sender<U>
let_stopped(sndr, f)set_stopped()f() → sender<U>

Note: the key difference from then/upon_error/upon_stopped is that the callback returns a sender, not a plain value.

let_value —— Branching on Success

cpp
sender auto handle_request(http_request req) {
  return just(req)
    | let_value([](http_request& req) -> sender auto {
        if (req.method == "GET")
          return handle_get(req);       // returns a sender
        else if (req.method == "POST")
          return handle_post(req);      // returns another sender
        else
          return just_error(std::make_exception_ptr(
            std::runtime_error{"unsupported method"}));
      });
}

The code above dynamically picks a pipeline based on the HTTP method: GET goes to handle_get, POST to handle_post, anything else returns an error immediately. Each branch returns a sender — it could be one that reads a file, queries a database, or just a plain just.

let_value vs then

The core distinction:

cpp
// then: callback returns a plain value (auto-wrapped as set_value)
then(sndr, [](int x) -> int { return x + 1; })
// Completion signature: set_value_t(int)

// let_value: callback returns a sender (its completion determines downstream)
let_value(sndr, [](int x) -> sender auto {
  return just(x) | then([](int y) { return y + 1; });
})
// Completion signature: depends on the returned sender's completion signal

Rules summary:

  • then(f): f returns T, which is implicitly wrapped as just(f(args...)) and delivered via set_value
  • let_value(f): f returns sender<T>, and that sender’s set_value(T) is forwarded downstream

let_error —— Error Recovery

cpp
sender auto safe_divide(int a, int b) {
  return just(a, b)
    | let_value([](int a, int b) -> sender auto {
        if (b == 0)
          return just_error(std::make_exception_ptr(
            std::domain_error{"divide by zero"}));
        return just(a / b);
      })
    | let_error([](std::exception_ptr e) -> sender auto {
        return just(0);  // default to 0 on division by zero
      });
}

The code above first checks via let_value whether the divisor is zero — if so, returns just_error. Then let_error catches this error and returns a sender with a default value of 0.

Note the difference between let_error and upon_error: let_error’s callback returns a sender, so you can launch new async operations here (e.g., write a log, retry a request), not just return a replacement value.

let_stopped —— Cleanup on Cancellation

cpp
sender auto with_cleanup(sender auto work) {
  return work
    | let_stopped([] {
        return just(0);  // cleanup done, return sender
      });
}

Server Theme Complete Example

From examples/server_theme/let_value.cpp — request validation → processing → response:

cpp
schedule(io_sched)
  | then([] { return read_request(); })                       // Read request
  | let_value([](request req) -> sender auto {                // Decide route based on request
      if (!validate(req))
        return just_error(std::make_exception_ptr(
          std::runtime_error{"bad request"}));
      return process(req);                                   // Return processing sender
    })
  | then([](response resp) { return send_response(resp); })  // Send response
  | let_error([](auto e) -> sender auto {                     // Error recovery
      return send_error_response(e);
    })
  | let_stopped([] -> sender auto {                           // Cancellation handling
      return send_error_response(499, "cancelled");
    });

The code above shows a complete HTTP server request flow: read request → validate → (fork: process if valid, error if not) → send response. Each segment dynamically decides the next sender via let_value.

Signal Transformation Rules

let_value only intercepts set_value; set_error and set_stopped pass through:

cpp
just_error(ptr)
  | let_value([](int x) { return just(x + 1); });
// set_error(ptr) passes through directly, let_value's callback never runs

just_stopped()
  | let_value([](int x) { return just(x + 1); });
// set_stopped() passes through

Nested Async Operations

The true power of let_value lies in nested async operations — you can launch an entirely new async pipeline inside the callback, and it gets seamlessly embedded into the outer pipeline:

cpp
schedule(io_sched)
  | then([] { return open_file("data.txt"); })
  | let_value([](file_handle f) -> sender auto {
      return read_file(f, 4096)
           | continues_on(cpu_sched)
           | then([](buffer buf) { return parse(buf); });
    })
  | then([](parsed_data data) {
      return process(data);
    });

In the code above, after open_file completes, let_value’s callback starts a new sub-pipeline (read file → switch to CPU pool → parse data). The result of this sub-pipeline (parsed_data) automatically flows into the outer process(data).

Internally, let_value creates new connect + start calls, flattening the nested sender into the outer pipeline.

Limitations

Each branch returned by let_value must satisfy the same sender_to<Receiver> constraint. If different branches return different sender types (e.g., one returns sender<set_value_t(int)>, another sender<set_value_t(double)>), the compiler will error. In that case, use into_variant from chapter 21 to unify the types.

Also, let_value re-connects and re-starts on every invocation, which has some runtime overhead. For simple value transformations, then is lighter.

18 · starts_on / continues_on / on — Switching Execution Contexts

Why Switch Execution Contexts?

A modern network service follows a typical flow:

(1) IO thread reads the request data (2) CPU thread pool processes the request (3) IO thread writes back the response

Each stage runs on a different execution resource. starts_on and continues_on are the building blocks that make this “thread hopping” possible.

Overview

AdaptorEffect
starts_on(sch, sndr)Starts sndr on sch
continues_on(sndr, sch)Continues on sch after sndr completes
on(sch, sndr)Equivalent to starts_on(sch, continues_on(sch, sndr))

starts_on —— Starting on a Specific Scheduler

cpp
exec::static_thread_pool io_pool{2};
exec::static_thread_pool cpu_pool{4};

// The entire pipeline starts on io_pool
starts_on(io_pool.get_scheduler(),
  just(42)
    | then([](int x) { return x + 1; })
)
  | sync_wait;

In the code above, just(42) | then(...) — the entire sender — runs on io_pool’s threads from the moment start() is called.

starts_on is a sender adaptor with two calling forms:

cpp
// Function form
starts_on(sch, sndr)

// Pipeline form
starts_on(sch)(sndr)  // equivalent to starts_on(sch, sndr)

continues_on —— Switching to a New Resource After Completion

cpp
// After IO completes, switch to the CPU pool for computation
schedule(io_pool.get_scheduler())
  | then([] { return read_data(); })
  | continues_on(cpu_pool.get_scheduler())  // ← Switch point
  | then([](data d) { return compute(d); })
  | sync_wait;

The code above shows the classic IO→CPU pattern: read data on an IO thread, then continues_on switches subsequent operations to the CPU thread pool. read_data runs on an IO thread, compute runs on a CPU thread.

Switching Semantics

graph LR
  A["read_data()"] --> B["Runs on IO thread"]
  B --> C["continues_on(cpu)"]
  C --> D["Scheduled onto CPU thread pool"]
  D --> E["compute(d)"]
  E --> F["Runs on CPU thread"]

continues_on schedules downstream operations onto the target scheduler after the upstream sender completes.

on —— Start and Stay on the Same Resource

cpp
// Start on cpu_pool, and everything downstream stays on cpu_pool
on(cpu_pool.get_scheduler(),
  just(42)
    | then([](int x) { return x + 1; })
    | then([](int x) { return x * 2; })
);

on(sch, sndr) is equivalent to starts_on(sch, continues_on(sch, sndr)) — “use sch for everything.”

Complete Example: Post-IO Processing

From examples/server_theme/on_transfer.cpp — an HTTP image processing server:

cpp
schedule(io_sched)
  | then([] { return recv_http_request(); })
  | continues_on(cpu_sched)                      // Switch to CPU for request processing
  | let_value([](http_request req) -> sender auto {
      return starts_on(io_sched,                  // Switch back to IO for reading image
               then(just(req.path),
                  [](auto path) { return read_image(path); }));
    })
  | continues_on(cpu_sched)                      // Switch back to CPU for image processing
  | then([](image img) { return process_image(img); })
  | continues_on(io_sched)                       // Switch back to IO for sending response
  | then([](response resp) { return send_response(resp); });

The code above shows a full IO↔CPU round-trip: receive request (IO) → process request (CPU) → read image (IO) → process image (CPU) → send response (IO). continues_on and starts_on explicitly mark the execution location at each step.

Context Switching and the Receiver Environment

When the execution context switches, the receiver’s environment changes too:

cpp
schedule(pool1.get_scheduler())
  | then([] {
      // get_scheduler(env) returns pool1's scheduler
      return 1;
    })
  | continues_on(pool2.get_scheduler())
  | then([](int x) {
      // get_scheduler(env) returns pool2's scheduler
      return x + 1;
    });

This means read_env(get_scheduler) returns different schedulers at different stages of the pipeline — the context switch truly changes the environment.

Implementation Sketch

A simplified implementation of continues_on:

cpp
template<sender Sndr, scheduler Sch>
struct continues_on_sender {
  Sndr sndr_;
  Sch sch_;

  template<receiver R>
  struct operation {
    // Inner receiver: upon receiving set_value from upstream,
    // reschedules via sch_, then calls the outer receiver's set_value
  };
};

Key point: when the inner receiver receives set_value, it’s still on the current thread. Then continues_on schedules a “continue operation” via sch_, executing the downstream set_value on the target scheduler.

Limitations

continues_on only affects operations after it, not before. If you use let_value for dynamic branching, each branch may need its own independent continues_on.

Also, frequent continues_on calls (e.g., switching after every tiny operation) incur scheduling overhead — each switch may involve cross-thread task submission and synchronization.

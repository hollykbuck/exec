18 · starts_on / continues_on / on — Switching Execution Contexts

Standard sections: [exec.starts.on] [exec.continues.on] [exec.on] stdexec reference: examples/server_theme/on_transfer.cpp Related tests: tests/test_starts_on.cpp

Overview

These three adaptors handle execution context switching:

AdaptorEffect
starts_on(sch, sndr)Starts sndr on sch
continues_on(sndr, sch)Continues on sch after sndr completes
on(sch, sndr)Equivalent to starts_on(sch, continues_on(sch, sndr))

starts_on — Starting on a Specific Resource

cpp
// Corresponding test: test_starts_on.cpp → "starts_on launches work on a specific scheduler"
exec::static_thread_pool io_pool{2};
exec::static_thread_pool cpu_pool{4};

// Start the entire pipeline on io_pool
starts_on(io_pool.get_scheduler(),
  just(42)
    | then([](int x) { return x + 1; })
)
  | sync_wait;

Note: starts_on is a sender adaptor that takes two arguments:

cpp
// Function style
starts_on(sch, sndr)

// Pipeline style (sch as closure argument)
// starts_on(sch)(sndr) is equivalent to starts_on(sch, sndr)

continues_on — Switching to a New Resource

cpp
// Corresponding test: test_starts_on.cpp → "continues_on switches execution context"
// I/O completes on the thread pool → switch to CPU thread pool for computation
schedule(io_pool.get_scheduler())
  | then([] { return read_data(); })
  | continues_on(cpu_pool.get_scheduler())  // ← Switch point
  | then([](data d) { return compute(d); })
  | sync_wait;

Switching Semantics

graph LR
  A["read_data()"] --> B["Executes on I/O thread"]
  B --> C["continues_on(cpu)"]
  C --> D["Schedule to CPU thread pool"]
  D --> E["compute(d)"]
  E --> F["Executes on CPU thread"]

continues_on schedules subsequent operations onto a new scheduler after the previous sender completes.

on — Start and Stay on the Same Resource

cpp
// Corresponding test: test_starts_on.cpp → "on starts and continues on the same scheduler"
// Start on cpu_pool and always stay on cpu_pool
on(cpu_pool.get_scheduler(),
  just(42)
    | then([](int x) { return x + 1; })
    | then([](int x) { return x * 2; })
);

on(sch, sndr) is equivalent to starts_on(sch, continues_on(sch, sndr)).

Complete Example: Post-I/O Processing

From examples/server_theme/on_transfer.cpp:

cpp
// Corresponding test: test_starts_on.cpp
// HTTP image processing server
schedule(io_sched)
  | then([] { return recv_http_request(); })
  | continues_on(cpu_sched)                      // Switch to CPU for request processing
  | let_value([](http_request req) -> sender auto {
      return starts_on(io_sched,                  // Switch back to I/O for reading image
               then(just(req.path),
                  [](auto path) { return read_image(path); }));
    })
  | continues_on(cpu_sched)                      // Switch back to CPU for image processing
  | then([](image img) { return process_image(img); })
  | continues_on(io_sched)                       // Switch back to I/O for sending response
  | then([](response resp) { return send_response(resp); });

Execution Context and the Receiver Environment

When the execution context switches, the receiver’s environment also changes:

cpp
// Corresponding test: test_starts_on.cpp → "continues_on with same scheduler"
schedule(pool1.get_scheduler())
  | then([] {
      // get_scheduler(env) returns pool1's scheduler
      return 1;
    })
  | continues_on(pool2.get_scheduler())
  | then([](int x) {
      // get_scheduler(env) returns pool2's scheduler
      return x + 1;
    });

This means read_env(get_scheduler) returns different schedulers at different stages of the pipeline.

Brief Implementation Analysis

A simplified implementation approach for continues_on:

cpp
// Corresponding test: test_impl_then.cpp
template<sender Sndr, scheduler Sch>
struct continues_on_sender {
  Sndr sndr_;
  Sch sch_;

  template<receiver R>
  struct operation {
    // Inner receiver: after receiving set_value from the previous sender,
    // reschedule via sch_, then call the outer receiver's set_value
  };
};

Key point: When the inner receiver receives set_value, it is on the current thread. Then continues_on schedules a “continue operation” via sch_, executing the downstream set_value on the target scheduler.

18b · schedule_from and Scheduler Transition Internals

Standard Section: [exec.continues_on] [exec.schedule_from] [exec.on]
Draft Reference: exec.tex lines 3659–4004

The Hidden Adaptor

Chapter 18 showed you continues_on, starts_on, and on. But there is a fourth adaptor that makes them work — schedule_from.

schedule_from is not meant for user code. It is a customization hook that lets scheduler implementations control how execution leaves their resource.

schedule_from

cpp
execution::schedule_from(sndr)

It takes a sender and returns a parent sender that:

  1. Runs sndr to completion on the current execution agent
  2. Captures the child’s completion result (value, error, or stopped)
  3. Stores it in a variant inside the operation state
  4. On completion of sndr, schedules a continuation onto the destination scheduler (obtained from the receiver’s environment)
  5. The continuation replays the captured completion to the downstream receiver on the new resource

The note in the standard is explicit:

schedule_from is used by schedulers to control how to transition off of their schedulers’ associated execution contexts.

This is the off-ramp — the transition from the source scheduler’s resource.

How continues_on is Built

The default implementation of continues_on(sndr, sch) is:

cpp
make-sender(continues_on, sch, schedule_from(sndr))

The internal structure is:

continues_on (sndr, sch)
  └── schedule_from(sndr)     -- the child
       └── sndr               -- the original sender

When connected:

  1. schedule_from(sndr) connects and starts sndr
  2. sndr runs on whatever resource it was on
  3. On completion, schedule_from captures the result in a variant and schedules sch via connect(schedule(sch), schedule_receiver)
  4. The schedule_receiver reads the captured result from the variant and forwards it to the downstream receiver

The internal state type stores:

cpp
struct state-type {
  Rcvr& rcvr;                           // downstream receiver (on destination)
  variant<monostate,                     // captured results from sndr
          decayed-tuple<set_value_t, Args...>...,
          decayed-tuple<set_error_t, exception_ptr>>
    async-result;
  operation_t op-state;                  // connect(schedule(sch), receiver-type{this})
};

The receiver-type

The receiver connected to schedule(sch) is exposition-only:

cpp
struct receiver-type {
  state-type* state;

  void set_value() && noexcept {
    visit([this]<class Tuple>(Tuple& result) noexcept {
      if constexpr (!same_as<monostate, Tuple>) {
        auto& [tag, ...args] = result;
        tag(move(state->rcvr), move(args)...);
      }
    }, state->async-result);
  }

  template <class Error>
  void set_error(Error&& err) && noexcept {
    set_error(move(state->rcvr), forward<Error>(err));
  }

  void set_stopped() && noexcept {
    set_stopped(move(state->rcvr));
  }

  decltype(auto) get_env() const noexcept {
    return FWD-ENV(get_env(state->rcvr));
  }
};

When the scheduled work fires on sch:

  1. set_value() is called (the schedule operation succeeded)
  2. It visits the async-result variant
  3. It extracts the stored tag and args from the child’s completion
  4. It replays the completion on the downstream receiver — now executing on sch’s resource

The Two-Step Transition Model

The transition between execution resources is a two-step process:

Source Resource                    Destination Resource
  ┌──────────┐                     ┌──────────────┐
  │ sndr     │                     │ downstream   │
  │ runs     │                     │ runs here    │
  │ here     │                     │              │
  └────┬─────┘                     └──────┬───────┘
       │                                  ▲
       │  Step 1: continues_on            │
       │  (customized by source)          │ Step 2: schedule_from
       │                                  │ (customized by destination)
       └──────────────────────────────────┘

Step 1 (continues_on): The source scheduler can customize how it off-ramps. The default captures results in a variant and schedules a work item on the destination scheduler.

Step 2 (schedule_from): The destination scheduler controls the on-ramp. The default implementation of continues_on calls schedule(sch) which returns a sender that runs on sch. The destination scheduler can override what happens in the receiver-type above — for example, it might inject allocator information or set up thread-local state.

A custom scheduler that wants to control both off-ramp and on-ramp would specialize for both continues_on and schedule_from.

The Three-Argument form of on

The standard defines a three-argument form of on:

cpp
on(sndr, sch, closure)

This executes sndr on the current resource, transfers to sch, executes closure there, then transfers back:

on(sndr, sch, [](auto result) { return process(result); })

  Effect:
    1. sndr runs on current resource
    2. Result is transferred to sch
    3. closure(result) executes on sch
    4. closure's result is transferred back to the original resource

The default implementation expands to:

cpp
continues_on(
  write_env(
    let_value(sndr, closure),
    get_scheduler, get_completion_scheduler<set_value_t>(get_env(sndr))
  ),
  sch
)

The write_env call ensures the completion is routed back to the right scheduler.

Summary

ComponentRole
schedule_from(sndr)Captures child’s result, schedules replay on destination
continues_on(sndr, sch)make-sender(continues_on, sch, schedule_from(sndr))
on(sndr, sch)starts_on(sch, continues_on(sch, sndr))
on(sndr, sch, closure)Transfer to sch, execute closure, transfer back
receiver-typeExposition-only; reads captured result and replays on destination

The key insight: schedule_from is the transition primitive. Every context switch goes through it.

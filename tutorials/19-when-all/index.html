19 · when_all — Waiting for Multiple Senders in Parallel

Why Do We Need when_all?

Imagine a page that needs to query a database, call an external API, and read a local cache — three independent operations that could all run concurrently. If you called sync_wait three times sequentially, the total time would be the sum of all three — wasted time.

when_all solves this: launch multiple senders at once, wait for all results to arrive, then bundle them into a tuple for the downstream.

Basic Usage

cpp
namespace std::execution {
  inline constexpr when_all_t when_all{};
}

The simplest usage:

cpp
sync_wait(
  when_all(
    just(42),
    just(3.14),
    just("hello")
  )
);
// Result: optional{tuple{42, 3.14, "hello"}}

// With structured bindings
auto [i, d, s] = sync_wait(
  when_all(just(42), just(3.14), just("hello"))
).value();

The code above creates three senders via just. when_all launches them concurrently, waits for all three to finish, then bundles 42, 3.14, and "hello" into a tuple for sync_wait.

Real-World Example: Parallel Image Processing

From examples/server_theme/split_bulk.cpp — running three edge detection algorithms on the same image in parallel:

cpp
schedule(io_sched)
  | then([] { return load_image("photo.jpg"); })
  | let_value([](image img) -> sender auto {
      return when_all(
        schedule(cpu_sched) | then([img] { return sobel_edge_detect(img); }),
        schedule(cpu_sched) | then([img] { return canny_edge_detect(img); }),
        schedule(cpu_sched) | then([img] { return laplacian_edge_detect(img); })
      )
      | then([](auto sobel, auto canny, auto lap) {
          return merge_results(sobel, canny, lap);
        });
    })
  | then([](image result) {
      return save_result(result);
    });

The code above runs three detection algorithms on the same image, each on a separate CPU thread. when_all waits for all three results before merging. A sequential run would take 3x the time — when_all compresses it to roughly the time of the slowest algorithm.

Error Propagation

when_all’s error rule: if any child sender fails, the entire when_all fails.

cpp
sync_wait(
  when_all(
    just(42),
    just_error(std::make_exception_ptr(std::runtime_error{"oops"}))
  )
);
// Result: set_error triggers immediately; unstarted children are never started
// The entire when_all emits set_error

Cancellation Propagation

Likewise, if any child sender is cancelled:

cpp
sync_wait(
  when_all(
    just(42),
    just_stopped()
  )
);
// Result: set_stopped()

when_all_with_variant —— when_all for Heterogeneous Results

When child senders have different value types, when_all_with_variant packs the results into a variant:

cpp
// when_all requires all child senders' value signatures to be consistent
// when_all_with_variant packs results into variant<tuple<Ts...>, tuple<Us...>>

using Sigs = completion_signatures_of_t<
  decltype(when_all_with_variant(just(42), just(3.14)))
>;
// value type: variant<tuple<int>, tuple<double>>

Composing with then

cpp
sync_wait(
  when_all(
      schedule(pool.get_scheduler()) | then(work1),
      schedule(pool.get_scheduler()) | then(work2),
      schedule(pool.get_scheduler()) | then(work3)
  )
  | then([](auto r1, auto r2, auto r3) {
      return combine(r1, r2, r3);
    })
);

The code above runs three jobs concurrently, then merges the results via then.

Completion Signatures of when_all

cpp
when_all(
  sender<set_value_t(int)>,         // value: tuple<int>
  sender<set_value_t(double, char)> // value: tuple<double, char>
)
// Result value signature: set_value_t(int, double, char)
// i.e., flattens all child senders' value types into one big tuple

Key Points

  1. All child senders launch concurrently — no deterministic execution order
  2. Short-circuit logic: any child failing/cancelling triggers an immediate short-circuit
  3. No shared scheduler required: child senders can run on different schedulers
  4. Resource management: when_all’s operation state holds all child sender operation states

19 · when_all — Waiting for Multiple Senders in Parallel

Standard section: [exec.when.all] stdexec reference: examples/server_theme/split_bulk.cpp Related tests: tests/test_when_all.cpp, examples/hello_world.cpp

Overview

when_all launches multiple senders concurrently, waits for all of them to complete, and aggregates their results into a tuple.

cpp
namespace std::execution {
  inline constexpr when_all_t when_all{};
}

Basic Usage

cpp
// Corresponding test: test_when_all.cpp → "when_all waits for all senders"
sync_wait(
  when_all(
    just(42),
    just(3.14),
    just("hello")
  )
);
// Result: optional{tuple{42, 3.14, "hello"}}

// Using structured bindings
auto [i, d, s] = sync_wait(
  when_all(just(42), just(3.14), just("hello"))
).value();

Error Propagation

when_all’s error propagation rule: if any child sender fails, the entire when_all fails.

cpp
// Corresponding test: test_when_all.cpp → "when_all error propagation"
sync_wait(
  when_all(
    just(42),
    just_error(std::make_exception_ptr(std::runtime_error{"oops"}))
  )
);
// Result: set_error triggers immediately, other child operations that haven't started won't start
// The entire when_all emits set_error

Cancellation Propagation

Similarly, if any child sender is cancelled:

cpp
// Corresponding test: test_when_all.cpp → "when_all stopped propagation"
sync_wait(
  when_all(
    just(42),
    just_stopped()
  )
);
// Result: set_stopped()

when_all_with_variant — when_all for Heterogeneous Results

If multiple child senders have different completion signatures (but the value types can be unified into a variant):

cpp
// Corresponding test: test_when_all.cpp → "when_all_with_variant compiles with just values"
// when_all requires all child senders' value signatures to be consistent (at the value-type level)
// when_all_with_variant packs results into variant<tuple<Ts...>, tuple<Us...>>

using Sigs = completion_signatures_of_t<
  decltype(when_all_with_variant(just(42), just(3.14)))
>;
// value type: variant<tuple<int>, tuple<double>>

Composing with then

cpp
// Corresponding test: test_when_all.cpp → "when_all with then composition"
sync_wait(
  when_all(
      schedule(pool.get_scheduler()) | then(work1),
      schedule(pool.get_scheduler()) | then(work2),
      schedule(pool.get_scheduler()) | then(work3)
  )
  | then([](auto r1, auto r2, auto r3) {
      return combine(r1, r2, r3);
    })
);

Server Theme Example: Parallel Image Processing

From examples/server_theme/split_bulk.cpp:

cpp
// Corresponding test: test_when_all.cpp → "when_all on thread pool"
// Dispatch image to three parallel edge detectors
schedule(io_sched)
  | then([] { return load_image("photo.jpg"); })
  | let_value([](image img) -> sender auto {
      return when_all(
        schedule(cpu_sched) | then([img] { return sobel_edge_detect(img); }),
        schedule(cpu_sched) | then([img] { return canny_edge_detect(img); }),
        schedule(cpu_sched) | then([img] { return laplacian_edge_detect(img); })
      )
      | then([](auto sobel, auto canny, auto lap) {
          return merge_results(sobel, canny, lap);
        });
    })
  | then([](image result) {
      return save_result(result);
    });

Completion Signatures of when_all

The completion signatures of when_all are the Cartesian product of all child senders’ completion signatures:

cpp
when_all(
  sender<set_value_t(int)>,         // value: tuple<int>
  sender<set_value_t(double, char)> // value: tuple<double, char>
)
// Result value signature: set_value_t(int, double, char)
// i.e., flatten all child senders' value types into one big tuple

Notes

  1. All child senders launch concurrently — no deterministic execution order
  2. Short-circuit logic: any child sender failing/cancelling causes immediate short-circuit
  3. No requirement for the same scheduler: child senders can run on different schedulers
  4. Resource management: when_all’s operation state holds all child sender operation states

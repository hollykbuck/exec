20 · bulk — Data Parallel Operations

On this page

The Bottleneck of for Loops

When you process a million elements with a for loop on a single thread, every other CPU core sits idle. A for loop is sequential — element 1 must finish before element 2 starts.

bulk takes a completely different approach: it hands N independent operations to the scheduler all at once, letting the scheduler decide how to distribute them across multiple threads (or even thousands of GPU cores) for parallel execution.

If the underlying scheduler has only one thread, bulk degrades to a sequential for loop. On a GPU scheduler, it becomes a CUDA kernel running on thousands of cores in parallel.

Basic Syntax

cpp
namespace std::execution {
  inline constexpr bulk_t bulk{};
  inline constexpr bulk_chunked_t bulk_chunked{};
  inline constexpr bulk_unchunked_t bulk_unchunked{};
}

These are the three variants. The most basic form:

cpp
bulk(sndr, shape, f)
// shape: size of the parallel index space
// f: callback, signature invocable(Idx, Ts...) → void
// For each idx ∈ [0, shape), calls f(idx, upstream values...)

Basic Usage

cpp
auto result = sync_wait(
  just(100)
    | bulk(10, [](int idx, int& total) {
        // idx ranges [0, 10), total = 100
        // These callbacks may execute concurrently on multiple threads
        printf("idx %d, total %d\n", idx, total);
      })
);

The code above: just(100) produces the value 100, and bulk(10, ...) passes it to 10 parallel callbacks. Each callback sees the same total = 100 (hence int& — passed by reference), but a different idx.

bulk vs for

for loopbulk
Single thread, sequential; index order guaranteedMulti-thread/multi-core parallel; index execution order not guaranteed
Operates on data directlyData flows through sender pipeline
Callbacks share variables directlyShared data requires external synchronization (atomics, locks)

Execution Semantics

graph LR
  A["Input: set_value(Ts...)"] --> B["For each idx ∈ [0, shape)"]
  B --> C["Execute f(idx, Ts...)<br/>may be parallel on<br/>different execution agents"]
  C --> D["Output: set_value(Ts...)<br/>input values pass through unchanged"]

Key Properties

  • shape defines the parallel index space [0, shape)
  • f is called once per index, calls may run on different execution agents
  • f receives the index and upstream values (as references)
  • Side effects of f (writing shared data) require external synchronization

Parallel Sum Example

cpp
std::vector<int> data = {1, 2, 3, 4, 5, 6, 7, 8};
std::atomic<int> sum{0};

sync_wait(
  just(std::ref(data), std::ref(sum))
    | bulk(data.size(), [](int idx, auto& data, auto& sum) {
        sum.fetch_add(data[idx], std::memory_order_relaxed);
      })
);
// sum = 36

The code above: data has 8 elements, bulk spawns 8 parallel callbacks. Each callback reads data[idx] and atomically adds it to sum. Note the std::atomic — a plain int with concurrent writes is undefined behavior.

bulk_chunked and bulk_unchunked

The standard provides three variants:

VariantBehavior
bulkDefault; scheduler decides whether to chunk
bulk_chunkedGuarantees chunking (index range split into contiguous chunks, one chunk per execution agent)
bulk_unchunkedGuarantees no chunking (one execution agent per index)

Bulk on GPU

nvexec (NVIDIA’s GPU backend) provides a GPU implementation of bulk:

cpp
nvexec::stream_scheduler gpu_sch{};

starts_on(gpu_sch,
  just(device_buffer)
    | nvexec::bulk(1024, [] __device__(int idx, device_buffer& buf) {
        buf[idx] = buf[idx] * 2 + 1;
      })
);

The code above processes 1024 elements on the GPU with 1024 CUDA threads, each thread handling one index.

Completion Signatures

cpp
// bulk's completion signature = input sender's completion signature (passthrough)
decltype(just(42) | bulk(10, [](int, int&){}))
// Completion signature: completion_signatures<set_value_t(int)>

Limitations

There is no guaranteed execution order between index callbacks — index 9 may finish before index 1. Callbacks must not have data dependencies on each other (e.g., using bulk for a prefix sum is incorrect).

Shared data requires external synchronization (atomics, mutexes, or scheduler-provided sync primitives). If you need sequential “each step depends on the previous” semantics, use a traditional then chain instead of bulk.

Also, bulk callbacks can only return void — it’s designed for side-effect operations. You cannot pass per-index computation results downstream; collect them yourself in a shared container inside the callback.

The Type Divergence Problem

C++‘s type system is static. When you branch inside let_value, different types cause compile errors:

cpp
sender auto branch(int x) -> sender auto {
  if (x > 0)
    return just(x);     // set_value_t(int)
  else
    return just();      // set_value_t()  —— different type!
}

A single pipeline produces two different value types — the compiler rejects this. A sender’s type must be determined at compile time; it can’t be int at one moment and void at the next.

into_variant, stopped_as_optional, and stopped_as_error are the three adaptors that solve this “type divergence” problem.

Overview

into_variant(sndr)Unifies multiple possible value types into variant<Ts...>
stopped_as_optional(sndr)Converts set_stopped to set_value(optional<T>)
stopped_as_error(sndr, err)Converts set_stopped to set_error

into_variant —— Unifying Value Types

When a sender may have multiple value completion signatures, into_variant packs them into a single variant:

cpp
// Conditional branches produce different value types
sender auto branch(int x) -> sender auto {
  if (x > 0)
    return just(x);        // set_value_t(int)
  else
    return just();         // set_value_t()
}

// Direct use fails: value_types_of_t has multiple types
auto s1 = just(42) | then([](int x) { return branch(x); }) | sync_wait;
// Compile error!

// Unify with into_variant
auto s2 = just(42)
        | then([](int x) { return branch(x); })
        | into_variant()
        | sync_wait;
// Result: variant<tuple<int>, tuple<>>

In the code above, branch(42) returns just(42) (type int), while branch(-1) returns just() (type void). into_variant() packs both possibilities into variant<tuple<int>, tuple<>>. The downstream can then use std::visit to handle either case:

cpp
auto var = sync_wait(s2).value();
std::visit([](auto& tup) {
  if constexpr (tuple_size<decltype(tup)>{} > 0)
    print("Has value: {}\n", get<0>(tup));
  else
    print("Empty\n");
}, var);

stopped_as_optional —— Turning Cancellation into Optional

cpp
template<sender Sndr>
sender auto stopped_as_optional(Sndr sndr);

If the sender is cancelled (set_stopped), convert it to set_value(std::optional<T>{}):

cpp
auto result = sync_wait(
  just(42)
    | stopped_as_optional
);
// result = optional{tuple{optional{42}}}

auto cancelled = sync_wait(
  just_stopped()
    | stopped_as_optional
);
// result = optional{tuple{optional<int>{}}}  (value = nullopt)

The code above shows the effect of stopped_as_optional: success fills the optional, cancellation leaves it empty. This is perfect for cache-lookup scenarios — a cache hit has a value, a miss is just business as usual; no need to distinguish “miss” from “cancelled.”

Usage in Coroutines

cpp
sender auto async_fn() -> stdexec::task<std::optional<int>> {
  co_return co_await stopped_as_optional(some_sender);
}
// If some_sender is cancelled, async_fn returns nullopt

stopped_as_error —— Turning Cancellation into an Error

cpp
template<sender Sndr, class Error>
sender auto stopped_as_error(Sndr sndr, Error err);

If you consider cancellation to be an error (e.g., a transaction cancelled mid-way should throw), use stopped_as_error:

cpp
auto result = sync_wait(
  just_stopped()
    | stopped_as_error(std::make_exception_ptr(
        std::runtime_error{"operation cancelled"}))
);
// throws runtime_error

Type Unification for Container Storage

into_variant is especially useful when you need to store heterogeneous senders in a container:

cpp
// Can't store in vector: heterogeneous types
std::vector<sender auto> senders;  // Compile error!

// Use into_variant to unify types
std::vector<decltype(just(0) | into_variant())> senders;
senders.push_back(just(42) | into_variant());
senders.push_back(just() | into_variant());
// OK: all elements have the same type

Relationship with when_all_with_variant

when_all_with_variant is essentially when_all + into_variant combined:

cpp
// Roughly equivalent
when_all(s1 | into_variant(), s2 | into_variant())

The difference is that when_all_with_variant handles variant expansion internally, which may be more efficient.

Limitations

into_variant introduces variant overhead at runtime — every access requires std::visit or std::get. If you only have a few branches (typically 2-3), this is negligible, but if there are many branches or frequent calls, consider reducing the number of branches at the design level.

stopped_as_optional adds a layer of wrapping — set_value(int) becomes set_value(optional<int>), and the downstream must unwrap one extra level. In deeply nested pipelines this can make types verbose.

22 · sync_wait / sync_wait_with_variant —— Blocking Wait for Results

Standard Section: [exec.sync.wait] [exec.sync.wait.var] stdexec Reference: examples/hello_world.cpp Related Tests: tests/test_pipe.cpp

Overview

sync_wait is the most important sender consumer — it pulls the asynchronous world back into the synchronous world.

cpp
namespace std::this_thread {
  struct sync_wait_t {};
  inline constexpr sync_wait_t sync_wait{};

  struct sync_wait_with_variant_t {};
  inline constexpr sync_wait_with_variant_t sync_wait_with_variant{};
}

Note: sync_wait is in the std::this_thread namespace, not std::execution.

Internal Mechanism

cpp
// Simplified pseudo-code of sync_wait implementation
auto sync_wait(sender auto sndr) {
  run_loop loop;
  auto sch = loop.get_scheduler();

  // Create a receiver that provides run_loop scheduler in its environment
  // Connect sndr to this receiver
  auto op = connect(sndr, sync_wait_receiver{loop});

  start(op);      // Start the async operation
  loop.run();     // Block until a completion signal is received

  // After loop.run() returns, the completion signal has been written
  if (value_received)  return optional{tuple{...}};
  if (error_received)  throw error;
  if (stopped)         return nullopt;
}

Usage

cpp
// Corresponding test: test_basics.cpp → "just creates an immediate-value sender"
// Simple value
auto [i] = sync_wait(just(42)).value();

// Multiple values
auto [a, b, c] = sync_wait(when_all(just(1), just(2), just(3))).value();

// Error
auto opt = sync_wait(just_error(std::exception_ptr{}));
// opt == nullopt, exception is rethrown

// Cancellation
auto opt = sync_wait(just_stopped());
// opt == nullopt

sync_wait_with_variant

Use sync_wait_with_variant when the sender has multiple possible value types:

cpp
sender auto maybe(int x) -> sender auto {
  if (x > 0) return just(x);
  else return just();
}

// sync_wait(maybe(42));           // Compile error: multiple value types
auto r = sync_wait_with_variant(maybe(42)).value();
// r type: variant<tuple<int>, tuple<>>

Current Thread’s run_loop

hello_world.cpp demonstrates the run_loop scheduler provided by sync_wait:

cpp
// Corresponding test: test_run_loop.cpp → "run_loop scheduler can be used in let_value"
// sync_wait provides a get_scheduler in the receiver environment
// This scheduler is associated with a run_loop, driven inside sync_wait

auto t = sync_wait(get_scheduler()).value();
// t is tuple<run_loop::scheduler>
// This scheduler can be used in the pipeline:

auto y = let_value(get_scheduler(), [](auto sched) {
  return starts_on(sched, then(just(), [] {
    std::cout << "from run_loop\n";
    return 42;
  }));
});
sync_wait(std::move(y));

Comparison with Other Consumers

ConsumerBehaviorUse Case
sync_waitBlocks until completionSync boundary, main function
sync_wait_with_variantBlocks, unifies typesPolymorphic value types
spawnFire-and-forget, non-blockingBackground tasks
ensure_startedStarts immediately, keeps senderNeed to reference result

Notes

  1. Blocks current thread: Do not call inside an asynchronous thread (it will block that thread)
  2. Exception safety: set_error throws, needs to be used outside try/catch
  3. Once only: sync_wait can only consume once (sender is move-only)

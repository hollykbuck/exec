22 · sync_wait / sync_wait_with_variant —— Blocking Wait for Results

Why Do We Need sync_wait?

Senders live in the async world. But your main function is synchronous, and so are your unit test assert calls. At the final step, you always need a “bridge” to pull async results back into the synchronous world.

sync_wait is that bridge.

Basic Behavior

cpp
namespace std::this_thread {
  struct sync_wait_t {};
  inline constexpr sync_wait_t sync_wait{};

  struct sync_wait_with_variant_t {};
  inline constexpr sync_wait_with_variant_t sync_wait_with_variant{};
}

Note: sync_wait lives in std::this_thread, not std::execution — it’s a “thread operation,” not part of the sender system proper.

cpp
template<sender Sndr>
  requires sender_in<Sndr>

Internal Mechanism

cpp
auto sync_wait(sender auto sndr) {
  run_loop loop;
  auto sch = loop.get_scheduler();

  auto op = connect(sndr, sync_wait_receiver{loop});

  start(op);      // Start the async operation
  loop.run();     // Block until a completion signal is received

  if (value_received)  return optional{tuple{...}};
  if (error_received)  throw error;
  if (stopped)         return nullopt;
}

The code above reveals sync_wait’s essence: it creates a run_loop internally, pins the async operation onto this loop, then blocks the current thread running the loop until a completion signal arrives.

Usage

cpp
// Simple value
auto [i] = sync_wait(just(42)).value();

// Multiple values
auto [a, b, c] = sync_wait(when_all(just(1), just(2), just(3))).value();

// Error
auto opt = sync_wait(just_error(std::exception_ptr{}));
// opt == nullopt, exception is rethrown

// Cancellation
auto opt = sync_wait(just_stopped());
// opt == nullopt

sync_wait_with_variant

Use sync_wait_with_variant when the sender has multiple possible value types:

cpp
sender auto maybe(int x) -> sender auto {
  if (x > 0) return just(x);
  else return just();
}

// sync_wait(maybe(42));           // Compile error: multiple value types
auto r = sync_wait_with_variant(maybe(42)).value();
// r type: variant<tuple<int>, tuple<>>

sync_wait’s run_loop Scheduler

cpp
auto t = sync_wait(get_scheduler()).value();
// t is tuple<run_loop::scheduler>

auto y = let_value(get_scheduler(), [](auto sched) {
  return starts_on(sched, then(just(), [] {
    std::cout << "from run_loop\n";
    return 42;
  }));
});
sync_wait(std::move(y));

When to Use and When Not to Use

Use sync_wait for:

(1) The final step of main — printing results, saving to a file, or returning as a process exit code. (2) Asserting async results in unit tests — this is sync_wait’s most common use. (3) One-shot batch operations in CLI tools — start, finish, exit.

Don’t use sync_wait for:

(1) Hot paths or loops — each sync_wait creates an internal run_loop, which has overhead. (2) UI threads — will freeze the interface. (3) Request-handling loops in network servers — will block the event loop, killing throughput for all other requests.

Limitations

sync_wait blocks the calling thread. If that thread is an IO thread or event-loop thread, the entire program will appear to hang.

Also, sync_wait creates and drives a temporary run_loop internally, which has some runtime overhead. In high-frequency scenarios (hundreds of thousands of calls per second), this overhead accumulates.

After calling sync_wait, the sender is moved-from and consumed. A given sender can only be sync_wait-ed once. If you need to wait multiple times, prepend ensure_started (see chapter 23).

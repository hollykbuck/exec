23 · spawn / ensure_started —— Fire-and-Forget & Eager Start

The Shortcomings of sync_wait

The previous chapter covered sync_wait — it’s convenient, but it blocks the current thread until the task finishes. In a server program, you can’t block an entire thread just to handle one request. You need to “start and move on” — not stand waiting at the kitchen door.

spawn and ensure_started are two non-blocking ways to kick off work.

Analogy: Three Kinds of Consumers

Think of async operations as ordering food at a restaurant:

| sync_wait | Stand at the kitchen door waiting — grab the dish as soon as it’s ready | | spawn | Order and walk away — someone else will handle the food when it’s done | | ensure_started | Order ahead (prep in advance) — call for the dish when you’re ready to eat |

Overview

ConsumerBehavior
spawn(sndr)Starts async operation, does not wait (fire-and-forget), returns void
ensure_started(sndr)Starts async operation, returns a sender you can wait on later

spawn —— Fire-and-Forget

cpp
namespace std::execution {
  inline constexpr spawn_t spawn{};
}

// Start an async operation, don't care about the result
spawn(
  schedule(sch)
    | then([] { do_some_work(); })
);

// spawn returns void

The code above starts an async operation and returns immediately. The result is discarded — whether success, failure, or cancellation, spawn doesn’t track it.

spawn Lifecycle

Operations started by spawn must complete before their associated resources are destroyed, or behavior is undefined.

spawn Internals

The internal flow is simple:

  1. Create a default receiver (handles all three completion signals but discards results)
  2. connect(sndr, default_receiver)
  3. start(op)
  4. Return (do not wait for completion)

ensure_started —— Eager Start

cpp
// Start the async operation immediately, but keep a sender to collect the result later
sender auto work = ensure_started(
  schedule(sch) | then(heavy_computation)
);

// work is already running in the background
// You can wait for the result later
auto [result] = sync_wait(std::move(work)).value();

ensure_started vs Normal Sender

cpp
// Lazy: doesn't execute at construction time
sender auto lazy = schedule(sch) | then(work);

// Eager: starts executing immediately
sender auto eager = ensure_started(schedule(sch) | then(work));

// Both wait for results via sync_wait
sync_wait(std::move(lazy));   // Execution only starts now
sync_wait(std::move(eager));  // May already be nearly done

The code above highlights the key difference: normal senders are lazy — execution begins only after connect + start. ensure_started is eager — it starts at construction time; sync_wait just waits for the result.

Cooperating with async_scope

spawn and ensure_started are often paired with async_scope to manage task lifecycles:

cpp
exec::async_scope scope;

scope.spawn(schedule(sch) | then(work1));
scope.spawn(schedule(sch) | then(work2));

sync_wait(scope.on_empty());  // Wait for all work in the scope to finish

From examples/scope.cpp:

cpp
exec::static_thread_pool ctx{1};
exec::async_scope scope;

scheduler auto sch = ctx.get_scheduler();

scope.spawn(schedule(sch) | then([] { printf("void\n"); }));

sender auto future = scope.spawn_future(
  schedule(sch) | then([] { return 42; })
);

sender auto empty = scope.on_empty();

sync_wait(when_all(empty, future | then([](int x) { printf("%d\n", x); })));

spawn’s Receiver

spawn uses a special default receiver internally:

cpp
struct spawn_receiver {
  using receiver_concept = stdexec::receiver_tag;

  void set_value(auto&&...) noexcept {}
  void set_error(std::exception_ptr) noexcept { std::terminate(); }
  void set_stopped() noexcept {}

  auto get_env() const noexcept {
    return stdexec::prop{get_stop_token, stdexec::never_stop_token{}};
  }
};

Warning: set_error calls std::terminate() by default! Always ensure spawned operations can’t throw, or pre-handle errors with upon_error:

cpp
spawn(
  schedule(sch)
    | then(maybe_fail)
    | upon_error([](auto e) { /* handle error, avoid terminate */ })
);

Use Case Comparison

ScenarioRecommended
main waiting for multiple async taskssync_wait(when_all(...))
Background logging/monitoring with no interest in resultsspawn(...)
Start computation early, collect result laterensure_started(...)
Manage task lifecycleasync_scope::spawn() / async_scope::spawn_future()

Limitations

spawn discards its operation’s result. If an exception occurs without upon_error as a safety net, set_error will call std::terminate() — catastrophic in production. Use spawn only for side-path operations where error handling is complete and you truly don’t care about the outcome.

After ensure_started starts, the returned sender can be sync_wait-ed multiple times (since it internally holds the completed result), but it cannot be connect-ed multiple times — it remains a move-only sender.

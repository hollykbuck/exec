23 · spawn / ensure_started —— Fire-and-Forget & Eager Start

Standard Section: [exec.spawn] [exec.ensure_started] stdexec Reference: examples/scope.cpp Related Tests: tests/test_spawn_scope.cpp

Overview

spawn and ensure_started are sender consumers that immediately start asynchronous operations, without waiting synchronously like sync_wait.

ConsumerBehavior
spawn(sndr)Starts async operation, does not wait for completion (fire-and-forget)
ensure_started(sndr)Starts async operation, returns a sender that can await its result

spawn —— Fire-and-Forget

cpp
// Corresponding test: test_spawn_scope.cpp → "spawn via async_scope"
namespace std::execution {
  inline constexpr spawn_t spawn{};
}

// Start an async operation, don't care about the result
spawn(
  schedule(sch)
    | then([] { do_some_work(); })
);

// spawn returns void

spawn Lifecycle

Operations started by spawn must complete before their resources are destroyed, otherwise behavior is undefined.

Internal Mechanism of spawn

  1. Creates a default receiver (handles three completion signals but does not save results)
  2. connect(sndr, default_receiver)
  3. start(op)
  4. Returns (does not wait for completion)

ensure_started —— Eager Start

cpp
// Corresponding test: test_spawn_scope.cpp → "spawn_future and collect results"
// Immediately start the async operation, but retain a sender to observe the result
sender auto work = ensure_started(
  schedule(sch) | then(heavy_computation)
);

// work is already executing in the background
// Can await the result later
auto [result] = sync_wait(std::move(work)).value();

ensure_started vs Normal Sender

cpp
// Lazy: does not execute at construction time
sender auto lazy = schedule(sch) | then(work);

// Eager: starts executing immediately
sender auto eager = ensure_started(schedule(sch) | then(work));

// Both wait for results via sync_wait
sync_wait(std::move(lazy));   // Execution starts only now
sync_wait(std::move(eager));  // May already be nearly done

Cooperation with async_scope

spawn and ensure_started are often used with async_scope to manage lifecycle:

cpp
// Corresponding test: test_spawn_scope.cpp → "scope ensures structured concurrency"
exec::async_scope scope;

// Spawn work inside the scope
scope.spawn(schedule(sch) | then(work1));
scope.spawn(schedule(sch) | then(work2));

// Wait for all work in the scope to complete
sync_wait(scope.on_empty());

From examples/scope.cpp:

cpp
exec::static_thread_pool ctx{1};
exec::async_scope scope;

scheduler auto sch = ctx.get_scheduler();

// spawn fire-and-forget
scope.spawn(schedule(sch) | then([] { printf("void\n"); }));

// spawn_future starts and returns an observable sender
sender auto future = scope.spawn_future(
  schedule(sch) | then([] { return 42; })
);

// Triggered when scope is empty
sender auto empty = scope.on_empty();

// Wait for all to complete
sync_wait(when_all(empty, future | then([](int x) { printf("%d\n", x); })));

spawn’s Receiver

spawn internally uses a special default receiver:

cpp
// Corresponding test: test_spawn_scope.cpp
// spawn's default receiver (simplified)
struct spawn_receiver {
  using receiver_concept = stdexec::receiver_tag;

  void set_value(auto&&...) noexcept {}
  void set_error(std::exception_ptr) noexcept { std::terminate(); }
  void set_stopped() noexcept {}

  auto get_env() const noexcept {
    return stdexec::prop{get_stop_token, stdexec::never_stop_token{}};
  }
};

Note: set_error calls std::terminate() by default! So ensure spawn internals do not error, or use upon_error to handle errors:

cpp
spawn(
  schedule(sch)
    | then(maybe_fail)
    | upon_error([](auto e) { /* handle error, no terminate */ })
);

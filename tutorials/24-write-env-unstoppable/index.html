24 · write_env / unstoppable —— Modifying Environment Properties

Standard Section: [exec.write.env] [exec.unstoppable] stdexec Reference: include/stdexec/__detail/ Related Tests: tests/test_write_env.cpp

Overview

write_env and unstoppable allow you to modify the environment properties of a receiver in a pipeline.

AdaptorEffect
write_env(sndr, props...)Injects environment properties downstream
unstoppable(sndr)Disables stop token (ignores cancellation)

write_env —— Injecting Environment Properties

cpp
// Corresponding test: test_write_env.cpp → "write_env modifies the receiver environment"
// Standard definition
namespace std::execution {
  inline constexpr unspecified write_env{};
}

// Usage: write_env(sndr, prop{query, value}...)
sync_wait(
  just(42)
    | write_env(prop{get_scheduler, my_scheduler})  // Replace downstream scheduler
    | then([](int x) {
        // Here get_scheduler(env) returns my_scheduler
        return x;
      })
);

Multiple Properties

cpp
// Corresponding test: test_write_env.cpp → "write_env chains multiple properties"
sync_wait(
  just(42)
    | write_env(
        prop{get_scheduler, cpu_scheduler},
        prop{get_allocator, my_allocator},
        prop{my_custom_query, custom_value}
      )
    | then([](int x) {
        // Environment contains three custom properties
        return x;
      })
);

unstoppable —— Disabling Cancellation

cpp
// Corresponding test: test_write_env.cpp → "write_env with never_stop_token"
// Standard definition
namespace std::execution {
  inline constexpr unspecified unstoppable{};
}

// Replace upstream stop_token with never_stop_token
auto result = sync_wait(
  cancellable_work
    | unstoppable      // This pipeline segment is no longer cancellable
);

unstoppable is actually a special case of write_env — it replaces the result of the get_stop_token query with never_stop_token.

Implementation Principle

cpp
// Equivalent implementation of unstoppable
write_env(sndr, prop{get_stop_token, never_stop_token{}})

Environment Property Merging Rules

write_env uses override semantics:

graph LR
  A["Existing env: {get_scheduler → A, get_stop_token → B}"]
  B["write_env(prop{get_scheduler → C})"]
  C["New env: {get_scheduler → C, get_stop_token → B}<br/>↑ Overridden"]
  A --> B --> C

write_env does not modify the upstream environment; it stacks new property layers on top of the existing environment.

Use Cases

Scenario 1: Replacing Scheduler in Tests

cpp
// Corresponding test: test_write_env.cpp
// Production code uses a thread pool
auto pipeline = just(42) | then(heavy_compute);

// In tests, replace with inline_scheduler (synchronous, easier to debug)
auto test_pipeline = pipeline
  | write_env(prop{get_scheduler, exec::inline_scheduler{}});

Scenario 2: Specifying Custom Allocator

cpp
custom_allocator alloc;

auto pipeline = fetch_large_data()
  | write_env(prop{get_allocator, alloc})
  | then([](data d) { /* use memory allocated by alloc */ });

Scenario 3: Protecting Non-Cancellable Operations

cpp
// Corresponding test: test_write_env.cpp
// Critical write operation is non-cancellable
schedule(sch)
  | then(read_data)
  | unstoppable                // Non-cancellable from here on
  | then(write_to_database)
  | upon_stopped([] { /* will never be executed */ });

Environment Propagation Rules

Review the entire environment propagation chain:

graph LR
  A["Outer receiver's environment"]
  A --> B["connect(sndr, outer_rcvr)"]
  B --> C["sndr's operation state gets the environment"]
  C --> D["Passed to child sender's receiver"]
  D --> E["write_env can overlay properties"]
  E --> F["Final child receiver's environment = original + write_env overlays"]

write_env creates an environment layer within the same pipeline, affecting only downstream senders and receivers.

24 · write_env / unstoppable —— Modifying Environment Properties

A Callback to Chapter 14

Chapter 14 taught us how to read environment information (read_env). This chapter covers how to write it (write_env). The two adaptors relate like C++ getters and setters, or like Unix shell environment variables:

bash
$ env VAR=value command

You temporarily override an environment variable’s value, but only for that command — the global environment is untouched. write_env does exactly the same thing, scoped to the downstream of the current pipeline.

Overview

AdaptorEffect
write_env(sndr, props...)Injects environment properties downstream
unstoppable(sndr)Disables the stop token (ignores cancellation)

write_env —— Injecting Environment Properties

cpp
namespace std::execution {
  inline constexpr unspecified write_env{};
}

// Usage: write_env(sndr, prop{query, value}...)
sync_wait(
  just(42)
    | write_env(prop{get_scheduler, my_scheduler})  // Override downstream scheduler
    | then([](int x) {
        // get_scheduler(env) returns my_scheduler here
        return x;
      })
);

The code above inserts an environment override between just(42) and then. The downstream then sees the scheduler as my_scheduler rather than the original.

Setting Multiple Properties

cpp
sync_wait(
  just(42)
    | write_env(
        prop{get_scheduler, cpu_scheduler},
        prop{get_allocator, my_allocator},
        prop{my_custom_query, custom_value}
      )
    | then([](int x) {
        // Environment now contains all three custom properties
        return x;
      })
);

unstoppable —— Disabling Cancellation

cpp
namespace std::execution {
  inline constexpr unspecified unstoppable{};
}

auto result = sync_wait(
  cancellable_work
    | unstoppable      // This segment of the pipeline can no longer be cancelled
);

unstoppable is actually a special case of write_env — it replaces the get_stop_token query result with never_stop_token:

cpp
// Equivalent implementation of unstoppable
write_env(sndr, prop{get_stop_token, never_stop_token{}})

Environment Property Override Rules

write_env uses override semantics. It doesn’t modify the upstream environment; it stacks a new property layer on top:

graph LR
  A["Existing env: {get_scheduler → A, get_stop_token → B}"]
  B["write_env(prop{get_scheduler → C})"]
  C["New env: {get_scheduler → C, get_stop_token → B}↑ Overridden"]
  A --> B --> C

Typical Use Cases

Scenario 1: Replacing the Scheduler in Tests

cpp
// Production code uses a thread pool
auto pipeline = just(42) | then(heavy_compute);

// Tests swap in inline_scheduler (synchronous, easier to debug)
auto test_pipeline = pipeline
  | write_env(prop{get_scheduler, exec::inline_scheduler{}});

Scenario 2: Specifying a Custom Allocator

cpp
custom_allocator alloc;

auto pipeline = fetch_large_data()
  | write_env(prop{get_allocator, alloc})
  | then([](data d) { /* this memory is allocated by alloc */ });

Scenario 3: Protecting Non-Cancellable Critical Operations

cpp
schedule(sch)
  | then(read_data)
  | unstoppable                // Non-cancellable from here onward
  | then(write_to_database)    // Database write must complete
  | upon_stopped([] { /* will never execute */ });

The code above uses write_to_database for a database transaction commit — an operation that must either not happen at all, or complete fully. unstopped ensures the write isn’t interrupted even if upstream sends a cancellation signal.

Environment Propagation Chain

graph LR
  A["Outer receiver's environment"]
  A --> B["connect(sndr, outer_rcvr)"]
  B --> C["sndr's operation state gets the environment"]
  C --> D["Passed to child sender's receiver"]
  D --> E["write_env can overlay properties"]
  E --> F["Final child receiver env = original + write_env overlays"]

write_env creates an environment layer within the pipeline, affecting only the downstream senders and receivers.

Limitations

write_env only affects its downstream receiver environment. If the pipeline forks (e.g., when_all splitting into multiple child senders), you need to carefully consider the scope of environment overrides — write_env covers the current linear chain, and different branches may see different environments.

unstopped forcibly ignores cancellation signals, which can cause resource leaks — if a downstream operation runs long and can’t be cancelled, callers may wait forever. Use it only in truly critical regions where the operation must complete and is guaranteed to finish promptly on its own.

25 · Query System & Environment

Standard Section: [exec.queries]
stdexec Reference: include/stdexec/functional.hpp, include/stdexec/__detail/__env.hpp
Related Tests: tests/test_env_query.cpp · tests/test_stop_token.cpp

Overview

The query system is the property query mechanism of std::execution. It allows querying properties of objects at both compile time and runtime.

Standard definition (exec.tex:116):

A queryable object is a read-only collection of key/value pair where each key is a customization point object known as a query object.

The query system is ubiquitous in std::execution:

graph TD
  A["sync_wait(sndr)"]
  A --> B["connect(sndr, rcvr)"]
  B --> C["transform_sender(sndr, get_env(rcvr))"]
  C --> D["get_domain(env) ← query domain"]
  C --> E["get_scheduler(env) ← query scheduler"]
  A --> F["start(op)"]
  F --> G["get_stop_token(get_env(rcvr)) ← query stop token"]
  G --> H["decide whether to set_stopped"]

Every core operation relies on the query system to adapt to the execution context.

Query Object

A query object is a CPO that accepts a queryable and returns a property:

cpp
// Built-in query objects
get_allocator                      // → allocator
get_stop_token                     // → stop token
get_scheduler                      // → current scheduler
get_domain                         // → domain (for algorithm customization)
get_forward_progress_guarantee     // → forward_progress_guarantee
get_completion_scheduler<Tag>      // → scheduler for a specific completion signal
get_completion_domain              // → domain for completion signals

Query objects are also CPOs, meaning they can be customized via tag_invoke.

How query Works —— Bidirectional Dispatch

There are two equivalent ways to invoke a query:

cpp
// Way 1: Query object calls queryable
auto sched = get_scheduler(env);

// Way 2: queryable calls query() member
auto sched = env.query(get_scheduler);

Implementation of get_scheduler_t in stdexec:

cpp
struct get_scheduler_t {
  // Main entry: try queryable::query(get_scheduler_t)
  template <class _Env>
    requires requires (_Env&& __env) {
      static_cast<_Env&&>(__env).query(*this);
    }
  auto operator()(_Env&& __env) const
    noexcept(noexcept(static_cast<_Env&&>(__env).query(*this)))
    -> decltype(static_cast<_Env&&>(__env).query(*this))
  {
    return static_cast<_Env&&>(__env).query(*this);
  }

  // Fallback: if queryable has no query member, try tag_invoke
  template <class _Env>
    requires (!requires (_Env&& __env) { __env.query(*this); })
          && tag_invocable<get_scheduler_t, _Env>
  auto operator()(_Env&& __env) const
    noexcept(nothrow_tag_invocable<get_scheduler_t, _Env>)
    -> tag_invoke_result_t<get_scheduler_t, _Env>
  {
    return tag_invoke(*this, static_cast<_Env&&>(__env));
  }
};

Dispatch chain:

graph TD
  A["get_scheduler(env)"]
  A --> B["env.query(get_scheduler) ?"]
  B --> C["Check query member"]
  C --> D["env handles it → return value"]
  A --> E["tag_invoke(get_scheduler, env) ?"]
  E --> F["ADL lookup"]
  F --> G["Custom handling → return value"]

prop uses the query member: prop<Query, Value>’s query(q) returns the value when q matches the key.

Receiver uses tag_invoke: a receiver’s get_env() returns a custom env, and the env implements query() or relies on ADL.

prop —— Single-Key Environment

prop<Query, Value> is a simple environment that responds to only one query:

cpp
template <class Query, class Value>
struct prop {
  Value value_;

  // Only responds to matching queries
  constexpr Value query(Query) const noexcept {
    return value_;
  }
};

Usage:

cpp
auto p = stdexec::prop{get_stop_token, never_stop_token{}};

get_stop_token(p);   // → never_stop_token{}
get_scheduler(p);    // ← Compile error! prop does not respond to get_scheduler

Key behavior of prop: only responds to the query key specified at construction, all other queries are ill-formed.

env —— Multi-Key Composite Environment

env<Envs...> merges multiple sub-envs into one:

cpp
template <queryable... Envs>
struct env : Envs... {
  // Inherits query() members from all sub-envs
  using Envs::query...;
};

Usage:

cpp
auto e = stdexec::env{
  stdexec::prop{get_scheduler, my_scheduler},
  stdexec::prop{get_stop_token, my_stop_token}
};

get_scheduler(e);    // → my_scheduler (found via prop::query)
get_stop_token(e);   // → my_stop_token

Query Lookup Rules (using Inheritance)

When env inherits multiple query functions via using Envs::query..., C++‘s overload resolution rules work automatically:

graph LR
  E["env&lt;prop&lt;A, int&gt;, prop&lt;B, string&gt;&gt;"]
  E --> QA["query(A)"]
  QA --> R1["int ← exact match first prop"]
  E --> QB["query(B)"]
  QB --> R2["string ← exact match second prop"]
  E --> QC["query(C)"]
  QC --> R3["Compile error ← no match"]

If two sub-envs both respond to the same query (env<prop<A,int>, prop<A,string>>), calling query(A) is ambiguous — compile error. This is by design: query keys in an environment must be unique.

Internal Implementation of prop and env

In stdexec, prop’s query implementation uses the __mconst metafunction:

cpp
template <class _Query, class _Value>
struct __prop {
  _Value __value_;

  template <class _Other>
    requires __same_as<_Query, __decay_t<_Other>>
  constexpr _Value query(_Other) const noexcept {
    return __value_;
  }
};

The __same_as constraint ensures only exactly matching queries are responded to.

forwarding_query —— Query Passthrough

Certain queries need to penetrate environment wrappers, allowing upper adapters to still access properties of the original environment:

cpp
forwarding_query(get_scheduler)     // true
forwarding_query(get_stop_token)    // true
forwarding_query(get_allocator)     // true
forwarding_query(get_domain)        // true

Semantics of forwarding_query

graph TD
  A["write_env(sndr, prop{get_scheduler, new_sch})"]
  A --> B["Child sender's environment"]
  B --> C["get_scheduler → new_sch ← overridden"]
  B --> D["get_stop_token → passthrough ← forwarding"]
  B --> E["get_domain → passthrough ← forwarding"]
  A --> F["Non-forwarded queries are inaccessible"]
  F --> G["Custom query → invisible"]

__hide_query in stdexec

stdexec implements passthrough via __hide_query:

cpp
template <class _Env, class... _Queries>
struct __hide_query {
  _Env __env_;

  // Passthrough all queries not in the _Queries list
  template <__none_of<_Queries...> _Query, class... _As>
    requires __queryable_with<_Env, _Query, _As...>
  auto query(_Query, _As&&... __as) const noexcept
    -> decltype(__env_.query(_Query{}, static_cast<_As&&>(__as)...))
  {
    return __env_.query(_Query{}, static_cast<_As&&>(__as)...);
  }
};
  • Hide list (_Queries...): queries overridden by write_env
  • __none_of: only allows queries not in the hide list to pass through
  • Other queries are blocked: return ill-formed

This is why explicitly overridden queries in write_env are replaced, while non-overridden standard queries pass through to the upper environment.

TRY-QUERY —— Query with Fallback

The standard uses TRY-QUERY for optional queries:

cpp
// TRY-QUERY(q, tag, args...):
// 1. Try q.query(tag, args...)
// 2. If not supported, try q.query(tag)
// 3. If neither is supported, expression is ill-formed

This pattern is crucial in scheduler affinity. For example, let_value needs to query whether the receiver has get_scheduler:

cpp
template <class _Receiver>
auto __query_scheduler(const _Receiver& __rcvr) noexcept {
  if constexpr (requires { get_scheduler(get_env(__rcvr)); }) {
    return std::optional{get_scheduler(get_env(__rcvr))};
  } else {
    return std::optional<__none_t>{};
  }
}

TRY-QUERY allows sender algorithms to optionally query environment properties: if the environment provides the query, use it; otherwise fall back to default behavior.

try_query Helper Function

stdexec’s receiver_proxy provides try_query:

cpp
template <class Value, class Query>
auto try_query(Query q) const noexcept -> std::optional<Value> {
  std::optional<Value> result;
  __query_env(__mtypeid<Query>, __mtypeid<Value>, &result);
  return result;
}

This is a type-erased version of TRY-QUERY: it forwards queries to the actual receiver environment via runtime type information.

Practical Applications of Queries

1. Getting Completion Scheduler from a Sender

cpp
sender auto s = schedule(pool.get_scheduler());

auto env = get_env(s);
auto sched = get_completion_scheduler<set_value_t>(env);
// sched == pool.get_scheduler()

get_completion_scheduler<set_value_t> returns on which scheduler the sender will invoke set_value. A sender’s environment typically exposes this query, allowing algorithms like continues_on to identify the completion context.

Different completion signals can have different schedulers:

cpp
// Some senders may complete set_value and set_error on different schedulers
auto val_sched = get_completion_scheduler<set_value_t>(env);
auto err_sched = get_completion_scheduler<set_error_t>(env);

2. Implementing Scheduler Affinity

scheduler affinity guarantees that a receiver’s completion functions execute on the correct execution context:

cpp
// starts_on implementation mechanism:
// 1. Schedule on the target scheduler
// 2. Execute the work
// 3. Query get_scheduler in the receiver's environment
// 4. If the receiver's scheduler is not the current one, continue switching to it

// continues_on implementation mechanism:
// 1. After completing work, query get_scheduler in the receiver's environment
// 2. If it exists, switch to that scheduler
// 3. Finally call set_value

continues_on(sndr, sch) makes the sender complete on sch:

cpp
schedule(pool_a)
  | then(work)                     // → executes work on pool_a
  | continues_on(pool_b)           // → switch to pool_b
  | then(result_handler);           // → executes result_handler on pool_b

3. Custom Queries

cpp
// Define a query
struct my_query_t {
  template <class Env>
  auto operator()(const Env& env) const {
    return env.query(*this);
  }
};
inline constexpr my_query_t my_query{};

// Provide it in an env
auto e = stdexec::prop{my_query, 42};
auto v = my_query(e);  // → 42

4. Querying Receiver Environment Inside a Sender

cpp
// Custom sender queries receiver environment in start()
template <typename R>
struct op {
  R rcvr_;

  void start() & noexcept {
    auto env = get_env(rcvr_);

    // Query stop token
    auto token = get_stop_token(env);
    if (token.stop_requested()) {
      set_stopped(std::move(rcvr_));
      return;
    }

    // Query scheduler (if present)
    if constexpr (requires { get_scheduler(env); }) {
      auto sch = get_scheduler(env);
      // Use scheduler to schedule subsequent work
    }

    // Continue execution...
  }
};

Key Takeaways

  1. Bidirectional dispatch: query(env) is equivalent to env.query(query), stdexec prefers env.query() first
  2. prop<Query, Value>: single-key environment, responds to only one exact-match query
  3. env<Envs...>: multi-key environment, merges multiple props via using inheritance, query keys must be unique
  4. forwarding_query: declares which queries can penetrate environment wrappers (write_env hides overridden queries, passes through non-overridden ones)
  5. TRY-QUERY: optional query pattern, sender algorithms decide behavior based on whether the receiver environment provides a query
  6. Scheduler affinity: the query system is the foundation of scheduler affinity; continues_on and starts_on both depend on the get_scheduler query

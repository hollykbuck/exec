25 · Query System — Reading Properties from the Environment

Why Do We Need a Query System?

In C++, dependency injection is traditionally done by passing template parameters around. If you want to pass an allocator to an algorithm, you either add a template parameter:

cpp
template <class Alloc>
void do_work(Alloc alloc) { /* ... */ }

Or use a global variable, or store it as a member. When an algorithm chain is 10 layers deep, every intermediate algorithm has to forward the allocator — even if it doesn’t use it.

P2300 takes a different approach — it stuffs the allocator, scheduler, stop_token, and everything else into an environment — a “context bundle” carried by every receiver. A sender that needs something just reaches in and grabs it. No more layer-by-layer forwarding.

What is the query system, simply put? It’s the mechanism for reaching in and asking. Want to know the current scheduler? get_scheduler(env). Want to know if cancellation has been requested? get_stop_token(env). Want to know the allocator? get_allocator(env).

Analogy: Environment as a Shipping Label

Think of every receiver as a package. Taped to the package is a label sheet that says:

Dispatch warehouse (scheduler): Thread Pool A Interceptable (stop_token): Yes Packaging (allocator): std::allocator

The sender doesn’t need to know who put the label there — it just reads the label. The query system is the set of conventions that lets the sender ask these questions.

Built-in Query Objects

The standard defines the following query objects — each one is like a “standard question”:

Query ObjectReturn TypeMeaning
get_allocatorallocatorMemory allocator of the current environment
get_stop_tokenstop_tokenToken that can signal cancellation
get_schedulerschedulerThe scheduler where the receiver lives
get_domaindomainDomain for algorithm customization
get_forward_progress_guaranteeenumForward progress guarantee level
get_completion_scheduler<Tag>schedulerScheduler for a specific completion signal

These query objects are themselves CPOs (Customization Point Objects) — they can be customized via tag_invoke.

Basic Usage: prop — Single-Key Environment

An environment can be composed from one or more props. Let’s start with the simplest — one that answers only one question:

cpp
// Create an environment that only answers "get_stop_token"
auto p = stdexec::prop{get_stop_token, never_stop_token{}};

// Ask it a question it can answer
get_stop_token(p);   // → never_stop_token{}

// Ask it a question it can't answer
get_scheduler(p);    // ← Compile error! prop only answers its declared key

Above, prop{Key, Value} creates a single-key environment. It responds only to queries matching the Key type. All other queries are compile errors. This design guarantees you won’t accidentally read a wrong value — if you ask a prop for a key it doesn’t have, you get a compile error, not a silent default.

prop Internals

cpp
template <class Query, class Value>
struct prop {
  Value value_;

  // Only responds to exact-match queries
  template <class Other>
    requires std::same_as<Query, std::decay_t<Other>>
  constexpr Value query(Other) const noexcept {
    return value_;
  }
};

The key line is requires std::same_as<Query, Other> — only an exact match on the query key gets a response. Everything else is eliminated at compile time.

env — Multi-Key Composite Environment

In real code, a receiver needs to answer multiple questions — the scheduler, the stop token, the allocator. env stitches multiple props together:

cpp
auto e = stdexec::env{
  stdexec::prop{get_scheduler, my_scheduler},
  stdexec::prop{get_stop_token, my_stop_token}
};

get_scheduler(e);    // → my_scheduler
get_stop_token(e);   // → my_stop_token

env uses multiple inheritance + a using declaration to merge the query() member functions from all sub-environments:

cpp
template <queryable... Envs>
struct env : Envs... {
  using Envs::query...;  // Bring every sub-env's query() into scope
};

C++ overload resolution automatically picks the matching query(). If a prop is like a single label on a package, env is the label booklet with all the labels stapled together.

Important Rule: Query Keys Must Be Unique

cpp
// Compile error! Two props both claim to answer get_scheduler
auto bad = stdexec::env{
  stdexec::prop{get_scheduler, sched_a},
  stdexec::prop{get_scheduler, sched_b}
};
// get_scheduler(bad) → ambiguous → compile failure

Duplicate keys are forbidden by design. If an environment has two contradictory answers, something is wrong — the compiler rejects it immediately, which is far better than a runtime bug.

Bidirectional Dispatch: Two Equivalent Calls

The query system supports two invocation styles, which are completely equivalent:

cpp
// Style 1: query object calls the queryable
auto sched = get_scheduler(env);

// Style 2: queryable responds to the query object
auto sched = env.query(get_scheduler);

stdexec internally prefers the .query() member function path (more efficient), falling back to ADL-based tag_invoke if no member exists. The full dispatch chain:

get_scheduler(env)
  → Does env have a query(get_scheduler) member?
    → Yes: call env.query(get_scheduler), return result
    → No: try tag_invoke(get_scheduler, env) via ADL
      → Found: call the custom tag_invoke overload
      → Not found: compile error

forwarding_query — Which Queries Penetrate Wrappers?

This is a more subtle concept. Let’s use a scenario to understand why it exists:

cpp
// write_env overrides get_scheduler downstream
write_env(sndr, prop{get_scheduler, new_sched})
// Question: can get_stop_token still pass through to downstream?

write_env overrides one property, but it shouldn’t block all others. So the standard introduces forwarding_query — a marker for which queries can “penetrate” environment wrappers:

forwarding_query(get_scheduler)     // true — can penetrate
forwarding_query(get_stop_token)    // true — can penetrate
forwarding_query(get_allocator)     // true — can penetrate
forwarding_query(get_domain)        // true — can penetrate
// Custom queries default → false — blocked

Internally, stdexec’s __hide_query maintains a “blocklist” — queries on the list are intercepted (because an upper layer overrode them), everything else passes through. This way write_env only replaces what it said it would replace, and other properties flow through unaffected.

TRY-QUERY — “Do You Have This Info?”

Not every query is required. Some are “nice to have.” For example, the let_value algorithm, before starting a child sender, asks the receiver’s environment: “Which scheduler are you bound to?” If there is one, it switches there; if not, it stays put. This “ask and see” pattern is called TRY-QUERY:

cpp
template <class Receiver>
auto try_query_scheduler(const Receiver& rcvr) noexcept {
  if constexpr (requires { get_scheduler(get_env(rcvr)); }) {
    return std::optional{get_scheduler(get_env(rcvr))};
  } else {
    return std::optional<int>{};  // Not available — that's fine
  }
}

Above, if constexpr checks at compile time whether the receiver’s environment provides get_scheduler. This pattern is used heavily in scheduler affinity internals — sender algorithms “ask” the receiver for its scheduler preference, respect it if present, and fall back if not.

Limitations

  1. prop’s exact-match semantics mean no flexibility — you can’t define one prop that answers a family of related queries with the same default. Each query key needs its own prop.
  2. forwarding_query defaults to false for custom queries — a custom query won’t automatically penetrate write_env wrappers unless you explicitly specialize forwarding_query.
  3. The query system is read-only — it only provides the ability to read environment information. To modify values in the environment, use write_env (Chapter 24).
  4. Compile-time exact matching means a wrong query key produces a compiler error, not a silent fallback — this is usually a good thing, but in generic code where you want “return if available, ignore if not” semantics, you need if constexpr + TRY-QUERY.

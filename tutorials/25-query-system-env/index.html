25 · Query System — Reading Properties from the Environment

Why Do We Need a Query System?

In C++, dependency injection is traditionally done by passing template parameters around. If you want to pass an allocator to an algorithm, you either add a template parameter:

cpp
template <class Alloc>
void do_work(Alloc alloc) { /* ... */ }

Or use a global variable, or store it as a member. When an algorithm chain is 10 layers deep, every intermediate algorithm has to forward the allocator — even if it doesn’t use it.

C++26 std::execution takes a different approach — it stuffs the allocator, scheduler, stop_token, and everything else into an environment — a “context bundle” carried by every receiver. A sender that needs something just reaches in and grabs it. No more layer-by-layer forwarding.

What is the query system, simply put? It’s the mechanism for reaching in and asking. Want to know the current scheduler? get_scheduler(env). Want to know if cancellation has been requested? get_stop_token(env). Want to know the allocator? get_allocator(env).

Analogy: Environment as a Shipping Label

Think of every receiver as a package. Taped to the package is a label sheet that says:

Dispatch warehouse (scheduler): Thread Pool A Interceptable (stop_token): Yes Packaging (allocator): std::allocator

The sender doesn’t need to know who put the label there — it just reads the label. The query system is the set of conventions that lets the sender ask these questions.

Overview: How Queries Work

At its core, the query system is a type-keyed dispatch mechanism. Every query is a unique type (a CPO like get_scheduler_t), and every environment answers queries through member function overload resolution.

The lifecycle of a query is:

  1. connect(sndr, rcvr) stores the receiver inside the operation state
  2. get_env(rcvr) extracts the receiver’s environment — a type-keyed bag of props
  3. A sender calls get_scheduler(env) (or env.query(get_scheduler) — equivalent)
  4. The environment dispatches to the matching prop or sub-environment via overload resolution or linear search
  5. The value is returned — the sender now has the scheduler, stop token, allocator, etc.

Each query is type-keyed — the compiler selects the right prop at compile time by matching the query type. No runtime dispatch, no string lookups, no virtual calls.

The standard defines the following query objects — each one is like a “standard question”:

These query objects are themselves CPOs (Customization Point Objects) — they can be customized via tag_invoke.

Basic Usage: prop — Single-Key Environment

An environment can be composed from one or more props. Let’s start with the simplest — one that answers only one question:

cpp
// Create an environment that only answers "get_stop_token"
auto p = stdexec::prop{get_stop_token, never_stop_token{}};

// Ask it a question it can answer
get_stop_token(p);   // → never_stop_token{}

// Ask it a question it can't answer
get_scheduler(p);    // ← Compile error! prop only answers its declared key

Above, prop{Key, Value} creates a single-key environment. It responds only to queries matching the Key type. All other queries are compile errors. This design guarantees you won’t accidentally read a wrong value — if you ask a prop for a key it doesn’t have, you get a compile error, not a silent default.

prop Internals

cpp
template <class Query, class Value>
struct prop {
  [[no_unique_address]] Query __query;
  [[no_unique_address]] Value __value;

  // Only responds to exact-match queries — _Query is the non-deduced
  // class template parameter, so this overload only matches when the
  // query type matches exactly.
  constexpr auto query(_Query, auto &&...) const noexcept
    -> _Value const & {
    return __value;
  }
};

prop only has a single query() overload where the first parameter type _Query is the non-deduced class template parameter. This means only an exact type match triggers a response — there is no template argument deduction that could accidentally broaden the match. All other queries are eliminated at compile time.

env — Multi-Key Composite Environment

In real code, a receiver needs to answer multiple questions — the scheduler, the stop token, the allocator. env stitches multiple props together:

cpp
auto e = stdexec::env{
  stdexec::prop{get_scheduler, my_scheduler},
  stdexec::prop{get_stop_token, my_stop_token}
};

get_scheduler(e);    // → my_scheduler
get_stop_token(e);   // → my_stop_token

env stores sub-environments in a tuple and dispatches queries via linear search — it scans the sub-environments in order and returns the first one that can answer the query:

cpp
// Simplified: actual stdexec uses __get_1st_env to search linearly
template <class... Envs>
struct env : __tuple<Envs...> {
  template <class Query, class... Args>
    requires /* at least one sub-env can answer Query */
  constexpr auto query(Query, Args &&...args) const {
    auto &sub_env = __get_1st_env<Query, Args...>()(*this);
    return __query<Query>()(sub_env, static_cast<Args &&>(args)...);
  }
};

The __get_1st_env helper (concept-based linear search) iterates through the sub-environments at compile time and returns a reference to the first one that satisfies __queryable_with<Env, Query, Args...>:

This design avoids diamond-inheritance issues and allows mixing prop types with other queryable objects (like __sched_attrs or non_dependent_attrs) in the same env.

Important: Duplicate Keys Resolve by Position

cpp
auto e = stdexec::env{
  stdexec::prop{get_scheduler, sched_a},
  stdexec::prop{get_scheduler, sched_b}
};
get_scheduler(e);  // → sched_a — the FIRST match wins

Unlike the multiple-inheritance approach (which would produce an ambiguity error), the tuple-based linear search silently picks the first sub-environment that answers the query. This is intentional: it gives JOIN-ENV-style composition predictable priority semantics and avoids spurious compilation failures when unrelated sub-environments happen to support overlapping queries.

Bidirectional Dispatch: Two Equivalent Calls

The query system supports two invocation styles, which are completely equivalent:

cpp
// Style 1: query object calls the queryable
auto sched = get_scheduler(env);

// Style 2: queryable responds to the query object
auto sched = env.query(get_scheduler);

stdexec internally prefers the .query() member function path, with a deprecated tag_invoke fallback:

flowchart TD

Note: The tag_invoke path for queries is [[deprecated]] as of P2300R10. New code should only implement query(MyQuery, args...) as a member function of the environment, not as a free tag_invoke overload.

forwarding_query — Which Queries Penetrate Wrappers?

This is a more subtle concept. Let’s use a scenario to understand why it exists:

cpp
// write_env overrides get_scheduler downstream
write_env(sndr, prop{get_scheduler, new_sched})
// Question: can get_stop_token still pass through to downstream?

write_env overrides one property, but it shouldn’t block all others. So the standard introduces forwarding_query — a marker for which queries can “penetrate” environment wrappers.

The dispatch for determining forwarding status:

  1. If q.query(forwarding_query_t) is well-formed, use its result (a bool constant expression);
  2. Otherwise, true if q derives from forwarding_query_t;
  3. Otherwise, false.

This means a custom query can opt in to forwarding simply by inheriting from forwarding_query_t:

cpp
struct my_forwarding_query_t
  : stdexec::__query<my_forwarding_query_t>
  , stdexec::forwarding_query_t        // ← opt in
{
  using stdexec::__query<my_forwarding_query_t>::operator();
};

Which Queries Forward?

QueryForwards?Mechanism
get_schedulerYesExplicit query(forwarding_query_t) -> true
get_stop_tokenYesInherits default + forwarding_query_t tag
get_allocatorYesExplicit query(forwarding_query_t) -> true
get_domainYesExplicit query(forwarding_query_t) -> true
get_delegation_schedulerYesExplicit query(forwarding_query_t) -> true
get_completion_scheduler<Tag>YesExplicit query(forwarding_query_t) -> true
get_forward_progress_guaranteeNoNo override, no derivation
__execute_may_block_callerNoNo override, no derivation

The rule: runtime-context queries (scheduler, stop token, allocator) forward; static-property queries (forward progress, may-block-caller) do not. get_completion_scheduler is an exception — though it feels like a static property, it forwards because the completion scheduler may depend on outer environment context.

Internally, __fwd (FWD-ENV) wraps an environment and constrains its query() template with __forwarding_query:

cpp
template <__forwarding_query _Query, class... _Args>
  requires __queryable_with<_Env, _Query, _Args...>
constexpr auto query(_Query, _Args &&...__args) const
  -> __query_result_t<_Env, _Query, _Args...>
{
  return __query<_Query>()(__env_, static_cast<_Args &&>(__args)...);
}

Only queries that satisfy __forwarding_query can pass through FWD-ENV. Non-forwarding queries are blocked at compile time.

TRY-QUERY — “Do You Have This Info?”

Not every query is required. Some are “nice to have.” For example, the let_value algorithm, before starting a child sender, asks the receiver’s environment: “Which scheduler are you bound to?” If there is one, it switches there; if not, it stays put. This “ask and see” pattern is called TRY-QUERY:

cpp
template <class Receiver>
auto try_query_scheduler(const Receiver& rcvr) noexcept {
  if constexpr (requires { get_scheduler(get_env(rcvr)); }) {
    return std::optional{get_scheduler(get_env(rcvr))};
  } else {
    return std::optional<int>{};  // Not available — that's fine
  }
}

Above, if constexpr checks at compile time whether the receiver’s environment provides get_scheduler. This pattern is used heavily in scheduler affinity internals — sender algorithms “ask” the receiver for its scheduler preference, respect it if present, and fall back if not.

Deeper Dive: Query Implementation Mechanics

The __query CRTP Base: Defaults and Transforms

Under the hood, every query CPO inherits from STDEXEC::__query<Query, Default, Transform>:

cpp
template <class _Query, auto _Default = __no_default, class _Transform = __q1<__midentity>>
struct __query : __query<_Query, __no_default, _Transform> {
  // Calls _Default when no environment answers
  constexpr auto operator()(__ignore, _Args &&...) const noexcept
    -> __mcall1<_Transform, __mtypeof<_Default>> {
    return _Default;
  }
};

The three template parameters are:

ParameterMeaningExample
_QueryThe CPO type itselfget_allocator_t
_DefaultFallback value when no environment answersforward_progress_guarantee::weakly_parallel for get_forward_progress_guarantee, true for __execute_may_block_caller
_TransformType transformation applied to the result__q1<__decay_t> for get_forward_progress_guarantee (decays return type)

For queries with the default (the primary template), the operator()(__ignore, ...) overload fires when the environment does not have the query — it silently returns _Default instead of producing a compile error. get_forward_progress_guarantee_t uses this to default to weakly_parallel:

cpp
struct get_forward_progress_guarantee_t
  : __query<get_forward_progress_guarantee_t,
            forward_progress_guarantee::weakly_parallel,
            __q1<__decay_t>>
{ ... };

cprop — Compile-Time Constant Queries

For queries whose answer is known at compile time, use cprop<Query, Value>:

cpp
auto env = stdexec::env{
  stdexec::cprop<get_domain_t, my_domain>{}
};

cprop provides a static constexpr query() with no data members — zero overhead for compile-time known values. It’s used internally by __mk_secondary_env to inject domain information without runtime storage.

__root_env — The Bottom of the Stack

Every environment chain eventually bottoms out at __root_env, which answers exactly one query:

cpp
struct __root_env {
  static constexpr auto query(__root_t) noexcept -> bool { return true; }
};

The __as_root_env wrapper joins an environment with __root_env, guaranteeing there is always a base that can answer __root_t. This is used by let_value/finally to anchor the secondary environment chain.

get_domain — Three-Tier Dispatch

get_domain has a richer resolution order than a simple query:

  1. Member function: If the environment has a query(get_domain_t) member, use it.
  2. Start scheduler derivation: Otherwise, if the environment provides get_start_scheduler, derive the domain from the completion scheduler of that scheduler’s set_value_t completion.
  3. Default: Otherwise, return default_domain (the no-op domain).
cpp
// Simplified logic
auto operator()(_Env const &env) const noexcept -> auto {
  if constexpr (has_query_member<get_domain_t>) {
    return /* env's explicit domain */;
  } else if constexpr (has_start_scheduler) {
    auto sched = get_start_scheduler(env);
    auto cmpl_sched = get_completion_scheduler<set_value_t>(sched, hide_scheduler{env});
    return /* domain of cmpl_sched */;
  } else {
    return default_domain{};
  }
}

This means that even without an explicit get_domain in the environment, the domain is automatically derived from whichever scheduler is running the pipeline — the scheduler’s own get_domain query flows up through the completion scheduler chain.

Extension: get_scheduler Falls Back to get_start_scheduler

stdexec includes a NOT-TO-SPEC extension: if the environment does not have get_scheduler but does have get_start_scheduler, then get_scheduler returns the start scheduler:

cpp
template <class _Env>
  requires(!__callable<__query<get_scheduler_t>, _Env const &>)
       && __callable<get_start_scheduler_t, _Env const &>
constexpr auto operator()(_Env const &__env) const noexcept {
  return get_start_scheduler(__env);
}

This is a quality-of-life extension — it means you can always call get_scheduler(env) in generic code and get a reasonable answer as long as a start scheduler is present.

__hide_scheduler — Preventing Recursive Queries

When one query internally needs to call another query on the same environment (e.g., get_domain calling get_start_scheduler and then get_completion_scheduler on the result), the system uses __hide_scheduler to avoid infinite recursion. It wraps the scheduler and blocks get_scheduler / get_start_scheduler queries on the wrapped type, ensuring the domain derivation logic terminates.

Notes

  1. prop’s exact-match semantics mean no flexibility — you can’t define one prop that answers a family of related queries with the same default. Each query key needs its own prop.
  2. forwarding_query defaults to false for custom queries unless they derive from forwarding_query_t — a custom query won’t automatically penetrate write_env wrappers unless you either override query(forwarding_query_t) or inherit from forwarding_query_t.
  3. The query system is read-only — it only provides the ability to read environment information. To modify values in the environment, use write_env (Chapter 24).
  4. Compile-time exact matching means a wrong query key produces a compiler error, not a silent fallback — this is usually a good thing, but in generic code where you want “return if available, ignore if not” semantics, you need if constexpr + TRY-QUERY. (Queries with a _Default parameter in __query are the exception — they silently return the default.)

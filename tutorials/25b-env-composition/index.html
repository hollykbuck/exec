25b · Environment Composition and Query Forwarding

On this page

Beyond Single-Key Environments

Chapter 25 introduced prop for single-key environments. But real environments are composites — a receiver’s environment combines queries from its parent, the scheduler, and any write_env injectors.

If you are familiar with go context, environments in C++ std::execution serve similiar purposes.

Three exposition utilities govern how environments compose and how queries propagate through them.

forwarding_query — Which Queries Cross Boundaries

When a sender wraps another sender (e.g., then wraps its child), the environment seen by the child is not the same as the outer environment. Some queries pass through; others are blocked.

forwarding_query determines which queries are forwarded through queryable adaptors:

cpp
template <class T>
concept forwarding-query = forwarding_query(T{});

For a query object q:

  1. If q.query(forwarding_query) is well-formed, use its result (a bool constant expression)
  2. Otherwise, true if q derives from forwarding_query_t
  3. Otherwise, false

Which Queries Forward?

QueryForwards?Reason
get_schedulerYesforwarding_query(get_scheduler) is true
get_stop_tokenYesStop token must propagate through wrappers
get_allocatorYesAllocator must propagate for consistent allocation
get_delegation_schedulerYesExplicitly specified as forwarding
get_domainYesDomain must propagate for algorithm customization
get_completion_scheduler<Tag>YesCompletion scheduler depends on outer env context
get_forward_progress_guaranteeNoEach scheduler provides its own guarantee

The rule: runtime-context queries (scheduler, stop token, allocator) forward; static-property queries (forward progress, may-block-caller) generally do not. get_completion_scheduler is a notable exception — though it feels like a static property, it forwards because the completion scheduler can depend on outer environment context, and the recursive scheduler-chain walk in the implementation needs to see through environment layers.

The filter is enforced at the constraint level — __fwd::query() is templated with __forwarding_query _Query, so a non-forwarding query fails template argument deduction before any runtime code is reached.

FWD-ENV — The Forwarding Filter

FWD-ENV(env) wraps a queryable object and blocks non-forwarding queries:

cpp
// Given a receiver's environment:
auto env = get_env(rcvr);

// FWD-ENV(env) only answers queries where forwarding_query(q) is true
auto fwd = FWD-ENV(env);

get_scheduler(fwd);     // OK — forwards
get_stop_token(fwd);    // OK — forwards
get_completion_scheduler<set_value_t>(fwd);  // OK — forwards!
get_forward_progress_guarantee(fwd);         // Ill-formed — blocked!

Internally, every adaptor’s basic-receiver calls FWD-ENV(get_env(rcvr)) when the child sender queries the environment. This blocks non-forwarding queries (like get_forward_progress_guarantee) from propagating beyond their owning layer, while letting runtime-context queries (scheduler, stop token, allocator) pass through.

MAKE-ENV — Single-Key Environment Factory

MAKE-ENV(q, v) creates a queryable object that answers exactly one query:

cpp
// Create an environment that answers get_scheduler with my_sched
auto env = MAKE-ENV(get_scheduler, my_sched);

get_scheduler(env);   // → my_sched
get_stop_token(env);  // Ill-formed — not answered

MAKE-ENV is the exposition equivalent of prop{q, v} from Chapter 25.

JOIN-ENV — Composing Environments

JOIN-ENV(env1, env2) merges two environments. The first takes priority:

cpp
auto base = MAKE-ENV(get_scheduler, io_sched);
auto override = MAKE-ENV(get_scheduler, cpu_sched);

auto combined = JOIN-ENV(override, base);

get_scheduler(combined);  // → cpu_sched  (override wins)

The resolution order:

  1. Try env1.query(q, as...) — if well-formed, use it
  2. Otherwise, try env2.query(q, as...) — if well-formed, use it
  3. Otherwise, ill-formed

Where JOIN-ENV Is Used

write_env uses JOIN-ENV to merge the injected environment with the receiver’s environment:

cpp
// write_env(sndr, env) internally:
impls-for<write_env-t>::join-env(state, FWD-ENV(get_env(rcvr)))
// = JOIN-ENV(state, FWD-ENV(get_env(rcvr)))
// state (the injected env) takes priority over the receiver's env

starts_on uses JOIN-ENV to inject the target scheduler:

cpp
// starts_on(sch, sndr) ensures the child sees sch as its scheduler
JOIN-ENV(MAKE-ENV(get_scheduler, sch), FWD-ENV(get_env(rcvr)))

Secondary Environments — let_value, finally, sequence

When one sender’s completion triggers the start of another sender (e.g., let_value runs a child sender when set_value arrives), the child needs an environment that reflects the completing sender’s properties, not the outer receiver’s. This is called a secondary environment (__mk_secondary_env).

__mk_secondary_env<SetTags...> builds the child’s environment from:

  1. Completion scheduler: If the completing sender has a completion scheduler for the triggering tag, that scheduler becomes the child’s start scheduler (wrapped in __sched_env which provides get_start_scheduler and optionally get_domain).
  2. Completion domain: If no single completion scheduler can be determined but a completion domain is available, it injects the domain as a cprop.
  3. Empty fallback: If neither is available, the child gets an empty env<>.
cpp
// Simplified: let_value(set_value_t) builds:
auto child_env = __mk_secondary_env<set_value_t>(
  __cv{},
  parent_sender,
  FWD-ENV(get_env(outer_receiver))
);

// If parent_sender completes on a thread_pool scheduler:
get_start_scheduler(child_env)  // → the thread_pool

This is what makes let_value and finally “stay” on the completing scheduler — the child sender automatically inherits the completion environment of its parent.

__sched_env — Scheduler-Backed Environment

__sched_env<Scheduler> wraps a scheduler into an environment that answers:

  • get_start_scheduler → the scheduler itself
  • get_domain → the scheduler’s domain (derived via __scheduler_domain_t)

This is used not only by __mk_secondary_env but also by the starts_on adaptor to inject the target scheduler into the child’s environment.

Sender Adaptor Environment Chain

Every sender adaptor follows the same pattern when constructing the child sender’s environment:

  1. Extract the outer receiver’s environment via get_env(rcvr)
  2. Filter through FWD-ENV to block non-forwarding queries
  3. Merge with algorithm-specific state via JOIN-ENV (if applicable)

Every sender adaptor in the standard uses this pattern: wrap the receiver’s environment in FWD-ENV, then optionally JOIN-ENV with algorithm-specific state. This ensures queries propagate correctly without leaking implementation details across layers.

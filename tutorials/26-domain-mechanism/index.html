26 · Domain — Letting Schedulers Take Over Algorithm Implementation

Same Algorithm, Different Scheduler, Different Best Version

Different schedulers have different optimal implementations for the same algorithm. A GPU scheduler’s bulk can compile down to a CUDA kernel executing on thousands of cores; a CPU scheduler’s bulk becomes a std::thread parallel loop. The catch: user code looks the same:

cpp
bulk(sndr, N, f)

Who decides how this line actually executes? The Domain is that decision-maker — it gives a scheduler the chance to replace a generic algorithm with a backend-optimized version at compile time.

More concretely:

cpp
// GPU scheduler wants when_all / bulk to use GPU-optimized versions
nvexec::stream_scheduler gpu_sch{};

starts_on(gpu_sch,
  when_all(gpu_work1(), gpu_work2())
);
// Without domain → uses the default CPU version of when_all
// With domain   → GPU backend injects its own when_all

Domain allows replacing algorithm implementations without changing user code. This is the core of C++26 std::execution’s extension mechanism.

Two-Phase Processing: apply_sender → transform_sender

If you think of sender pipeline construction as a compiler, domain participates in two key phases:

Phase 1: apply_sender — Parsing

When the user writes sndr | then(f), the pipe operator first gets the current domain, then calls domain.apply_sender(then_t{}, sndr, f). This step produces an unresolved expression tree — still an abstract description at this point.

graph TD
  A["sndr | then(f)"] --> B["get_domain(get_env(sndr)) — fetch domain"]
  B --> C["domain.apply_sender(then_t{}, sndr, f)"]
  C --> D["Returns unresolved sender expression tree"]

Phase 2: transform_sender — Code Generation

When connect(sndr, rcvr) is called, domain gets a second chance — this time via transform_sender. Here, a domain can replace the entire expression tree:

graph TD
  A["connect(sndr, rcvr)"] --> B["transform_sender(sndr, get_env(rcvr))"]
  B --> C["get_domain(env) — fetch current domain"]
  C --> D["domain.transform_sender(tag, sndr, env)"]
  D --> E["default_domain: do nothing, return as-is"]
  D --> F["Custom domain: replace with optimized impl"]

If you think of it as a compiler:

  • apply_sender is the parser, producing an AST
  • transform_sender is the code generator, translating the AST into target-specific instructions

default_domain — The Laziest Translator

default_domain does nothing. It has two transform_sender overloads:

cpp
struct default_domain {
  // Priority 1: if the sender's own tag provides transform_sender, delegate
  template <class OpTag, class Sender, class Env>
    requires __has_transform_sender<tag_of_t<Sender>, OpTag, Sender, Env>
  constexpr auto transform_sender(OpTag, Sender&& sndr, Env const& env) const {
    return tag_of_t<Sender>().transform_sender(OpTag{}, sndr, env);
  }

  // Fallback: do nothing, return as-is
  template <class OpTag, class Sender, class Env>
  constexpr auto transform_sender(OpTag, Sender&& sndr, Env const&) const {
    return static_cast<Sender>(sndr);
  }
};

default_domain itself performs no optimization — it just passes the sender along. When you don’t need customization, all domain machinery is optimized away at compile time: zero runtime overhead.

indeterminate_domain — When Two Schedulers Disagree

This is the most important design decision in the domain mechanism. Consider this scenario:

cpp
starts_on(cpu_sch,
  continues_on(gpu_sch,
    bulk(work)
  )
);

Two different schedulers are involved — a CPU thread pool and a GPU stream. The bulk in the middle: should it use the CPU version or the GPU version? starts_on introduced one domain (cpu_domain), continues_on introduced another (gpu_domain). Two domains are fighting over who gets to handle bulk.

indeterminate_domain’s ruling principle is simple:

Either everyone agrees (behavior is consistent), or it’s a compile error — no guessing, no “it sort of works” false confidence.

cpp
template <class... Domains>
struct indeterminate_domain {
  static constexpr auto transform_sender(OpTag, Sndr&& sndr, Env const& env) {
    // Require all domains to behave the same as default_domain
    static_assert((__default_domain_like<Domains, OpTag, Sndr, Env> && ...),
      "ERROR: indeterminate domains: cannot pick an algorithm customization");
    // If consistent, use default_domain's "do nothing"
    return default_domain().transform_sender(OpTag{}, sndr, env);
  }
};

In other words: if every domain says “I won’t replace this algorithm,” the default is used. If one domain says “I want to replace it” and another disagrees, the compiler rejects the code outright — forcing you to resolve the architectural conflict at compile time, not discover inconsistent behavior at runtime.

The Shortest Custom Domain Example

A domain that does exactly one thing — print logs before and after then:

cpp
struct log_domain : stdexec::default_domain {
  template <stdexec::sender Sndr, class F>
  static auto transform_sender(stdexec::then_t, Sndr&& sndr, F&& f) {
    return std::move(sndr) | stdexec::then([f = std::forward<F>(f)](auto... args) {
      std::printf("[log] then begin\n");
      auto result = f(std::move(args)...);
      std::printf("[log] then end\n");
      return result;
    });
  }
};

Above, we only override then. All other algorithms (when_all, bulk, let_value, etc.) lack a corresponding transform_sender overload, so they fall through to default_domain’s “do nothing” path (inherited).

To activate this domain, just have your scheduler return it from the get_domain query:

cpp
struct my_scheduler {
  friend log_domain tag_invoke(
    stdexec::tag_t<stdexec::get_domain>,
    const my_scheduler&) noexcept {
    return {};
  }
};

Domain + write_env

write_env can inject or override a domain mid-pipeline:

cpp
sync_wait(
  write_env(
    just(40) | then([](int x) { return x + 2; }),
    prop{get_scheduler, my_scheduler}  // also carries my_scheduler's bound domain
  )
);

The scheduler injected by write_env carries its associated domain, affecting which implementation downstream algorithms choose during transform_sender.

Limitations

  1. Conflict = compile error — indeterminate_domain won’t try to “pick the right one.” It demands behavior consistency. Multi-backend hybrid pipelines require you to plan domain boundaries at design time, not hope for runtime auto-adaptation.
  2. Domain only affects sender algorithms (then / bulk / when_all etc.), not CPOs (schedule / connect / start). To customize a scheduler, you still need tag_invoke.
  3. Custom domain’s transform_sender resolves at compile time — you can’t do runtime conditionals to choose which implementation. If that flexibility is required, you may need type-erased senders.

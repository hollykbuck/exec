26 · Domain Mechanism

Standard Section: [exec.domain] stdexec Reference: include/stdexec/__detail/__domain.hpp Related Tests: tests/test_domain.cpp

Overview

Domain is the extension point system of std::execution. It allows a scheduler or environment to customize the behavior of sender algorithms at composition time — injecting optimized implementations for specific backends.

cpp
namespace std::execution {
  struct default_domain;                      // Default: no transformation
  template<class... Domains>
  struct indeterminate_domain;                // Multi-domain conflict resolution

  // Core functions
  template<sender Sndr, queryable Env>
    constexpr decltype(auto) transform_sender(Sndr&& sndr, const Env& env);

  template<class Domain, class Tag, sender Sndr, class... Args>
    constexpr decltype(auto) apply_sender(Domain dom, Tag, Sndr&& sndr, Args&&... args);
}

Why Does Domain Exist?

cpp
// GPU scheduler wants when_all / bulk to use GPU-optimized versions
nvexec::stream_scheduler gpu_sch{};

starts_on(gpu_sch,
  when_all(gpu_work1(), gpu_work2())
);
// Without domain → uses default CPU version of when_all
// With domain → GPU backend can inject its own when_all

Domain allows a scheduler to replace sender algorithm implementations without changing user code.

Call Chain Trace: From operator| to connect

When a user writes sndr | then(f), two phases happen behind the scenes:

Phase 1: apply_sender — Create Unresolved Sender

graph TD
  A["sndr | then(f)"] --> B["then_t::apply_sender(domain, sndr, f)"]
  B --> C["get_domain(get_env(previous_sender))<br/>Get domain from scheduler or environment"]
  B --> D["domain.apply_sender(then_t{}, sndr, f)"]
  D --> E["Returns __sexpr<br/>(unresolved sender expression tree)"]

apply_sender is called during pipe operations, before connect:

cpp
// Equivalent behavior of then_t::apply_sender in stdexec:
template <class Domain, sender Sndr, class F>
auto apply_sender(Domain dom, then_t, Sndr&& sndr, F&& f) {
  // Create sender expression tree: _then_ (Sndr, F)
  return __make_sexpr<then_t>(__data{std::forward<F>(f)},
                              __children{std::forward<Sndr>(sndr)});
}

The domain parameter of apply_sender comes from the previous sender’s environment:

graph LR
  A["schedule(pool)"] --> B["env: {get_domain: default_domain{}}"]
  C["sndr"] --> D["get_env(sndr)"]
  D --> E["get_domain(env)"]
  E --> F["Current domain"]
  B -.->|Each sender exposes its scheduler's domain in get_env| D

  A["connect(sndr, rcvr)"] --> B["transform_sender(sndr, get_env(rcvr))"]

default_domain Detailed

cpp
struct default_domain {
  // Priority: delegate to sender's tag (if the sender has a transform_sender overload)
  template <class _OpTag, class _Sender, class _Env>
    requires __has_transform_sender<tag_of_t<_Sender>, _OpTag, _Sender, _Env>
  constexpr auto transform_sender(_OpTag, _Sender&& __sndr, _Env const& __env) const
    -> decltype(auto)
  {
    return tag_of_t<_Sender>().transform_sender(
      _OpTag(), static_cast<_Sender&&>(__sndr), __env);
  }

  // Fallback: no transformation, return as-is
  template <class _OpTag, class _Sender, class _Env>
  constexpr auto transform_sender(_OpTag, _Sender&& __sndr, _Env const&) const
    -> _Sender
  {
    return static_cast<_Sender>(static_cast<_Sender&&>(__sndr));
  }

  // apply_sender: delegate to algorithm tag
  template <class _Tag, class... _Args>
    requires __has_apply_sender<_Tag, _Args...>
  constexpr auto apply_sender(_Tag, _Args&&... __args) const
    -> decltype(auto)
  {
    return _Tag().apply_sender(static_cast<_Args&&>(__args)...);
  }
};

default_domain has two transform_sender overloads:

  1. Delegation overload (priority): if the sender’s tag type has its own transform_sender (e.g., write_env sender needs to handle write_env’s domain propagation), delegate to it
  2. Identity overload (fallback): do nothing, return the original sender

This means default_domain itself does not do any optimization replacement — it just passes the sender to the next step.

indeterminate_domain: Multi-Domain Conflict Resolution

When multiple schedulers are mixed, domains may be inconsistent:

cpp
starts_on(sch_a,
  continues_on(sch_b,
    work()
  )
);
// sch_a's domain and sch_b's domain may differ

Rules for indeterminate_domain:

cpp
template <class... _Domains>
struct indeterminate_domain {
  // Requires all domains to behave identically on transform_sender
  // Otherwise triggers static_assert
  static constexpr auto transform_sender(_OpTag, _Sndr&& __sndr, _Env const& __env) {
    static_assert((__default_domain_like<_Domains, _OpTag, _Sndr, _Env> && ...),
      "ERROR: indeterminate domains: cannot pick an algorithm customization");
    return default_domain().transform_sender(
      _OpTag{}, static_cast<_Sndr&&>(__sndr), __env);
  }
};

__default_domain_like check: each _Domain’s transform_sender result must be consistent with default_domain. If inconsistent (i.e., one domain wants to replace an algorithm but another domain disagrees), compile error.

stdexec’s Domain Composition Algorithm

cpp
template <class... _Domains>
constexpr auto __common_domain_fn() noexcept {
  // Strategy 1: if common_type exists, use it
  if constexpr (__minvocable_q<std::common_type_t, _Domains...>)
    return std::common_type_t<_Domains...>{};

  // Strategy 2: if each domain is convertible to default_domain, use it
  else if constexpr (__minvocable_q<std::common_type_t, default_domain, _Domains...>)
    return std::common_type_t<default_domain, _Domains...>{};

  // Strategy 3: create indeterminate_domain (requires consistent behavior)
  else
    return __make_domain_t<_Domains...>{};
}

Complete Example: Custom Domain

The following implements a tracking_domain: it injects logging around then operations while keeping when_all unchanged.

Define Domain

cpp
struct tracking_domain : stdexec::default_domain {
  // Intercept then algorithm, inject logging
  template <stdexec::sender Sndr, std::invocable<std::decay_t<decltype(*std::declval<Sndr>())>> F>
  static auto transform_sender(
    stdexec::then_t, Sndr&& sndr, F&& f)
  {
    // Return a wrapper sender that prints logs around f
    return std::move(sndr) | stdexec::then([f = std::forward<F>(f)](auto... args) {
      std::printf("[tracking] then start\n");
      if constexpr (std::is_void_v<decltype(f(std::move(args)...))>) {
        f(std::move(args)...);
        std::printf("[tracking] then end\n");
      } else {
        auto result = f(std::move(args)...);
        std::printf("[tracking] then end\n");
        return result;
      }
    });
  }

  // Other algorithms (e.g., when_all) keep default behavior
  // Because it inherits from default_domain, uncovered algorithms take the identity path
};

Define Scheduler

cpp
struct tracking_scheduler {
  auto schedule() const noexcept {
    return stdexec::schedule(stdexec::run_loop::scheduler{});
  }

  bool operator==(const tracking_scheduler&) const = default;

  // Key: associate the domain with the scheduler
  friend tracking_domain tag_invoke(
    stdexec::tag_t<stdexec::get_domain>,
    const tracking_scheduler&) noexcept
  {
    return {};
  }
};

Test

tracking_scheduler sch{};

stdexec::sync_wait(

      return x + 2;
    })
  )
);
// Output:
//   [tracking] then start
//   [tracking] then end
// Result: 42

Domain and write_env

The write_env adaptor injects or overrides environment properties in a pipeline, including domain:

cpp
// Corresponding test: "domain propagation via write_env pipeline"
auto [v] = stdexec::sync_wait(
  stdexec::write_env(
    stdexec::just() | stdexec::then([] { return 42; }),
    stdexec::prop{stdexec::get_scheduler, my_scheduler}
  ) | stdexec::then([](int x) { return x; })
).value();
// write_env injects get_scheduler into child sender's environment
// domain follows the scheduler

Internally, write_env returns a merged environment from the __write_env sender’s get_env(), containing the get_scheduler query, which carries the domain.

Key Takeaways

  1. Two-phase processing: apply_sender creates an unresolved expression tree at pipe time, transform_sender resolves/replaces at connect time
  2. Domain source: obtained from get_domain / get_completion_domain queries on the scheduler and receiver environment
  3. default_domain: identity transformation — all senders pass through unchanged
  4. indeterminate_domain: requires all domains to behave consistently in case of conflicts, otherwise compile error
  5. Custom Domain: inherit default_domain, override transform_sender(tag, ...) to inject optimizations or interceptions for specific algorithms
  6. Zero overhead: when using default_domain, all domain code is optimized away at compile time

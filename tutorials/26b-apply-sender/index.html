26b · apply_sender — The Domain Hook Consumers Use

Standard Section: [exec.snd.apply]
Draft Reference: exec.tex lines 2870–2994

The Other Half

Chapter 26 explained transform_sender — the domain hook called by sender algorithms. But there is a second hook: apply_sender, called by consumers.

The standard defines it as:

cpp
template <class Domain, class Tag, sender Sndr, class... Args>
constexpr decltype(auto) apply_sender(Domain dom, Tag, Sndr&& sndr, Args&&... args);

It delegates to:

  1. dom.apply_sender(Tag(), sndr, args...) if that expression is well-formed
  2. Otherwise, default_domain().apply_sender(Tag(), sndr, args...)

Where apply_sender is Called

Consumers call apply_sender to give the domain a final chance to transform the sender before connecting:

cpp
// sync_wait (simplified):
template <sender Sndr>
auto sync_wait(Sndr&& sndr) {
  // ... get domain from environment ...
  auto transformed = apply_sender(dom, sync_wait_t{}, std::forward<Sndr>(sndr));
  // ... connect and run ...
}

// spawn (simplified):
template <sender Sndr>
void spawn(Sndr&& sndr) {
  // ... get domain ...
  auto transformed = apply_sender(dom, spawn_t{}, std::forward<Sndr>(sndr));
  // ... connect and start ...
}

This gives the domain a chance to wrap or replace the sender after all algorithms have been applied but before the final connect.

Concrete Example

Suppose you want a custom domain that wraps every sender consumed by sync_wait with a timeout:

cpp
struct timeout_domain : stdexec::default_domain {
  template <stdexec::sender Sndr, class... Args>
  static auto apply_sender(stdexec::sync_wait_t, Sndr&& sndr, Args&&...) {
    return with_timeout(std::forward<Sndr>(sndr), 5s);
  }
};

auto result = sync_wait(
  write_env(
    my_work(),
    stdexec::prop{stdexec::get_domain, timeout_domain{}}
  )
);
// apply_sender wraps my_work() with a 5-second timeout

Every call to sync_wait with this domain in the environment automatically applies the timeout.

Why Two Hooks?

The two-phase design separates concerns:

  • transform_sender — per-algorithm customization (replace bulk with GPU kernel)
  • apply_sender — per-consumer customization (wrap in timeout, add logging, inject allocator)

A domain that only needs to swap algorithm implementations never touches apply_sender. A domain that needs to intercept consumption never touches transform_sender.

Default Domain Behavior

default_domain’s apply_sender simply returns the sender unchanged. The indirection through apply_sender compiles to nothing — zero runtime overhead.

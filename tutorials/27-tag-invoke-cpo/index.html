On this page

The C++ Extension Problem

Suppose you’ve written an async library with this core interface:

cpp
// Your library defines a CPO (Customization Point Object)
inline constexpr schedule_t schedule{};

You want users to write schedule(my_scheduler) and have it dispatch to user-defined logic. Traditional C++ offers three extension strategies, each with drawbacks:

(1) Virtual functions — users inherit scheduler_base and override schedule(). Problem: every schedule() call goes through a vtable (runtime overhead). Worse, virtual functions can’t use noexcept, which is a fatal limitation in GPU (CUDA) environments.

(2) Template specialization — users specialize schedule_impl<MyScheduler>. Problem: specialization is invasive — you need an explicitly specializable template class. And if multiple libraries try to specialize for the same type, ODR conflicts arise.

(3) Global function overloading — users define a schedule(const MyScheduler&) overload. Problem: only overloads declared in the same namespace are found via ADL; cross-namespace overrides produce surprising behavior.

tag_invoke provides a fourth path — combining the strengths of all three while eliminating their weaknesses:

cpp
// User defines this in their own namespace
namespace my_lib {
  struct my_scheduler {};

  my_schedule_sender tag_invoke(
    stdexec::tag_t<stdexec::schedule>,
    const my_scheduler& sch)
  {
    return my_schedule_sender{};
  }
}

The compiler finds the tag_invoke overload via ADL (Argument Dependent Lookup) in my_scheduler’s namespace. No inheritance, no class-definition modification, no template specialization — just a free function in your namespace.

tag_invoke’s Two-Step Dispatch

When you write schedule(my_scheduler), a two-step dispatch happens internally:

schedule(my_scheduler)
  → Step 1: Does my_scheduler have a schedule() member?
    → Yes: call my_scheduler.schedule() directly
    → No: proceed to step 2
  → Step 2: ADL lookup for tag_invoke(schedule_t{}, my_scheduler)
    → Found: call the custom tag_invoke overload
    → Not found: compile error

Step 2 is where tag_invoke lives. In stdexec, schedule_t::operator() looks roughly like this:

cpp
struct schedule_t {
  // Main entry: try tag_invoke
  template <class Sch>
  auto operator()(Sch&& sch) const
    noexcept(noexcept(tag_invoke(*this, std::forward<Sch>(sch))))
    -> decltype(tag_invoke(*this, std::forward<Sch>(sch)))
  {
    return tag_invoke(*this, std::forward<Sch>(sch));
  }
};

*this is schedule_t{} — the tag itself is passed as the first argument to tag_invoke. This distinguishes “this is a schedule call” from “this is a connect call.”

Compared to Virtual Functions

tag_invoke and virtual functions are two answers to the same question — how to call the right implementation when you don’t know the concrete type at compile time.

Virtual Functionstag_invoke
Dispatch timingRuntime (vtable)Compile time (ADL + overload resolution)
Runtime overheadYes (pointer indirection)None (direct call after compilation)
noexcept supportRestrictedFull support
GPU compatibleNot availableAvailable
InvasivenessHigh (must inherit base class)Low (just write a free function)

Since all overload resolution happens at compile time, the compiled code looks like a direct function call — no vtable indirection, no runtime query.

Defining Your Own CPO

To provide custom behavior for your type, you need two things:

(1) Define a CPO type (just an empty struct with operator()):

cpp
struct my_cpo_t {
  template <class T>
  auto operator()(T&& t) const
    -> decltype(tag_invoke(*this, std::forward<T>(t)))
  {
    return tag_invoke(*this, std::forward<T>(t));
  }
};
inline constexpr my_cpo_t my_cpo{};

(2) Provide a tag_invoke overload in your type’s namespace:

cpp
struct my_type {
  friend int tag_invoke(my_cpo_t, const my_type&) {
    return 42;
  }
};

my_cpo(my_type{});  // → 42 (found via ADL)

Note the friend keyword — it lets tag_invoke be defined inside the class body while still being a namespace-scope free function discoverable by ADL. This is a C++ idiom that keeps the definition compact.

Relationship with the Domain Mechanism

These two concepts are easy to confuse. A simple distinction:

tag_invokeDomain
GranularitySingle algorithm / single queryEntire scheduler level
Customizesschedule() / connect() / get_domain() etc. CPOsthen / bulk / when_all algorithm implementations
Use caseWriting a new scheduler or receiverMaking the same bulk have different backends on different hardware

The Domain mechanism is built on top of tag_invoke — when you write get_domain(env), the underlying call uses tag_invoke. Their division of labor is:

  • tag_invoke: answers “how do I use this thing?” — how to schedule, how to connect, how to query the environment
  • Domain: answers “at a high level, which implementation should be chosen?” — bulk on GPU should be a CUDA kernel, on CPU should be a thread loop

sender_adaptor_closure — Making Your Algorithm Pipeable

If you want users to call your custom sender adaptor with | pipe syntax:

cpp
struct my_adaptor : stdexec::sender_adaptor_closure<my_adaptor> {
  template <stdexec::sender Sndr>
  auto operator()(Sndr sndr) const {
    return my_algorithm(std::move(sndr));
  }
};

// Users can use it in pipelines
sender | my_adaptor{}

Inheriting sender_adaptor_closure automatically gives your type operator| support — the foundation for building composable pipelines.

Limitations

  1. ADL dependency — tag_invoke overloads must be in the same namespace as the argument type. If you’re using a third-party type and can’t modify its namespace, you can’t add tag_invoke overloads for it. You’ll need an adapter wrapper pattern. (Note: the C++26 draft replaced tag_invoke with member function customization for standard CPOs; tag_invoke remains in use by stdexec and other libraries.)
  2. Error messages can be cryptic — when tag_invoke can’t find a matching overload, the compiler often produces long template instantiation backtraces. This is because SFINAE runs many rounds behind the scenes. stdexec typically adds extra static_asserts to improve error messages, but this isn’t something tag_invoke itself can fix.
  3. Subtle relationship with member functions — stdexec CPOs try the member function path first; if your type coincidentally has a member with the same name, that member takes priority over any tag_invoke overload (which is the deprecated fallback). Standard CPOs (schedule, connect, etc.) check the member function first, and only fall back to tag_invoke if no member exists. Third-party CPO implementations may order dispatch differently, so check the documentation.

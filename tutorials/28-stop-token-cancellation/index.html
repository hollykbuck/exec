28 · Stop Token — Cooperative Cancellation

The Hardest Problem in Async Programming

You fire off a network request, wait 5 seconds, and it still hasn’t come back. You want to cancel it. Let’s look at what traditional C++ offers:

(1) std::thread — you can only .join() (block until it finishes) or .detach() (walk away, leak resources). There’s no graceful “please stop.”

(2) std::future — you can only .get() (block for the result) or .wait_for() (wait with timeout). There’s no .cancel() method.

(3) OS signals — pthread_cancel exists, but it’s essentially a violent thread kill. Resources may not be properly released. It’s been labeled “not recommended.”

C++26 std::execution’s approach is cooperative cancellation — you notify a sender “please stop,” and the sender decides where and how to stop. The key distinction: cancellation is a request, not a command. The sender chooses an appropriate checkpoint to end itself, guaranteeing resources are properly released.

What is a stop_token, simply put? It’s the carrier of this “please stop” request — the sender reads the token’s state, and if a stop has been requested, it responds with set_stopped.

Analogy: The Restaurant Call Button

Think of a stop_token as the “call server” button on a restaurant table:

  • You press the button — equivalent to stop_source.request_stop()
  • The button lights up — equivalent to stop_token.stop_requested() returning true
  • The server sees the light and comes over — equivalent to the sender detecting cancellation at the next checkpoint and completing with set_stopped()

But — pressing the button doesn’t mean the server appears instantly. She might be serving another table. This “delay” reflects the essence of cooperative cancellation: the sender actively checks the token; it isn’t interrupted.

Core Types

TypeRole
inplace_stop_sourceThe cancellation initiator — holds the “has been cancelled” state
inplace_stop_tokenThe cancellation reader — obtained from a source, read-only
never_stop_tokenCompile-time guarantee “never cancels” — enables zero-overhead optimization

Basic Usage

cpp
stdexec::inplace_stop_source source;       // The source of cancellation
stdexec::inplace_stop_token token = source.get_token();  // Get a "reader" from the source

// At some point, another thread decides to cancel
source.request_stop();

// Readers can check
bool stopped = token.stop_requested();  // true

Above, source and token are separate — the owner of source controls “whether to cancel,” while recipients of token can only “ask if cancelled.” This separation guarantees that token holders can’t cancel the source in reverse.

never_stop_token — Compile-Time Optimization

cpp
stdexec::never_stop_token token;
static_assert(token.stop_requested() == false);  // Always false

If a sender knows at compile time that its token is never_stop_token, it can skip every stop_requested() check entirely — zero runtime overhead.

Using in a Pipeline

Reading the Stop Token from the Environment

cpp
sender auto check_stop() {
  return read_env(get_stop_token)
       | then([](auto token) {
           if (token.stop_requested()) {
             // Detected cancellation — do cleanup here…
             // Typically leads to set_stopped()
           }
         });
}

Above, read_env(get_stop_token) pulls the current stop token from the receiver’s environment, then the callback checks whether a stop has been requested.

Providing a Stop Token in a Cancellable Pipeline

cpp
stdexec::inplace_stop_source stop_source;

// A receiver whose environment contains a stop token
struct cancellable_receiver {
  stdexec::inplace_stop_token token_;

  auto get_env() const noexcept {
    return stdexec::prop{get_stop_token, token_};
  }

  void set_value(auto&&...) && noexcept { /* ... */ }
  void set_error(auto&&) && noexcept { /* ... */ }
  void set_stopped() && noexcept { /* ... */ }
};

Above, the receiver exposes the stop token through get_env(). When another thread calls stop_source.request_stop(), the sender detects token_.stop_requested() == true at the next checkpoint and gracefully exits with set_stopped.

Key Cancellation Checkpoints

Sender algorithms should check the stop token at these points:

(1) Before each iteration of a long loop — if your sender has an inner loop processing a million elements, check the token at the top of each iteration so you don’t waste cycles on work whose result is no longer needed.

(2) Before starting child operations — when_all should glance at the token before launching each child sender: if already cancelled, don’t start new children, just set_stopped immediately.

(3) Before waiting on external resources — if a sender is about to block waiting for an IO event, checking the token before blocking avoids “getting stuck.”

cpp
void start() & noexcept {
  // Checkpoint: before starting real work
  if (get_stop_token(get_env(rcvr_)).stop_requested()) {
    set_stopped(std::move(rcvr_));
    return;
  }
  // Proceed with actual work…
}

Registering a Stop Callback

Beyond periodic polling, you can register a callback that fires immediately when cancellation occurs:

cpp
struct stop_callback {
  stdexec::inplace_stop_token token_;
  operation_state* op_;

  void operator()() noexcept {
    // stop_source.request_stop() was called — this runs
    // Perform emergency cleanup here
    op_->cancel();
  }
};

auto cb = token_.stop_callback(stop_callback{token_, this});

Above, stop_callback is an RAII object — it registers the callback with the token on construction and unregisters on destruction. Use it to respond to cancellation and perform urgent cleanup.

Cooperation with async_scope

When an async_scope is destroyed, it automatically requests all child operations to stop, then waits for them all to finish. This mechanism directly relies on stop tokens — the scope internally holds an inplace_stop_source and calls request_stop() in its destructor:

cpp
{
  exec::async_scope scope;
  scope.spawn(long_running_work);   // long_running_work checks stop token internally
  // scope destructor:
  //   1. Internal stop_source.request_stop()
  //   2. Wait for all child ops to complete (or exit after cancellation)
}

This guarantees that even if long_running_work is “interrupted mid-flight,” the scope’s destruction won’t leak resources or leave dangling pointers.

Limitations

  1. Cooperative means it depends on the cancel-ee’s cooperation. If a sender contains an infinite loop that never checks stop_token, the cancellation request falls on deaf ears — the caller waits forever. This is not a “force termination” mechanism.
  2. Thread safety — multiple threads concurrently calling request_stop() is safe, but there’s no strict memory ordering guarantee between stop_requested() and request_stop(). Token state changes may have some delay before other threads observe them.
  3. One-shot — once inplace_stop_source::request_stop() is called, you can’t “un-cancel” it. If you need a resettable cancellation signal, you’ll need to wrap a new source at a higher level.

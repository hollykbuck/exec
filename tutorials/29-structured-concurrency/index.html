29 · Structured Concurrency — Giving Every Async Operation a Home

On this page

Traditional Multithreading Is Like a Parking Lot With No Exit

Look at this code:

cpp
void bad() {
  int local_data = 42;
  spawn(async_work(&local_data));  // Fire and forget
  // Function returns, local_data is destroyed
  // But async_work may still be running on another thread…
  // It's accessing a variable that's already been destroyed
}

This is where use-after-free comes from. Traditional std::thread + detach turns your async work into “orphans” — you don’t know when they finish, and you can’t guarantee the data they reference still exists when they do.

Structured concurrency aims to eliminate this uncertainty. Its core guarantee:

Before a scope object is destroyed, every operation it manages has already completed (or been cancelled).

In other words, every piece of async work has a clear “guardian.” The guardian won’t leave before every child has come home.

async_scope — stdexec’s Structured Concurrency Solution

cpp
namespace exec {
  class async_scope {
  public:
    void spawn(sender auto sndr);               // Fire and forget
    sender auto spawn_future(sender auto sndr);  // Spawn, collect result later
    sender auto nest(sender auto sndr);           // Don't start, just bind lifetime
    sender auto on_empty() noexcept;              // Wait until scope is empty
    void request_stop() noexcept;                 // Request all children to stop
  };
}

A scope is like a babysitter — it keeps track of every operation it has spawned. When the babysitter goes off duty (destructor), she first tells all the kids “come home” (request_stop), then stands at the door waiting for the last one to walk in (waiting for all operations to finish), and only then locks up.

spawn — Order Takeout, Leave, Don’t Wait

cpp
exec::async_scope scope;

scope.spawn(schedule(sch) | then(work1));
scope.spawn(schedule(sch) | then(work2));

// Wait for all work to complete
sync_wait(scope.on_empty());

spawn is the simplest — you toss work into the scope. The scope guarantees the work will finish before the scope itself is destroyed. The return value? spawn doesn’t care — it returns void. Errors? The default receiver for spawn does nothing with set_error; exceptions are silently swallowed.

So spawn is for: you don’t care about the result, you only care about “is it done yet.” Think: printing a log line, flushing a cache, sending a heartbeat.

spawn_future — Here’s Your Order Number

cpp
sender auto future = scope.spawn_future(
  schedule(sch) | then([] { return 42; })
);

// Pick it up later
auto [result] = sync_wait(std::move(future)).value();
// result == 42

spawn_future doesn’t throw away the result like spawn — it returns a sender you can use to retrieve the result later with sync_wait (or plug into another pipeline).

Comparing the three consumer patterns as restaurant analogies:

MethodAnalogyBehavior
sync_wait(sndr)Stand at the kitchen door waiting, grab the dish as it comes outBlocks, gets the result
scope.spawn(sndr)Order and leave, the dish goes in the trash when doneDoesn’t block, doesn’t care about result
scope.spawn_future(sndr)Order, get a number, pick it up when you wantDoesn’t block, result available later

nest — Put Your Address on File

cpp
sender auto nested = scope.nest(schedule(sch) | then(work));

// nested can be passed around elsewhere
// Its execution will complete under the scope's constraints

nest neither starts execution nor returns a result — it does exactly one thing: binds a sender pipeline’s lifetime to the scope. If the scope is destroyed, this pipeline has either already finished or been cancelled. nest is for: you have a sender you want to pass to another function to start later, but you need a guarantee it will be done before the current scope ends.

Scope Destruction Flow

graph TD
  A["scope destructor begins"] --> B["1. request_stop() — tell all children 'please stop'"]
  B --> C["2. Wait for all children to complete (or exit after cancellation)"]
  C --> D["3. All children are done — safe to destroy scope"]

This flow is deterministic and reliable. The scope destructor blocks until every child operation has ended. When destructing a scope, you never have to worry about work still “in the air.”

Complete Example: Parallel Work + Collect Results

cpp
exec::static_thread_pool ctx{4};
exec::async_scope scope;
auto sch = ctx.get_scheduler();

scope.spawn(schedule(sch) | then([] { printf("side work\n"); }));

sender auto future = scope.spawn_future(
  schedule(sch) | then([] { return 42; })
);

// Wait for both scope empty and future result simultaneously
sync_wait(when_all(
  scope.on_empty(),
  future | then([](int x) { printf("result: %d\n", x); })
));

Above, scope.on_empty() only emits set_value after every spawned operation in the scope has finished. when_all waits for both “scope is empty” and “future has a result” conditions simultaneously.

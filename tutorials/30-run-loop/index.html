Why Do We Need run_loop?

What’s the most painful thing about debugging async code? You set a breakpoint to trace a sender’s execution, but the work runs on another thread and your breakpoint never hits.

run_loop exists to pull async back into sync — it concentrates all sender execution onto a single thread that you drive manually. With run_loop you can:

  • Run senders sequentially on one thread in unit tests, making assertions straightforward
  • Debug sender pipelines with printf, because every operation shares the same call stack
  • Implement a simple event loop that serializes all async work

This also explains why sync_wait uses run_loop internally — sync_wait creates a temporary run_loop, runs the entire sender pipeline on it, and returns the result.

Basic Usage

cpp
stdexec::run_loop loop;

// Run the event loop on a background thread
std::thread worker([&loop] { loop.run(); });

auto sch = loop.get_scheduler();

// Submit work
auto [v] = stdexec::sync_wait(
  schedule(sch) | then([] { return 42; })
).value();

loop.finish();   // Signal the event loop to exit
worker.join();   // Wait for the background thread to finish

Three steps: create the loop → get the scheduler and submit work → call finish() to exit.

Architecture Overview: Four Layers

run_loop’s internals are organized into four layers, each with a single responsibility:

Layer 4: __scheduler_t     ← User-visible scheduler, provides schedule()
Layer 3: __sndr_t          ← Sender returned by schedule(), provides connect() / get_env()
Layer 2: __opstate_t       ← Operation state produced by connect(), embeds the concrete receiver
Layer 1: __task            ← Type-erased work unit, hanging on the lock-free queue

Let’s look from the bottom up.

Layer 1: __task — Function Pointers Instead of Virtual Functions

Every operation scheduled on run_loop becomes a __task hanging on a queue:

cpp
struct __task {
  using __execute_fn_t = void(__task*) noexcept;
  __execute_fn_t* __execute_fn_;  // Function pointer → actual execution function
  __task*         __next_;        // Linked list pointer → for the intrusive queue

  void __execute() noexcept {
    (*__execute_fn_)(this);       // Indirect call through function pointer
  }
};

The key design decision above: why a function pointer instead of a virtual function?

Because __execute needs noexcept (the standard requires start() not to throw), and virtual functions are unavailable in GPU (CUDA) environments. Function pointers + static member function templates are the most portable, zero-overhead “late binding” solution. Each __opstate_t<Rcvr> specialization provides its own __execute_impl static function, and the function pointer is set to point to it.

Layer 2: __opstate_t<Rcvr> — What connect Produces

When connect(sndr, rcvr) is called, it produces an __opstate_t that inherits from __task:

cpp
template <class Rcvr>
struct __opstate_t : __task {
  __atomic_intrusive_queue* __queue_;   // → Points to run_loop's task queue
  std::atomic<size_t>*      __task_count_; // → In-flight task counter
  Rcvr                      __rcvr_;    // → Embedded receiver

  void start() noexcept {
    __task_count_->fetch_add(1, std::memory_order_release);  // Count +1
    __queue_->push(this);                                    // Enqueue
  }
  // __execute_fn_ points to this static function:
  static void __execute_impl(__task* p) noexcept {
    auto& rcvr = static_cast<__opstate_t*>(p)->__rcvr_;
    if (get_stop_token(get_env(rcvr)).stop_requested())
      set_stopped(std::move(rcvr));      // Cancelled
    else
      set_value(std::move(rcvr));        // Normal completion
  }
};

Key design points:

  • start() does no real work — it only pushes this onto the queue and bumps the task counter
  • Actual work happens when the run() thread pulls the task off the queue
  • __execute_impl checks the stop token before calling set_value; if cancelled, goes to set_stopped

Zero-overhead optimization: if the receiver’s environment has never_stop_token (compile-time known to never cancel), the stop_requested() check is optimized out at compile time.

Layer 3: __sndr_t — The Sender Returned by schedule

cpp
struct __sndr_t {
  // connect: directly constructs __opstate_t, no heap allocation
  template <class Rcvr>
  auto connect(Rcvr rcvr) const -> __opstate_t<Rcvr> {
    return {&__queue_, &__task_count_, std::move(rcvr)};
  }

  // get_completion_signatures: compile-time decision based on stop token type
  template <class, class RcvrEnv>
  static consteval auto get_completion_signatures() noexcept {
    if constexpr (unstoppable_token<stop_token_of_t<RcvrEnv>>)
      return completion_signatures<set_value_t()>{};  // Never cancelled → no set_stopped
    else
      return completion_signatures<set_value_t(), set_stopped_t()>{};
  }
};

connect directly constructs the operation state — no heap allocation, all state lives inside the operation state object. get_completion_signatures decides at compile time whether to declare set_stopped based on the receiver’s stop token type, enabling dead code elimination downstream.

MPSC Lock-Free Queue: Treiber Stack

The core data structure of run_loop is a multi-producer, single-consumer lock-free queue:

push(node):                     pop_all():
  CAS write to stack head         atomic exchange the whole linked list
  → notify_one() if first node    → reverse the list (LIFO → FIFO)
                                  → return batch of tasks

Why a stack-and-reverse instead of a double-ended queue? Because a Treiber Stack’s push only needs to CAS one atomic variable (head) — simple and efficient. pop_all grabs everything at once in a batch and reverses, amortizing the reversal cost.

alignas(64) ensures head occupies its own cache line, avoiding false sharing.

The Dance of run() and finish()

run() — The Event Loop

while (!finishing) {
  queue.wait_for_item();    // Block until new tasks arrive (atomic::wait)
  execute_all();            // Batch-execute all queued tasks
}
// Cleanup phase: finish() has been called
while (execute_all() || task_count > 0)
  ;  // Drain remaining tasks, wait for all in-flight ops to complete

Two phases:

  • Phase 1 (normal operation): block waiting for new tasks → batch-dequeue everything → execute one by one → back to waiting
  • Phase 2 (cleanup): finish() has been called, drain the remaining queue, wait until every in-flight task has ended

finish() — Request Exit

cpp
void finish() noexcept {
  __task_count_.fetch_add(2, std::memory_order_release);  // Protect self from early run() exit
  if (!__finishing_.exchange(true)) {
    __queue_.push(&__noop_task);  // Push a dummy task to wake run()
    __task_count_.fetch_sub(1);
  }
}

The net effect of +2 -1 = +1 is to prevent run() from exiting and freeing the loop object before finish() has completed — a clever lifetime protection trick.

The batch execution loop also has a subtle iteration order: advance the iterator first, then execute the task. Executing a task may destroy the current node (the receiver’s set_value could free the operation state), but the iterator has already moved to the next node, so the access is safe.

How sync_wait Uses run_loop

graph TD
  A["sync_wait(sndr)"] --> B["Create run_loop"]
  B --> C["connect(sndr, internal receiver)"]
  C --> D["receiver.get_env() → {get_scheduler: loop.get_scheduler()}"]
  D --> E["start(op)"]
  E --> F["loop.run() — blocks the calling thread"]
  F --> G["receiver's set_value / set_error calls loop.finish()"]
  G --> H["loop.run() returns → sync_wait returns result"]

This is the secret of how sync_wait turns async into sync — it creates a run_loop internally, then stands on loop.run() blocking until the sender pipeline completes and calls finish().

This also explains why sync_wait(get_scheduler()) returns run_loop::scheduler — the sync_wait internal receiver’s environment has its get_scheduler query pointing to this internal run_loop’s scheduler.

Wrapping run_loop as Your Own Event Loop

cpp
class my_event_loop {
  stdexec::run_loop loop_;
  std::thread worker_;

public:
  my_event_loop()
    : worker_([this] { loop_.run(); }) {}

  auto get_scheduler() { return loop_.get_scheduler(); }

  ~my_event_loop() {
    loop_.finish();
    worker_.join();
  }
};

Above, we wrap run_loop into an event loop object with its own worker thread — started in the constructor, gracefully shut down in the destructor. This is a common usage pattern.

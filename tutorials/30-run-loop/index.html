30 · run_loop — The Simplest Execution Context

Standard section: [exec.run.loop] stdexec reference: include/stdexec/__detail/__run_loop.hpp Related tests: tests/test_run_loop.cpp

Overview

run_loop is the simplest execution context in the standard. It is a thread-safe FIFO work queue, manually driven.

cpp
namespace std::execution {
  class run_loop {
  public:
    class scheduler;
    scheduler get_scheduler();
    void run();        // Consume work from the queue, blocks until finish()
    void finish();     // Signal completion, run() returns
  };
}

Basic Usage

cpp
stdexec::run_loop loop;

std::thread worker([&loop] { loop.run(); });

auto sch = loop.get_scheduler();

auto [v] = stdexec::sync_wait(
  schedule(sch) | then([] { return 42; })
).value();

loop.finish();
worker.join();

Architecture Overview

run_loop’s internals consist of 4 layers:

Layer 1: __task — Type-erased Work Unit

All operations scheduled on run_loop inherit from __task:

cpp
struct __task {
  using __execute_fn_t = void(__task*) noexcept;

  __execute_fn_t* __execute_fn_ = nullptr;
  __task*         __next_       = nullptr;

  void __execute() noexcept {
    (*__execute_fn_)(this);   // Call via function pointer
  }
};

Core design:

  • __execute_fn_: Function pointer pointing to __opstate_t::__execute_impl static function. This is a manual implementation of a virtual function — each __opstate_t<Rcvr> specialization provides its own __execute_impl, dispatching set_value/set_stopped to the receiver.
  • __next_: Intrusive linked list node pointer, used by __atomic_intrusive_queue.
  • __execute(): Indirectly calls the actual execution function via the function pointer.

Why not virtual functions? Because __execute needs noexcept, and virtual functions are unavailable in GPU (CUDA) environments. Function pointers + static member functions are the most portable choice across platforms.

Layer 2: __opstate_t<Rcvr> — Sender’s Operation State

The sender returned by schedule(sch) produces __opstate_t when connect(rcvr) is called:

cpp
template <class _Rcvr>
struct __opstate_t : __task {
  __atomic_intrusive_queue<&__task::__next_>* __queue_;
  __std::atomic<std::size_t>*                __task_count_;
  _Rcvr                                      __rcvr_;

  static constexpr void __execute_impl(__task* __p) noexcept {
    auto& __rcvr = static_cast<__opstate_t*>(__p)->__rcvr_;

    if constexpr (unstoppable_token<stop_token_of_t<env_of_t<_Rcvr>>>) {
      set_value(static_cast<_Rcvr&&>(__rcvr));
    }
    else if (get_stop_token(get_env(__rcvr)).stop_requested()) {
      set_stopped(static_cast<_Rcvr&&>(__rcvr));
    }
    else {
      set_value(static_cast<_Rcvr&&>(__rcvr));
    }
  }

  void start() noexcept {
    __task_count_->fetch_add(1, std::memory_order_release);
    __queue_->push(this);
  }
};

start() does only two things:

  1. Increment the task count (__task_count_) — notifies run_loop of a new in-flight task
  2. Enqueue (__queue_->push(this)) — submits itself to the MPSC queue

Note: start() does not execute any actual work. Work is deferred until the run() thread consumes the queue.

Cancellation Check in __execute_impl

__execute_impl checks the stop token before calling set_value. If the environment’s stop token is never_stop_token (compile-time known to be non-cancellable), it skips the check and directly calls set_value. This is a zero-overhead abstraction.

Layer 3: __sndr_t — Schedule Sender

The sender returned by run_loop::scheduler::schedule():

cpp
struct __sndr_t {
  using sender_concept = sender_tag;

  template <class _Rcvr>
  auto connect(_Rcvr __rcvr) const noexcept -> __opstate_t<_Rcvr> {
    return __opstate_t<_Rcvr>{
      &__loop_->__task_count_,
      &__loop_->__queue_,
      static_cast<_Rcvr&&>(__rcvr)
    };
  }

  template <class, class _RcvrEnv>
  static consteval auto get_completion_signatures() noexcept {
    if constexpr (unstoppable_token<stop_token_of_t<_RcvrEnv>>)
      return completion_signatures<set_value_t()>{};
    else
      return completion_signatures<set_value_t(), set_stopped_t()>{};
  }

  auto get_env() const noexcept -> __attrs_t {
    return __attrs_t{__loop_};
  }
};

Key points:

  • connect: Directly constructs __opstate_t, passing the queue pointer and task counter. No allocation — all state is stack- or embedded-allocated.
  • get_completion_signatures: Determines whether set_stopped is possible based on the stop token type of the receiver’s environment. This is a compile-time conditional branch with zero runtime overhead.
  • get_env: The returned environment contains the get_completion_scheduler<set_value_t> query, enabling algorithms like continues_on to find the run_loop scheduler.

Layer 4: __scheduler_t — Scheduler

cpp
struct __scheduler : __attrs_t {
  using scheduler_concept = scheduler_tag;

  auto schedule() const noexcept -> __sndr_t {
    return __sndr_t{this->__loop_};
  }
};

__scheduler inherits from __attrs_t, which implements queries like get_completion_scheduler, get_completion_domain, __execute_may_block_caller, etc.

__atomic_intrusive_queue: MPSC Lock-Free Queue

run_loop’s core data structure is a multi-producer, single-consumer lock-free queue:

cpp
template <class _Tp, _Tp* _Tp::* _NextPtr>
class alignas(64) __atomic_intrusive_queue {
  std::atomic<_Tp*> __head_{nullptr};

public:
  bool push(_Tp* __node) noexcept {
    _Tp* __old_head = __head_.load(std::memory_order_relaxed);
    do {
      __node->*_NextPtr = __old_head;
    } while (!__head_.compare_exchange_weak(
        __old_head, __node, std::memory_order_acq_rel));

    if (__old_head != nullptr)
      return false;                // Queue not empty, no notification needed

    __head_.notify_one();          // Queue went from empty→non-empty, wake consumer
    return true;
  }

  void wait_for_item() noexcept {
    __head_.wait(nullptr);          // Block until __head_ != nullptr
  }

  auto pop_all() noexcept -> __intrusive_queue {
    auto* __list = __head_.exchange(nullptr, std::memory_order_acquire);
    return __intrusive_queue::make_reversed(__list);
  }
};

How It Works

graph LR
  subgraph push ["push(node)"]
    P1["CAS push onto stack"]
    P2["notify (if needed)"]
    P1 --> P2
  end
  subgraph pop_all ["pop_all()"]
    Q1["exchange(nullptr)"]
    Q2["Reverse stack → FIFO queue"]
    Q1 --> Q2
  end
  push --> pop_all
  1. push: Uses the Treiber stack algorithm (CAS at the head). If the queue was empty before insertion, calls notify_one() to wake the consumer.
  2. pop_all: Atomically swaps out the entire linked list, then reverses it (since the Treiber stack is LIFO; reversing makes it FIFO).
  3. wait_for_item: Uses std::atomic::wait primitive to block, more efficient than busy-waiting.

alignas(64) ensures __head_ occupies its own cache line, avoiding false sharing.

Why Stack + Reverse Instead of a Queue?

The Treiber stack’s push only needs CAS at the head, making it simple to implement. pop_all + make_reversed batch-retrieves all elements, spreading the cost. Compared to CAS at the tail of a linked list or a double-linked scheme, this design is more efficient in the MPSC scenario.

Coordination of run() and finish()

run() — Event Loop

cpp
void run() noexcept {
  while (!__finishing_.load(std::memory_order_acquire)) {
    __queue_.wait_for_item();       // Block waiting for work
    __execute_all();                 // Execute batch tasks
  }
  // Cleanup phase: execute remaining tasks, wait for in-flight tasks to complete
  while (__execute_all()
      || __task_count_.load(std::memory_order_acquire) > 0)
    ;
}

Two phases:

Phase 1 — Normal loop: When __finishing_ is false, block waiting for new tasks → batch-execute all queued tasks → continue waiting. Multiple threads may call run() simultaneously (but there is only one consumer, since pop_all is single-consumer).

Phase 2 — Cleanup: finish() has been called. Continue executing remaining tasks and wait for all in-flight tasks to complete (__task_count_ reaches zero).

finish() — Request Stop

cpp
void finish() noexcept {
  __task_count_.fetch_add(2, std::memory_order_release);  // Prevent lifetime race
  if (!__finishing_.exchange(true, std::memory_order_acq_rel)) {
    __queue_.push(&__noop_task);       // Wake consumer
    __task_count_.fetch_sub(1, std::memory_order_release);
    return;
  }
  __task_count_.fetch_sub(2, std::memory_order_release);  // Repeated finish
}

The __task_count_ increment logic here is key:

graph TD
  A["finish() adds 2"]
  A --> B["+1: protect itself<br/>prevent run() from exiting before finish() completes"]
  A --> C["+1: noop_task<br/>ensure it is consumed after enqueueing"]
  C --> D["-1 when noop_task executes"]
  A --> E["finish() decrements by 1"]
  D --> F["Net effect: +2 -1 -1 = 0"]
  E --> F

If finish() is called repeatedly, the second call directly does fetch_sub(2) and returns.

__execute_all() — Batch Consumption

cpp
bool __execute_all() noexcept {
  auto __queue = __queue_.pop_all();
  if (__queue.begin() == __queue.end())
    return false;

  std::size_t __task_count = 0;
  auto __it = __queue.begin();
  do {
    auto __prev = __it++;       // Advance iterator first, then execute task
    (*__prev)->__execute();     // Execute → may trigger set_value/set_stopped
    ++__task_count;
  } while (__it != __queue.end());

  __queue.clear();
  __task_count_.fetch_sub(__task_count, std::memory_order_release);
  return true;
}

Note the order in the do-while loop: advance the iterator first, then execute the task. Executing the task may cause the node pointed to by __prev to be destroyed (because the receiver’s set_value may release the operation state), but __it already points to the next node, so access is safe.

sync_wait’s Use of run_loop

sync_wait internally uses run_loop to drive asynchronous operations:

graph TD
  A["sync_wait(sndr)"]
  A --> B["Create run_loop loop"]
  A --> C["connect(sndr, internal receiver)"]
  C --> D["receiver.get_env() returns<br/>{get_scheduler: loop.get_scheduler()}"]
  A --> E["start(op)"]
  A --> F["loop.run()"]
  F --> G["Block current thread until finish()"]
  A --> H["Return result"]

This is why sync_wait(get_scheduler()) returns run_loop::scheduler:

cpp
// Test: "sync_wait provides run_loop::scheduler via get_scheduler"
auto [sched] = stdexec::sync_wait(stdexec::get_scheduler()).value();
// sched's type is run_loop::scheduler

sync_wait’s receiver environment points get_scheduler to the internal run_loop:

cpp
struct __receiver {
  run_loop* __loop_;

  auto get_env() const noexcept {
    return env{prop{get_scheduler, __loop_->get_scheduler()}};
  }
  // set_value / set_error / set_stopped all call __loop_.finish()
};

This means any work redirected to the run_loop scheduler via continues_on or starts_on will execute synchronously on the calling thread of sync_wait.

Custom Event Loop

run_loop is a minimal reference implementation for a manual event loop. It can be used to build dedicated execution contexts:

cpp
class my_event_loop {
  stdexec::run_loop loop_;
  std::thread worker_;

public:
  my_event_loop()
    : worker_([this] { loop_.run(); }) {}

  auto get_scheduler() { return loop_.get_scheduler(); }

  ~my_event_loop() {
    loop_.finish();
    worker_.join();
  }
};

Key Takeaways

  1. 4-layer architecture: __task → __opstate_t<Rcvr> → __sndr_t → __scheduler_t, each with a single responsibility
  2. Function pointer type-erasure: __execute_fn_ replaces virtual functions, supporting GPU compilation
  3. MPSC queue: Treiber stack + CAS + atomic::wait, reversed to FIFO during batch consumption
  4. __task_count_ lifetime management: prevents run() from exiting before in-flight tasks complete
  5. sync_wait integration: internal run_loop drives execution, get_scheduler query points to it
  6. Zero-overhead cancellation: unstoppable_token skips set_stopped check at compile time

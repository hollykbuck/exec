31 · parallel_scheduler — Standard Parallel Scheduler

Standard section: [exec.par.scheduler] stdexec reference: include/stdexec/__detail/__parallel_scheduler*.hpp Related tests: tests/test_parallel_scheduler.cpp

Overview

parallel_scheduler is a general-purpose parallel scheduler provided by the standard library for CPU parallel computing. It is the sender-model counterpart of the C++17 std::execution::par execution policy.

cpp
namespace std::execution {
  class parallel_scheduler;
  parallel_scheduler get_parallel_scheduler();
}

Basic Usage

cpp
auto par_sch = get_parallel_scheduler();

auto result = sync_wait(
  starts_on(par_sch,
    bulk(1000, [](int idx) {
      compute(idx);  // Execute on the parallel thread pool
    })
  )
);

Backend Replacement Mechanism

Compile-time Replacement (Weak Linking)

The parallel_scheduler_replacement namespace provides a weak symbol:

cpp
namespace std::execution::parallel_scheduler_replacement {
  // weak function: can be overridden by user definition
  [[nodiscard]]
  auto query_parallel_scheduler_backend()
    -> std::shared_ptr<parallel_scheduler_backend>;
}

Users replace the default backend by defining a strong symbol with the same name:

cpp
// my_backend.cpp — replace parallel_scheduler backend at compile time
#include <stdexec/execution.hpp>

auto std::execution::parallel_scheduler_replacement
::query_parallel_scheduler_backend()
  -> std::shared_ptr<parallel_scheduler_backend>
{
  return std::make_shared<my_parallel_backend>();
}

Runtime Replacement (Deprecated)

cpp
// Deprecated: runtime replacement may cause undefined behavior
namespace std::execution::parallel_scheduler_replacement {
  [[deprecated]]
  auto set_parallel_scheduler_backend(
    __parallel_scheduler_backend_factory_t new_factory
  ) -> __parallel_scheduler_backend_factory_t;
}

Backend Interface

parallel_scheduler_backend is a pure virtual interface defined in std::execution::parallel_scheduler_replacement:

cpp
struct parallel_scheduler_backend
  : __any::__iabstract<__iparallel_scheduler_backend>
{};

Its core virtual functions:

MethodPurpose
schedule(rcvr, scratch)Schedule a single unit of work
schedule_bulk_chunked(count, rcvr, scratch)Schedule bulk work (chunked)
schedule_bulk_unchunked(count, rcvr, scratch)Schedule bulk work (item-by-item)

These methods communicate with the frontend via receiver proxy:

sequenceDiagram
  participant F as Frontend
  participant B as Backend
  F->>B: schedule(proxy, scratch)
  Note over F: __receiver_proxy<br/>(inherits receiver_proxy)
  B-->>F: On completion, call proxy.set_value()
  F->>B: schedule_bulk_chunked(count, proxy, scratch)
  Note over F: __bulk_receiver_proxy<br/>(inherits bulk_item_receiver_proxy)
  B-->>F: For each chunk: proxy.execute(begin, end)
  B-->>F: On completion: proxy.set_value()

receiver_proxy and bulk_item_receiver_proxy

The frontend exposes the receiver’s communication capabilities to the backend through proxies:

cpp
class receiver_proxy {
public:
  virtual void set_value() noexcept                   = 0;
  virtual void set_error(std::exception_ptr) noexcept  = 0;
  virtual void set_stopped() noexcept                  = 0;

  // Type-erased environment query
  template <class Value, class Query>
  auto try_query(Query) const noexcept -> std::optional<Value>;
};

struct bulk_item_receiver_proxy : receiver_proxy {
  virtual void execute(size_t begin, size_t end) noexcept = 0;
};

The frontend’s __receiver_proxy_base<Rcvr> implements the bridge from a real receiver to the proxy:

classDiagram
  class receiver_proxy {
    +set_value()
    +set_error()
    +set_stopped()
    +try_query()
  }
  class __receiver_proxy_base~Rcvr~ {
    +set_value() → Rcvr::set_value()
    +set_error() → Rcvr::set_error()
    +set_stopped() → Rcvr::set_stopped()
    +try_query() → get_stop_token / get_allocator / get_start_scheduler
  }
  receiver_proxy <|-- __receiver_proxy_base~Rcvr~

Cancellation propagation is implemented via __stop_callback_for: when the backend requests a stop, the proxy notifies the frontend through inplace_stop_source.

parallel_scheduler and bulk Interaction

bulk(sndr, N, f) executes on parallel_scheduler through the following path:

graph TD
  A["bulk(just(0), 1000, work_fn)"]
  A --> B["domain.apply_sender(bulk_t{}, just(0), 1000, work_fn)"]
  B --> C["Extract parallel_scheduler's domain"]
  C --> D["Create __bulk_sender"]
  A --> E["connect(__bulk_sender, rcvr)"]
  E --> F["Create __bulk_opstate"]
  F --> G["Outer: the entire bulk operation<br/>(schedule + wait for all chunks to complete)"]
  F --> H["Inner: each chunk"]
  A --> I["start()"]
  I --> J["Call backend->schedule_bulk_chunked(N, proxy, scratch)"]
  J --> K["Backend splits [0, N) into chunks, assigns to worker threads"]
  K --> L["Each chunk: proxy.execute(begin, end)"]
  L --> M["for i in [begin, end): work_fn(i)"]
  I --> N["Wait for all chunks to complete → proxy.set_value()"]
  A --> O["rcvr.set_value()"]

The backend is responsible for the chunking strategy:

graph LR
  T0["Thread 0: [0, 250)"] --> E0["execute(0, 250)"]
  T1["Thread 1: [250, 500)"] --> E1["execute(250, 500)"]
  T2["Thread 2: [500, 750)"] --> E2["execute(500, 750)"]
  T3["Thread 3: [750, 1000)"] --> E3["execute(750, 1000)"]

schedule_bulk_unchunked is used for scenarios where chunking optimization is unnecessary (each item is very small, and chunking overhead is not worthwhile).

Default Implementation

stdexec’s default parallel backend uses std::thread with a work-stealing implementation:

block
  columns 1
  block["work_stealing_thread_pool"]
    block["Worker 0: [ ][ ][ ][ ] ← Local queue (LIFO)"]
    block["Worker 1: [ ][ ][ ][ ] ← Stealable"]
    block["Worker 2: [ ][ ][ ][ ] ← Other workers"]
    block["Worker 3: [ ][ ][ ][ ] ← Tail stealing"]
  end
  block["Idle threads steal work from busy threads' tails"]
  end

Benefits of work-stealing:

  • Load balancing: idle threads steal work from busy threads
  • Cache-friendly: threads prioritize their local queue (LIFO — recently added tasks are still in cache)
  • No centralized scheduler: reduces lock contention

task_scheduler: Type-erased Wrapper

task_scheduler is a type-erased scheduler wrapper:

cpp
class task_scheduler {
  template <scheduler Sch>
  task_scheduler(Sch sched, auto allocator);
};

It wraps any scheduler into a uniform parallel_scheduler_backend interface, allowing parallel_scheduler to dispatch work to different types of execution contexts through the backend mechanism.

Internal implementation: task_scheduler creates a receiver_proxy for the wrapped scheduler, forwarding parallel_scheduler_backend::schedule() as schedule(wrapped_sch) | then(complete_proxy).

Key Takeaways

  1. Frontend-Backend architecture: parallel_scheduler decouples scheduling policy from parallel implementation via the virtual parallel_scheduler_backend interface
  2. Compile-time backend replacement: the weak linking mechanism allows users to replace the default implementation (TBB / Taskflow / custom) without modifying library code
  3. receiver_proxy type erasure: bridges sender receivers and the parallel backend through virtual functions, hiding the concrete receiver type
  4. Native bulk support: the backend directly provides schedule_bulk_chunked / schedule_bulk_unchunked, avoiding manual chunking in the frontend layer
  5. Cancellation forwarding: the backend indirectly notifies the frontend of stop requests via inplace_stop_source
  6. task_scheduler: a type-erased wrapper that adapts any scheduler to parallel_scheduler_backend

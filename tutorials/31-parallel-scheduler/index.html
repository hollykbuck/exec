31 · parallel_scheduler — CPU Parallel Computing Scheduler

When Do You Need parallel_scheduler?

If you’re doing CPU-intensive work — image processing, scientific computing, data compression — you want to use every CPU core. You could build your own thread pool + task queue, but parallel_scheduler has built-in work-stealing that’s more efficient than a hand-rolled load balancer.

parallel_scheduler is C++26 std::execution’s official counterpart to C++17’s std::execution::par. It’s the only scheduler in the standard specifically designed for CPU parallel computation.

Basic Usage

cpp
auto par_sch = get_parallel_scheduler();

sync_wait(
  starts_on(par_sch,
    bulk(1000, [](int idx) {
      compute(idx);  // Parallelizable compute-intensive work
    })
  )
);

Simpler than static_thread_pool — you don’t specify thread count; the implementation auto-matches hardware concurrency.

Architecture: Frontend + Backend

parallel_scheduler uses a separated frontend (type erasure layer) and backend (concrete implementation) architecture:

flowchart TD
    U["User code: bulk(N, f)"]
    F["Frontend (sender/receiver API)<br/>← receiver_proxy for type erasure"]
    B["Backend (virtual interface)<br/>← schedule / schedule_bulk_*"]
    D["Default impl (std::thread pool)<br/>← Replaceable with TBB / Taskflow"]

    U --> F --> B --> D

This layering lets you replace the underlying implementation without changing a single line of user code. If your company has a custom GPU task scheduler, or uses TBB’s scheduling system, you just swap the backend layer — all user code stays the same.

Compile-Time Backend Replacement

The replacement uses C++ weak symbol tricks:

cpp
// Your file: my_backend.cpp
#include <stdexec/execution.hpp>

auto std::execution::parallel_scheduler_replacement
::query_parallel_scheduler_backend()
  -> std::shared_ptr<parallel_scheduler_backend>
{
  return std::make_shared<my_tbb_backend>();  // Swap in a TBB implementation
}

Above, defining a strong-symbol function is all it takes — the linker overrides the library’s default weak symbol. No preprocessor macros, no config files, no library modifications. Just one .cpp file.

How Frontend Talks to Backend: receiver_proxy

The frontend exposes the receiver to the backend through a receiver_proxy — using virtual functions to erase the concrete receiver type:

cpp
class receiver_proxy {
public:
  virtual void set_value() noexcept                  = 0;
  virtual void set_error(std::exception_ptr) noexcept  = 0;
  virtual void set_stopped() noexcept                   = 0;
  template <class Value, class Query>
  auto try_query(Query) const -> std::optional<Value>;  // Type-erased env query
};

The templated __receiver_proxy_base<Rcvr> implements this virtual interface, bridging “concrete receiver” and “type-erased proxy.” The backend only calls virtual functions — it never needs to know the concrete receiver type.

How bulk Executes on parallel_scheduler

graph TD
  A["bulk(just(0), 1000, work_fn)"] --> B["Fetch parallel_scheduler's domain"]
  B --> C["Domain replaces bulk with __bulk_sender"]
  C --> D["connect → __bulk_opstate"]
  D --> E["start() → call backend.schedule_bulk_chunked(N, proxy)"]
  E --> F["Backend splits N into chunks, dispatches to worker threads"]
  F --> G["Each chunk: proxy.execute(begin, end)"]
  G --> H["for i in [begin, end): work_fn(i)"]

The backend is responsible for the chunking strategy:

Thread 0: [0, 250)     → execute(0, 250)
Thread 1: [250, 500)   → execute(250, 500)
Thread 2: [500, 750)   → execute(500, 750)
Thread 3: [750, 1000)  → execute(750, 1000)

Each thread calls bulk_item_receiver_proxy::execute(begin, end) once, running a for loop over its assigned chunk.

Default Backend: Work-Stealing

stdexec’s default parallel_scheduler uses std::thread + work-stealing:

Worker 0: [task][task][task][ ]  ← Local queue (LIFO, cache-friendly)
Worker 1: [task][task][ ][ ]     ← Others can steal from here
Worker 2: [ ][ ][ ][ ]          ← Idle → will steal from others
Worker 3: [task][task][task][ ]  ← Tail can be stolen

Three benefits of work-stealing:

  • Load balancing: idle threads steal work from busy threads’ tails
  • Cache-friendly: each thread prioritizes its local queue (LIFO), so recently accessed data is still in cache
  • No central scheduler: no global task distribution lock, reducing contention

Limitations

  1. Default implementation depends on std::thread — may be unavailable in embedded or WASM environments. These platforms need a backend replacement.
  2. Not suited for precise thread count control — thread count is implementation-defined. If you need exactly N threads (e.g., hardware pinning), use static_thread_pool.
  3. Heap allocation — receiver_proxy type erasure may introduce heap allocation. By contrast, static_thread_pool operation states can be fully stack-allocated.
  4. Platform dependency — get_parallel_scheduler() behavior may differ subtly across Windows, Linux, and macOS.

32 · Implementing then from Scratch — Custom Sender + Receiver + Operation State

On this page

Why Implement then Yourself?

For the first 31 chapters you’ve been using senders and receivers. In this chapter you’ll hand-write one. The then you build won’t be faster than the standard library’s, but you’ll finally understand how a sender and receiver are glued together — how a sender’s connect produces an operation state, and how a receiver’s three signals get intercepted and forwarded.

It’s like the driving instructor popping the hood — you won’t necessarily be able to fix the engine after seeing it, but you’ll understand how the power gets to the wheels.

The Four Core Pieces

Implementing a sender adaptor requires four things. You can’t skip any of them:

(1) Sender type — holds the child sender and the function object, exposes connect() (2) Operation State — the product of connect(), holds the child operation state and the external receiver (3) Internal Receiver — intercepts the child sender’s completion signals, calls the function, forwards to the external receiver (4) Completion Signatures — compile-time computation of the post-function-transformation signal types

If these four pieces were an electrical circuit:

  • Sender is the junction box — connecting input to output
  • Operation State is the switch — controlling when current flows
  • Internal Receiver is the relay — receiving signals, processing, forwarding
  • Completion Signatures is the spec sheet — “input format X, output format Y”

Step 1: Define the Sender Type

cpp
template <stdexec::sender Sndr, std::invocable Fn>
struct then_sender {
  using sender_concept = stdexec::sender_tag;

  Sndr sndr_;
  Fn fn_;

  explicit then_sender(Sndr sndr, Fn fn)
    : sndr_(std::move(sndr))
    , fn_(std::move(fn))
  {}
};

Above, then_sender is just a struct holding two things — the child sender it “catches” and the function to apply to its result. The sender_concept tag tells the type system “this is a sender.”

Step 2: Declare Completion Signatures

Completion signatures tell downstream code at compile time “what signals I will emit”:

cpp
template <class... Args>
using _set_value_t =
  stdexec::completion_signatures<stdexec::set_value_t(std::invoke_result_t<Fn, Args...>)>;

template <class... Env>
using _completions_t = stdexec::__transform_completion_signatures_t<
  stdexec::completion_signatures_of_t<Sndr, Env...>,
  stdexec::completion_signatures<stdexec::set_error_t(std::exception_ptr)>,
  _set_value_t>;

// Inside then_sender:
template <class Self, class... Env>
static consteval auto get_completion_signatures() -> _completions_t<Env...> {
  return {};
}

The core idea above is “type translation” — the upstream sender says “I will emit set_value(int, double),” and after then(f), you say “I will emit set_value(std::invoke_result_t<f, int, double>).” __transform_completion_signatures_t is the tool that performs this translation. We also add a default set_error_t(std::exception_ptr) since the callback f might throw.

Step 3: Define the Operation State

The operation state is what connect() returns — it chains the child sender and the external receiver together:

cpp
template <stdexec::receiver R, class Sndr, class Fn>
struct then_op {
  Sndr sndr_;
  Fn fn_;
  R rcvr_;
  stdexec::connect_result_t<Sndr&, then_receiver<then_op>> op_;  // Child op state

  then_op(Sndr sndr, Fn fn, R rcvr)
    : sndr_(std::move(sndr))
    , fn_(std::move(fn))
    , rcvr_(std::move(rcvr))
    , op_(stdexec::connect(sndr_, then_receiver<then_op>{this}))
  {}

  void start() & noexcept {
    stdexec::start(op_);  // Only starts the child — does no work itself
  }
};

Above, the constructor does two things at once — saves the external receiver and calls connect to create the child operation state. start() is deliberately simple: it only starts the child operation, handing execution control to the child sender.

Step 4: Define the Internal Receiver

The internal receiver is the “brain” of then — it receives the child sender’s signals, calls the function f, then forwards the result to the external receiver:

cpp
template <class Op>
struct then_receiver {
  using receiver_concept = stdexec::receiver_tag;
  Op* op_;

  void set_value(auto&&... args) && noexcept {
    STDEXEC_TRY {
      if constexpr (std::same_as<void, std::invoke_result_t<Fn, decltype(args)...>>) {
        std::invoke(op_->fn_, std::forward<decltype(args)>(args)...);
        stdexec::set_value(std::move(op_->rcvr_));        // void → no-arg set_value
      } else {
        stdexec::set_value(std::move(op_->rcvr_),
          std::invoke(op_->fn_, std::forward<decltype(args)>(args)...));  // T → set_value(T)
      }
    }
    STDEXEC_CATCH_ALL {
      stdexec::set_error(std::move(op_->rcvr_), std::current_exception());  // Exception → set_error
    }
  }

  void set_error(auto&& e) && noexcept {
    stdexec::set_error(std::move(op_->rcvr_), std::forward<decltype(e)>(e));
  }

  void set_stopped() && noexcept {
    stdexec::set_stopped(std::move(op_->rcvr_));
  }

  auto get_env() const noexcept {
    return stdexec::get_env(op_->rcvr_);  // Pass environment through
  }
};

Above, set_value has three branches:

  • Function doesn’t throw → call and forward directly (fastest path)
  • Function may throw → wrap with STDEXEC_TRY, convert exceptions to set_error
  • Function returns void → set_value() with no arguments, since “there’s no data to forward”

set_error and set_stopped are trivial — just forward as-is to the external receiver. The environment (get_env) also passes through unchanged.

Step 5: Implement connect

cpp
// Inside then_sender:
template <stdexec::receiver R>
  requires stdexec::sender_to<Sndr, then_receiver<then_op<Sndr, Fn, R>>>
auto connect(R r) && -> then_op<Sndr, Fn, R> {
  return then_op<Sndr, Fn, R>{std::move(sndr_), std::move(fn_), std::move(r)};
}

Step 6: Pipe Adaptor

cpp
struct then_t {
  template <stdexec::sender Sndr, class Fn>
  auto operator()(Sndr sndr, Fn fn) const {
    return then_sender<Sndr, Fn>{std::move(sndr), std::move(fn)};
  }

  // Support partial application: then(fn) returns a pipeable closure
  template <class Fn>
  auto operator()(Fn fn) const {
    return stdexec::__sender_adaptor_closure{fn};
  }
};
inline constexpr then_t then{};

Complete Recap

Implementing then requires seven elements:

  1. Sender type — holds the child sender and the function object
  2. Completion signatures — compile-time computation of transformed signatures
  3. Operation state — holds the child operation state, the function, and the external receiver
  4. Internal receiver — receives the child operation’s result, executes the function, forwards to the external receiver
  5. connect() — creates the operation state
  6. start() — starts the child operation
  7. Pipe adaptor — supports operator|

stdexec’s examples/algorithms/then.hpp is the complete reference implementation containing all of the above.

Limitations

This hand-rolled implementation is a minimal educational version, missing production necessities:

  • Move-semantics optimization — extensive use of std::move to avoid accidental copies of large objects
  • SBO (Small Buffer Optimization) — local buffering of small function objects to avoid heap allocation
  • connect noexcept consistency — the standard requires that if connect is noexcept for one receiver with a given environment, it must be noexcept for all receivers with that environment. The educational implementation doesn’t track this.
  • Compile-time zero overhead — using [[no_unique_address]] and EBO to eliminate space wasted by empty function objects

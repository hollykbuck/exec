33 · Implementing retry from Scratch — Error Recovery Pattern

On this page

Why Do We Need retry?

You send a message on your phone. Bad signal — it fails. What does your phone do? Retry automatically. This is a fundamental requirement for virtually all networked applications.

In network programming, transient failures (network jitter, temporary service unavailability, timeouts) are the norm, not the exception. The core idea of the retry pattern is simple — if the failure might be temporary, try again.

How do we implement this in the sender world? The most intuitive idea: upon receiving set_error, run the same sender again. But there’s a technical snag — once a sender has been connected, it’s bound to a receiver. To retry, you need to tear down the old connection and build a new one. std::optional + emplace is the tool for this job.

Step 1: Define a “Sometimes Fails” Sender

For testing, write a sender that fails the first N times and succeeds afterwards:

cpp
struct fail_some {
  using sender_concept = ex::sender_tag;
  using completion_signatures = ex::completion_signatures<
    ex::set_value_t(int),
    ex::set_error_t(std::exception_ptr)
  >;

  mutable int failure_count_ = 2;

  template <class R>
  struct op {
    R r_;
    const fail_some* parent_;
    void start() & noexcept {
      if (parent_->failure_count_ > 0) {
        --parent_->failure_count_;
        ex::set_error(std::move(r_), std::exception_ptr{});  // First two times: pretend to fail
      } else {
        ex::set_value(std::move(r_), 42);                     // Third time: succeed
      }
    }
  };

  template <class R>
  auto connect(R r) const -> op<R> { return {std::move(r), this}; }
};

Above, failure_count_ decrements — the first two invocations return set_error, the third returns set_value(42). This is the ideal input for testing retry logic.

Step 2: The Foundation — upon_error Turns Error Into Success

upon_error is the foundation of retry: it converts the set_error signal into set_value, so downstream sees “all good” no matter how many times the upstream failed:

cpp
auto [v] = ex::sync_wait(
  ex::just_error(std::make_exception_ptr(std::runtime_error{"fail"}))
    | ex::upon_error([](std::exception_ptr) { return 42; })
).value();
// v == 42  —— the error was "swallowed", downstream only sees set_value

Step 3: receiver_adaptor — Hijack Only the Signal I Care About

receiver_adaptor is like a “signal relay station.” By default, it forwards set_value, set_error, and set_stopped straight through to the downstream. You can choose to hijack only one signal — for example, hijack only set_error: instead of forwarding it, trigger a retry:

cpp
struct _retry_receiver : exec::receiver_adaptor<_retry_receiver> {
  _op* o_;

  auto base() && noexcept -> auto& { return o_->r_; }

  // Only hijack set_error; set_value and set_stopped auto-forward via receiver_adaptor to base()
  void set_error(auto&&) && noexcept {
    o_->_retry();  // Error → retry!
  }
};

Above, we only override set_error. set_value and set_stopped are untouched — they flow through receiver_adaptor’s default behavior, forwarding to base() (the external receiver).

Step 4: std::optional + emplace — Rebuild the Child Operation State

This is the most crucial step in retry — when _retry() is called, destruct the old operation state and construct a new one in-place inside the std::optional:

cpp
struct _op {
  std::optional<connect_result_t<S&, _retry_receiver>> o_;
  S s_;  // Child sender (keep a copy for rebuilding)
  R r_;  // External receiver

  void _retry() noexcept {
    STDEXEC_TRY {
      o_.emplace(connect(s_, _retry_receiver{this}));  // Re-connect
      start(*o_);                                       // Re-start
    }
    STDEXEC_CATCH_ALL {
      set_error(static_cast<R&&>(r_), std::current_exception());
    }
  }
};

The cleverness above is that emplace destructs the old state and constructs the new state in-place inside the optional — just like “closing the old connection and opening a new one.” If even connect fails (e.g., the child sender’s constructor throws), that’s a genuinely unrecoverable error — forward set_error directly.

Complete retry Flow

graph TD
  A["start() → launch child sender"]
  A --> B["Child sender executing…"]
  B --> C["set_error → _retry_receiver::set_error"]
  C --> D["Call _retry()"]
  D --> E["optional.emplace(connect(...)) — rebuild operation state"]
  E --> F["start(*o_) — restart"]
  F --> B
  B --> G["set_value(42) → forward to external receiver ✓"]
  B -.->|retry loop| C

Bounded Retry

The implementation above retries infinitely — if the error is permanent (e.g., missing configuration), it loops forever. Production code needs a retry cap:

cpp
struct retry_with_limit {
  int max_retries_;
  int attempt_count_ = 0;

  void _retry() noexcept {
    if (++attempt_count_ > max_retries_) {
      set_error(std::move(r_), std::make_exception_ptr(
        std::runtime_error("max retries exceeded")));
      return;
    }
    // Otherwise continue retrying…
  }
};

Key Takeaways

  1. upon_error is the cornerstone of retry — it recovers errors onto the normal value path
  2. std::optional + emplace — rebuilds the child connection within the operation state’s lifetime without reallocation
  3. receiver_adaptor — conveniently intercept only the signal you care about, forward the rest unchanged
  4. Infinite vs bounded retry — the educational skeleton retries endlessly; production code must add a limit and backoff delay

stdexec’s examples/algorithms/retry.hpp contains the full reference implementation.

Limitations

  1. No discrimination of permanent vs transient errors — retry can’t distinguish “transient failure” (network jitter) from “permanent failure” (permission denied). Production code must decide whether to retry based on error type, or risk infinite looping on permanent errors.
  2. Child sender must be reconnectable — retry assumes that every connect(sender, receiver) call is valid. If the child sender is one-shot (e.g., just(std::move(unique_resource))), retrying is impossible.
  3. No backoff strategy — the educational retry retries immediately. Production environments typically need exponential backoff between retries to avoid congestion.

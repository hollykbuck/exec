34 · Writing a Custom Scheduler

On this page

Most of the Time You Don’t Need This Chapter

The standard library’s run_loop, static_thread_pool, and parallel_scheduler already cover 90% of scenarios. Only write a custom scheduler when you’re connecting to a specialized execution resource — an embedded RTOS, a browser’s Web Worker, a custom coroutine engine, or a GPU compute backend.

Before writing your own, check whether an existing scheduler fits:

Your NeedExisting Solution
Debugging / unit testing / prototypingrun_loop
General async tasksstatic_thread_pool
CPU parallel computationparallel_scheduler
Execute on the current threadinline_scheduler

If none of the four above satisfy your requirements, read on.

The Hard Threshold for Schedulers

For a type to qualify as a scheduler, it must satisfy three requirements:

(1) Destructible, copy-constructible, equality-comparable — schedulers are lightweight value types you can copy and pass around freely (2) schedule(my_scheduler) returns a sender — this is the sole entry point for user-scheduler interaction (3) Supports get_forward_progress_guarantee query — tells the runtime “how much concurrency my scheduler guarantees”

That’s a short list. As long as your type meets these three, it’s a scheduler.

Minimal Implementation: inline_scheduler

inline_scheduler is under 30 lines of code. Its effect: “execute immediately on the current thread” — start() synchronously calls set_value. Its value isn’t just “it works” but also “great for testing” — no threads needed, ideal for unit tests:

cpp
class inline_scheduler {
 public:
  friend forward_progress_guarantee tag_invoke(
    tag_t<get_forward_progress_guarantee>,
    const inline_scheduler&) noexcept {
    return forward_progress_guarantee::concurrent;
  }
  bool operator==(const inline_scheduler&) const = default;
};

// The sender returned by schedule
struct inline_schedule_sender {
  using sender_concept = sender_tag;
  using completion_signatures = completion_signatures<set_value_t()>;

  template <receiver R>
  struct op {
    R rcvr_;
    void start() & noexcept {
      set_value(static_cast<R&&>(rcvr_));  // Complete immediately on the current thread
    }
  };

  template <receiver R>
  auto connect(R r) && -> op<R> {
    return {static_cast<R&&>(r)};
  }
};

inline_schedule_sender tag_invoke(tag_t<schedule>, const inline_scheduler&) {
  return {};
}

Above clearly shows the three parts of a scheduler implementation — the scheduler class itself (provides queries), the schedule sender (provides connect / start), and the tag_invoke bridging them together.

Advanced: trampoline_scheduler — Deferred Execution

trampoline_scheduler demonstrates the “don’t execute immediately” pattern — instead of calling set_value directly, it enqueues work to be executed after the current call stack unwinds:

cpp
struct trampoline_schedule_sender {
  template <receiver R>
  struct op {
    R rcvr_;
    void start() & noexcept {
      trampoline_queue::enqueue([this] {
        set_value(static_cast<R&&>(rcvr_));
      });
    }
  };
  // connect is the same as inline_scheduler…
};

This pattern is useful for avoiding stack overflow — recursive calls can make the call stack grow unboundedly; trampolining turns each call into a queue entry so the stack is always one level deep.

Production-Grade: A Thread Pool Scheduler Skeleton

cpp
class thread_pool {
  std::vector<std::thread> threads_;
  std::queue<std::function<void()>> queue_;
  std::mutex mtx_;
  std::condition_variable cv_;
  bool done_ = false;

public:
  class scheduler {
    thread_pool* pool_;
  public:
    explicit scheduler(thread_pool* pool) : pool_(pool) {}

    friend forward_progress_guarantee tag_invoke(
      tag_t<get_forward_progress_guarantee>,
      const scheduler&) noexcept {
      return forward_progress_guarantee::parallel;  // Multi-threaded → parallel
    }
    bool operator==(const scheduler& other) const = default;
  };

  scheduler get_scheduler() { return {this}; }

  struct schedule_sender {
    thread_pool* pool_;
    using sender_concept = sender_tag;
    using completion_signatures = completion_signatures<set_value_t()>;

    template <receiver R>
    struct op {
      thread_pool* pool_;
      R rcvr_;
      void start() & noexcept {
        pool_->enqueue([this] { set_value(static_cast<R&&>(rcvr_)); });
      }
    };

    template <receiver R>
    auto connect(R r) && -> op<R> {
      return {pool_, static_cast<R&&>(r)};
    }
  };

  friend schedule_sender tag_invoke(tag_t<schedule>, const scheduler& sch) {
    return {sch.pool_};
  }
};

Limitations

  1. Correctness and thread safety are entirely your responsibility — the standard library won’t verify whether your scheduler actually delivers its claimed forward_progress_guarantee. If you claim parallel but only one thread ever executes, the compiler won’t complain.
  2. Domain coordination — if your scheduler needs to customize bulk or when_all behavior, you also need a custom domain (see Chapter 26), associating the domain with your scheduler via the get_domain query.
  3. operator== semantics must be correct — two schedulers comparing equal means they represent the same execution resource. If every scheduler instance is independent even when pointing to the same thread pool, scheduler affinity optimizations (like let_value) may not trigger.

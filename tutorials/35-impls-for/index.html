35 · The impls-for System

Standard section: [exec.snd.expos] stdexec reference: include/stdexec/__detail/

Overview

impls-for is a set of exposition-only entities in the std::execution standard, used to simplify the implementation of sender algorithms. Many algorithms within stdexec are built on this system.

Exposition-Only Entities in the Standard

The standard [exec.snd.expos] defines the following exposition-only types:

cpp
namespace std::execution {
  // basic-sender: generic sender wrapper
  template<class Tag, class Data, class... Child>
  struct basic-sender;

  // basic-operation: generic operation state
  template<class Sender, class Receiver>
  struct basic-operation;

  // basic-receiver: generic receiver wrapper
  template<class Operation>
  struct basic-receiver;

  // impls-for: algorithm-specific specialization interface
  template<class Tag>
    struct impls-for;

  // default-impls: default algorithm implementations
  struct default-impls;
}

default-impls

default-impls provides default implementations for sender algorithms:

cpp
struct default-impls {
  // Get completion signatures
  template<class Sender, class... Env>
  static constexpr auto get_completion_signatures() -> // ...

  // Build operation state from sender and receiver
  template<class Sender, class Receiver>
  static auto make_operation(Sender&&, Receiver&&) -> // ...

  // Start the operation state
  template<class Operation>
  static void start(Operation&) noexcept;

  // Get sender attributes
  template<class Sender>
  static auto get_env(Sender&&) noexcept -> // ...;
};

impls-for Specialization

Algorithms provide custom behavior by specializing impls-for:

cpp
template<>
struct impls-for<then_t> : default-impls {
  // Custom completion signature computation for then
  template<class Sender, class... Env>
  static constexpr auto get_completion_signatures() {
    // ...
  }

  // Custom start behavior for then
  template<class Operation>
  static void start(Operation& op) noexcept {
    // Start child operation, invoke fn in the internal receiver
  }
};

basic-sender

cpp
template<class Tag, class Data, class... Child>
struct basic-sender {
  // Automatically generates sender interface
  // connect() → basic-operation
  // get_env() → ...
  // Completion signatures → delegated to impls-for<Tag>
};

A simplified version of then written using basic-sender:

cpp
// then's "data" part: holds the function object
template <class Fn>
struct then_data {
  Fn fn_;
};

// then's impls specialization
template <>
struct impls_for<then_t> : default_impls {
  // Custom completion signatures
  template <class Self, class... Env>
  static constexpr auto get_completion_signatures() {
    using Sigs = completion_signatures_of_t<__child_of<Self, 0>, Env...>;
    return __transform_completion_signatures(Sigs, /* apply fn */);
  }

  // Custom start
  template <class Op>
  static void start(Op& op) noexcept {
    // op.__rcvr_: external receiver
    // op.__child_ops_[0]: child operation
    start_child(op.__child_ops_[0], then_receiver{&op});
  }
};

// Final then implementation ≈ basic_sender<then_t, then_data<Fn>, ChildSndr>

Usage Inside stdexec

Algorithms under stdexec’s __detail/ directory make heavy use of this system:

__then.hpp
__let.hpp
__when_all.hpp
__bulk.hpp

Each algorithm roughly follows this structure:

cpp
namespace stdexec {
  // 1. Define completion signature transformation
  template <class Fn>
  struct __then_value {
    template <class... Args>
    using __f = completion_signatures<set_value_t(__decay_t<std::invoke_result_t<Fn, Args...>>)>;
  };

  // 2. Specialize impls_for
  template <>
    struct impls_for<then_t> : __detail::__default_impls {
      // ...
    };

  // 3. User-facing API
  struct then_t {
    template <sender Sndr, class Fn>
    auto operator()(Sndr sndr, Fn fn) const -> sender auto {
      return basic_sender<then_t, then_data<Fn>, Sndr>{
        then_data<Fn>{std::move(fn)},
        std::move(sndr)
      };
    }
  };
}

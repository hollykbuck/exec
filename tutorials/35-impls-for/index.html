35 · The impls-for System — For Library Authors

If you only use sender/receiver for application code, please skip this chapter. This chapter is for library authors contributing new algorithms to the C++26 std::execution ecosystem.

Why impls-for?

In Chapter 32 we hand-rolled a then — wrote the sender class, receiver class, operation state class, completion signature transformation, the whole set. Writing one algorithm is fine. But writing ten? You’ll find yourself copy-pasting the same boilerplate for every algorithm:

  • Define a sender type → nearly identical to the last one
  • Define an internal receiver → only the callback inside set_value differs
  • Define an operation state → only member types differ
  • Write connect() → boilerplate

impls-for exists to eliminate this boilerplate. Its philosophy:

basic-sender handles the “common parts” (connect / get_env / operator|) impls-for handles only the “unique parts” (completion signature transformation / start behavior)

Analogy: From Class Components to Hooks

If you’re familiar with React’s evolution:

Class ComponentFunctional + Hooks
You writeconstructor / render / componentDidMount — full lifecycleOnly the useState / useEffect you care about
BoilerplateLots (every component repeats fixed methods)Little (focus on state logic only)

impls-for is to hand-rolling senders what hooks are to class components — it eliminates roughly 80% of the template code.

The Core of basic-sender

cpp
template <class Tag, class Data, class... Child>
struct basic_sender {
  // connect() → auto-generates basic_operation
  // get_env() → auto-generates (queries basic properties)
  // get_completion_signatures → delegates to impls-for<Tag>
};

You only need to provide three things:

  • Tag — the algorithm tag (e.g., then_t), used to locate the corresponding impls-for specialization
  • Data — algorithm-specific data (e.g., then’s callback function f)
  • Children — child senders (the pipeline’s input senders)

The Skeleton Templates

Under the hood, basic-sender delegates to four exposition-only templates:

product-type — The Sender’s Storage

basic-sender inherits from product-type<Tag, Data, Child...>:

cpp
template <class T0, class T1, ..., class Tn>
struct product-type {
  T0 t0;
  T1 t1;
  ...
  Tn tn;

  template <size_t I, class Self>
  constexpr decltype(auto) get(this Self&& self) noexcept;

  template <class Self, class Fn>
  constexpr decltype(auto) apply(this Self&& self, Fn&& fn) noexcept;
};

product-type is a tuple-like aggregate that supports structured binding. Every sender can be decomposed as:

cpp
auto&& [tag, data, ...children] = sndr;
// tag    → the algorithm type (e.g., then_t)
// data   → algorithm-specific data (e.g., the callable)
// children → child senders

emplace-from — Non-Movable Construction

Used internally to emplace non-movable types into tuples and variants:

cpp
template <callable Fun>
  requires is_nothrow_move_constructible_v<Fun>
struct emplace-from {
  Fun fun;
  using type = call-result-t<Fun>;

  constexpr operator type() && noexcept(nothrow-callable<Fun>) {
    return std::move(fun)();
  }
};

This is used in async-mutex style primitives where a completion handler is not movable.

basic-state — The Operation State’s Data

basic-state<Sndr, Rcvr> stores the receiver and algorithm-specific state:

cpp
template <class Sndr, class Rcvr>
struct basic-state {
  basic-state(Sndr&& sndr, Rcvr&& rcvr) noexcept(...)
    : rcvr(std::move(rcvr))
    , state(impls-for<tag_of_t<Sndr>>::get-state(std::forward<Sndr>(sndr), rcvr)) { }

  Rcvr rcvr;
  state-type<Sndr, Rcvr> state;
};

The state member is computed by the algorithm’s get-state hook. For then, this stores the callable; for continues_on, it stores a variant for captured results plus the scheduled operation.

basic-receiver — Completion Dispatch

basic-receiver<Sndr, Rcvr, Index> forwards completion signals to impls-for<Tag>::complete:

cpp
template <class Sndr, class Rcvr, class Index>
struct basic-receiver {
  using receiver_concept = receiver_tag;
  using tag-t = tag_of_t<Sndr>;
  using state-t = state-type<Sndr, Rcvr>;
  static constexpr const auto& complete = impls-for<tag-t>::complete;

  void set_value(Args&&... args) && noexcept {
    complete(Index(), op->state, op->rcvr, set_value_t(), std::forward<Args>(args)...);
  }

  void set_error(Error&& err) && noexcept {
    complete(Index(), op->state, op->rcvr, set_error_t(), std::forward<Error>(err));
  }

  void set_stopped() && noexcept {
    complete(Index(), op->state, op->rcvr, set_stopped_t());
  }

  auto get_env() const noexcept -> env-type<Index, Sndr, Rcvr> {
    return impls-for<tag-t>::get-env(Index(), op->state, op->rcvr);
  }

  basic-state<Sndr, Rcvr>* op;
};

Every completion handler calls complete(Index(), state, rcvr, tag, args...). The Index parameter identifies which child sender completed — this is how when_all knows which branch to mark as done.

basic-operation — The Complete Operation State

basic-operation<Sndr, Rcvr> inherits from basic-state and adds child operation connections:

cpp
template <class Sndr, class Rcvr>
struct basic-operation : basic-state<Sndr, Rcvr> {
  using operation_state_concept = operation_state_tag;
  using tag-t = tag_of_t<Sndr>;

  connect-all-result<Sndr, Rcvr> inner-ops;

  basic-operation(Sndr&& sndr, Rcvr&& rcvr) noexcept(...)
    : basic-state<Sndr, Rcvr>(std::forward<Sndr>(sndr), std::move(rcvr)),
      inner-ops(connect-all(this, std::forward<Sndr>(sndr), indices-for<Sndr>()))
  {}

  void start() & noexcept {
    auto& [...ops] = inner-ops;
    impls-for<tag-t>::start(this->state, this->rcvr, ops...);
  }
};

connect-all recursively connects child senders to basic-receiver instances. The start method delegates to impls-for<Tag>::start, which is the only part the algorithm author customizes.

The Flow

basic-sender<Tag, Data, Children...>
  │
  │  sndr.connect(rcvr)
  ▼
basic-operation<Sndr, Rcvr>
  │  [inherits basic-state → stores rcvr + algorithm state]
  │  [connect-all → connects children to basic-receiver instances]
  │
  │  op.start()
  ▼
impls-for<Tag>::start(state, rcvr, ops...)
  │  [algorithm-specific: start children, set up timers, etc.]
  ▼
basic-receiver::set_value / set_error / set_stopped
  │  [dispatches to impls-for<Tag>::complete]
  ▼
impls-for<Tag>::complete(Index, state, rcvr, tag, args...)
  │  [algorithm-specific: combine results, forward to rcvr]
  ▼
Downstream receiver notified

## Basic Usage

Rewriting `then` with `impls-for`:

```cpp
// 1. Define the data part (the function object)
template <class Fn>
struct then_data {
  Fn fn_;
};

// 2. Specialize impls-for (only write then's unique logic)
template <>
struct impls_for<then_t> : default_impls {
  // Compile-time completion signature transformation
  template <class Self, class... Env>
  static constexpr auto get_completion_signatures() {
    return __transform_completion_signatures(
      completion_signatures_of_t<child_of<Self, 0>, Env...>,
      _then_value_transform<data_of<Self>>   // Only write how you change the value
    );
  }

  // start(): create internal receiver and start child operation
  template <class Op>
  static void start(Op& op) noexcept {
    start_child(op.child_ops_[0], then_receiver{&op});
  }
};

Above, impls-for<then_t> only overrides two things — get_completion_signatures (changed) and start (changed). Everything else — connect, get_env, operator| — is auto-generated by basic_sender.

What default-impls Provides

cpp
struct default_impls {
  // Auto-fetch completion signatures
  template <class Sender, class... Env>
  static constexpr auto get_completion_signatures() -> /* ... */;

  // Auto-build operation state from sender and receiver
  template <class Sender, class Receiver>
  static auto make_operation(Sender&&, Receiver&&) -> /* ... */;

  // Default start behavior
  template <class Operation>
  static void start(Operation&) noexcept;

  // Get sender attributes
  template <class Sender>
  static auto get_env(Sender&&) noexcept -> /* ... */;
};

You only override what you’re unhappy with. Everything you don’t override takes the default.

Actual Usage Inside stdexec

Under stdexec’s __detail/ directory, nearly every algorithm uses impls-for:

__then.hpp       → impls_for<then_t>
__let.hpp        → impls_for<let_t>
__when_all.hpp   → impls_for<when_all_t>
__bulk.hpp       → impls_for<bulk_t>

Each algorithm follows the same pattern:

  1. Define completion signature transformation (how value changes, how error changes)
  2. Specialize impls-for
  3. User-facing API calls basic_sender<Tag, Data, Children...>

36 · Linux io_uring — True Async File I/O

What epoll Can’t Handle

On Linux, epoll is great for high-performance network services — it can monitor thousands of sockets simultaneously and notify you when any has data. But it has a fundamental blind spot: epoll only manages network sockets, not disk files.

When you call read() on a file and the data isn’t in the page cache, the kernel has to fetch it from disk — a process that can take milliseconds or even tens of milliseconds. In the traditional synchronous I/O model, the thread calling read() just sits there waiting.

The common “solution” is to use a thread pool to fake async file I/O — spawn a thread to read the file, then notify the main thread when done:

cpp
// Traditional: use a thread pool to 'fake' async file I/O
std::thread([] {
  auto data = read_file("large.bin");  // Blocks here
  notify_main_thread(data);            // Notify the main thread
}).detach();

The problems: thread switching has overhead, and you can’t spawn unlimited threads. If your server has 1000 concurrent requests all needing to read files, you can’t spawn 1000 threads.

io_uring, introduced in Linux 5.1, solves this at the root — it makes file I/O a truly asynchronous kernel operation. No extra threads needed, no callers blocked.

Platform restriction: io_uring is a Linux 5.1+ feature. It doesn’t work on macOS, Windows, or older Linux. Also io_uring is a rapidly evolving kernel feature. Different kernel versions may support different operation types and have different performance characteristics.

How io_uring Works

A traditional I/O call is like a “synchronous phone call” — you dial, wait for the other side to pick up, talk, hang up. Every step forces you to wait:

Traditional read():  app → syscall → kernel processing → wait for disk → return data → app continues (a 'phone call')

io_uring is more like “text messaging” — you write your request into a ring buffer (send it), the kernel processes it in the background, and marks the result in another ring buffer when done (you got a reply). Between sending and receiving, you’re free to do anything else:

io_uring:      app → write to SQ (Submission Queue) → go do other things
               kernel → process I/O → write to CQ (Completion Queue)
               app → check CQ → read the result

The core data structures of io_uring are two ring buffers shared between user space and kernel space:

  • SQ (Submission Queue) — the app writes I/O requests here and notifies the kernel to pick them up
  • CQ (Completion Queue) — the kernel writes I/O results here; the app polls or waits for notifications

Because these two queues live in shared memory, most operations don’t need a syscall — you write to SQ in user space, the kernel reads from the other side, zero-copy all the way.

stdexec’s io_uring_context

exec::io_uring_context wraps an entire io_uring instance as a scheduler:

cpp
namespace exec {
  class io_uring_context {
  public:
    class scheduler;
    scheduler get_scheduler() noexcept;
    void run_until_stopped();  // Start the event loop (blocks until stopped)
    void run_until_empty();    // Run until no more work is pending
    void request_stop();       // Request the event loop to stop
    void finish();             // Break out of run loop without stopping
    void reset();              // Reset the context for reuse
  };
}

The usage pattern is similar to run_loop — create a context, get its scheduler, submit work, and drive the event loop:

cpp
exec::io_uring_context ctx;
auto io_sched = ctx.get_scheduler();

std::thread worker([&] { ctx.run_until_stopped(); });  // Background thread drives the event loop

// Submit I/O work
sync_wait(
  schedule(io_sched)
    | then([] { printf("running on io_uring\n"); })
);

ctx.request_stop();
worker.join();

Here’s the timer example from examples/io_uring.cpp:

cpp
exec::io_uring_context io_ctx;
auto io_sched = io_ctx.get_scheduler();

// Execute after a 100ms delay
sync_wait(
  schedule_at(io_sched, 100ms)
    | then([] { printf("timer fired\n"); })
);

// Multiple simultaneous timers
sync_wait(when_all(
  schedule_at(io_sched, 50ms)  | then(t1),
  schedule_at(io_sched, 100ms) | then(t2),
  schedule_at(io_sched, 150ms) | then(t3)
));

Combining with a Thread Pool

A common high-performance I/O architecture combines io_uring with a CPU thread pool:

cpp
exec::io_uring_context io_ctx;
exec::static_thread_pool cpu_pool{4};

// I/O layer on io_uring, compute layer on thread pool
starts_on(io_ctx.get_scheduler(),
  read_file("data.bin")                        // IO: async read via io_uring
    | continues_on(cpu_pool.get_scheduler())    // Switch to CPU thread pool
    | then([](buffer buf) { return process(buf); })  // Compute: process data
    | continues_on(io_ctx.get_scheduler())      // Switch back to io_uring
    | then([](result r) { return write_response(r); })  // IO: write response
);

Above, starts_on launches the whole pipeline on io_uring, reading the file (async). After the read completes, continues_on switches to the CPU thread pool for computation. Once processing is done, it switches back to io_uring to write the response. Each switch is handled automatically by the scheduler affinity mechanism — no manual thread management.

This “I/O thread + CPU thread pool” separation is the classic pattern for high-performance network services — io_uring focuses on non-blocking I/O, the thread pool on CPU-intensive computation.

Lifecycle

cpp
exec::io_uring_context ctx;
auto sched = ctx.get_scheduler();

std::thread worker([&] { ctx.run_until_stopped(); });

// Submit work…
spawn(schedule(sched) | then(work));

// Graceful shutdown
ctx.request_stop();  // 1. Request the event loop to stop
worker.join();       // 2. Wait for run_until_stopped() to return
ctx.reset();         // 3. Reset the context (if you want to reuse it)

The difference between request_stop() and finish(): request_stop() initiates a stop request, still giving in-flight operations some time to complete. After calling reset(), the context can be reused — call run_until_stopped() again to start a fresh event loop.

Common Pitfall: Forgetting to Stop Before Join

run_until_stopped() is a persistent event loop — unlike run_until_empty(), it does not exit on its own. The wakeup operation inside the context perpetually re-queues itself to keep the loop alive. Calling worker.join() without first calling request_stop() will deadlock:

cpp
std::thread worker{[&] { ctx.run_until_stopped(); }};
sync_wait(schedule(sched) | then(work));
worker.join();  // DEADLOCK — run_until_stopped() never exits!

The correct pattern is to ensure request_stop() is called before joining. The safest way is an inner scope with a scope guard:

cpp
std::thread worker{[&] { ctx.run_until_stopped(); }};
{
  exec::scope_guard stop_{[&]() noexcept { ctx.request_stop(); }};
  //                          ^^^^^^^^^ runs on scope exit
  sync_wait(schedule(sched) | then(work));
  sync_wait(schedule_after(sched, 100ms) | then(work2));
  // scope_guard fires → request_stop() → worker stops
}
worker.join();  // safe: worker has already stopped

Or call request_stop() explicitly before join:

cpp
ctx.request_stop();
worker.join();

For one-shot work where you don’t need a persistent event loop, use run_until_empty() instead, typically driven by just() | then([&] { ctx.run_until_empty(); }) combined with when_all:

cpp
sync_wait(when_all(
  schedule(sched) | then(work),
  just() | then([&] { ctx.run_until_empty(); })
));  // No need to request_stop — context stops itself when empty

Notice

io_uring_context is designed to be exclusively driven by one thread (the caller of run_until_stopped() or run_until_empty()). For multi-threaded I/O handling, you need multiple context instances. Context cannot be shared across threads.

40 · Boost.Beast HTTP Server — Building a Network Service with Sender Pipeline

On this page

Standard Section: — stdexec Reference: exec/asio/use_sender.hpp, exec/asio/asio_thread_pool.hpp Related Tests: tests/test_beast_server.cpp

Overview

This section demonstrates how to combine Boost.Beast (HTTP/WebSocket library) with stdexec’s sender/receiver model to build a fully sender-pipeline-based HTTP server.

Core idea:

graph TD
  A["ASIO io_context (I/O execution resource)"]
  A --> B["get_scheduler()"]
  B --> C["stdexec Scheduler"]
  C --> D["exec::asio::use_sender"]
  D --> E["ASIO async_* operations → stdexec Sender"]
  E --> F["co_await / then / let_value"]
  F --> G["HTTP request processing pipeline"]

Step 1: Turn ASIO Operations into Senders with use_sender

exec::asio::use_sender is an ASIO completion token that makes any ASIO async operation return a stdexec sender:

cpp
namespace beast = boost::beast;
namespace http = beast::http;
namespace net = boost::asio;
using tcp = net::ip::tcp;
namespace ex = stdexec;
namespace asioex = exec::asio;

// Traditional ASIO style
void async_read(tcp::socket& sock, net::mutable_buffer buf,
                std::function<void(error_code, size_t)> cb);

// Using use_sender → returns sender
// net::async_read(sock, buf, asioex::use_sender)
// → sender<set_value_t(error_code, size_t)>

Wrapping socket accept as a sender

cpp
// Corresponding test: test_beast_server.cpp → "beast server accepts and responds"
ex::sender auto async_accept(tcp::acceptor& acceptor) {
  return net::async_accept(
    acceptor,                           // acceptor object
    tcp::socket{acceptor.get_executor()}, // pass empty socket
    asioex::use_sender                  // ← key: returns sender
  );
}

Calling async_accept returns a sender that emits set_value(error_code, socket) when a new connection arrives.

Step 2: HTTP Request Handling Coroutine

Use stdexec’s task<T> coroutine to write request handlers. Since task<T> can co_await any sender, ASIO’s use_sender senders work directly in coroutines:

cpp
// Corresponding test: test_beast_server.cpp
ex::task<http::response<http::string_body>> handle_request(
  http::request<http::string_body> req)
{
  auto path = std::string{req.target()};

  if (req.method() == http::verb::get && path == "/hello") {
    http::response<http::string_body> res{http::status::ok, req.version()};
    res.set(http::field::server, "stdexec-beast");
    res.set(http::field::content_type, "text/plain");
    res.body() = "Hello from stdexec + Boost.Beast!\n";
    res.prepare_payload();
    co_return res;
  }

  // 404
  http::response<http::string_body> res{http::status::not_found, req.version()};
  res.set(http::field::server, "stdexec-beast");
  res.set(http::field::content_type, "text/plain");
  res.body() = "Not found\n";
  res.prepare_payload();
  co_return res;
}

Step 3: Connection Lifecycle Management

Complete connection handling — read request → process → send response, all chained via sender pipeline:

cpp
// Corresponding test: test_beast_server.cpp
ex::task<void> read_and_echo(beast::tcp_stream& stream) {
  std::array<char, 2048> buf;

  // use_sender converts async_read_some to sender
  // On success: set_value(bytes_transferred), error_code is stripped
  // On failure: set_error(system_error), propagated as exception in task
  std::size_t n = co_await stream.socket().async_read_some(
    net::buffer(buf), asioex::use_sender);

  // Echo received data
  co_await net::async_write(
    stream.socket(), net::buffer(buf, n), asioex::use_sender);

  beast::error_code shut_ec;
  stream.socket().shutdown(tcp::socket::shutdown_send, shut_ec);
}

Pure Pipeline Version (Without Coroutines)

If you prefer sender pipelines over coroutines, you can chain with let_value:

cpp
// Corresponding test: test_beast_server.cpp
ex::sender auto handle_connection_pipeline(tcp::socket raw_sock) {
  auto stream = std::make_shared<beast::tcp_stream>(std::move(raw_sock));
  auto buffer = std::make_shared<beast::flat_buffer>();

  return http::async_read(*stream, *buffer,
           http::request<http::string_body>{}, asioex::use_sender)
    | ex::let_value([stream, buffer](http::request<http::string_body> req) {
        // Generate response
        auto res = make_response(std::move(req));
        return http::async_write(*stream, res, asioex::use_sender);
      })
    | ex::then([](auto...) {
        // Done
      });
}

Step 4: Accept Loop

Use async_scope to manage all connection lifecycles, ensuring structured concurrency:

cpp
// Corresponding test: test_beast_server.cpp → "beast server accept loop"
ex::task<void> run_server(tcp::endpoint endpoint) {
  net::io_context ctx{1};                        // ASIO execution resource
  tcp::acceptor acceptor{ctx, endpoint};

  exec::async_scope scope;                       // Structured concurrency management
  int64_t connection_id = 0;

  while (!ctx.stopped()) {
    auto [accept_ec, peer_sock] = co_await net::async_accept(
      acceptor, tcp::socket{ctx}, asioex::use_sender);

    if (accept_ec) {
      break;
    }

    // Each connection spawned as an independent task
    scope.spawn(
      handle_connection(std::move(peer_sock))
        | ex::upon_error([](std::exception_ptr e) {
            // Log connection error but don't affect the entire server
          })
    );
  }

  // Wait for all connections to finish
  co_await scope.on_empty();
}

Step 5: Start the Server

Run the server on an ASIO thread pool, with sync_wait blocking in main:

cpp
// Corresponding test: test_beast_server.cpp → "beast server start and stop"
int main() {
  // ASIO thread pool (also a stdexec scheduler)
  asioex::asio_thread_pool pool{4};
  auto sch = pool.get_scheduler();

  // Start server on ASIO thread pool
  ex::sync_wait(
    ex::starts_on(sch,
      run_server(tcp::endpoint{tcp::v4(), 8080})
    )
  );
}

Complete Server

Below is a complete HTTP server based on use_sender, driven by sync_wait for connection handling:

cpp
// Corresponding test: test_beast_server.cpp
#include <exec/asio/use_sender.hpp>
#include <stdexec/execution.hpp>

#include <boost/beast/core.hpp>
#include <boost/beast/http.hpp>
#include <boost/asio.hpp>

namespace beast = boost::beast;
namespace http = beast::http;
namespace net = boost::asio;
using tcp = net::ip::tcp;
namespace ex = stdexec;
namespace asioex = exec::asio;

// HTTP request handler — routing dispatch
ex::task<http::response<http::string_body>> handle_request(
  http::request<http::string_body> req)
{
  http::response<http::string_body> res;
  res.set(http::field::server, "stdexec-test/1.0");
  res.keep_alive(false);

  if (req.method() == http::verb::get && req.target() == "/api/ping") {
    res.result(http::status::ok);
    res.set(http::field::content_type, "application/json");
    res.body() = R"({"pong":true})";
  } else if (req.method() == http::verb::get && req.target() == "/api/echo") {
    res.result(http::status::ok);
    res.set(http::field::content_type, "text/plain");
    res.body() = req.target();
  } else {
    res.result(http::status::not_found);
    res.set(http::field::content_type, "application/json");
    res.body() = R"({"error":"not found"})";
  }

  res.prepare_payload();
  co_return res;
}

// Wrapper for async_read_some: reads raw bytes
// use_sender on success only returns bytes_transferred,
// error_code is stripped (on failure, converted to set_error / exception)
ex::task<std::string> read_some(beast::tcp_stream& stream) {
  std::array<char, 2048> buf;
  std::size_t n = co_await stream.socket().async_read_some(
    net::buffer(buf), asioex::use_sender);
  co_return std::string{buf.data(), n};
}

// Wrapper for async_write: sends raw bytes
ex::task<void> write_all(beast::tcp_stream& stream, std::string_view data) {
  co_await net::async_write(
    stream.socket(), net::buffer(data), asioex::use_sender);
  beast::error_code shut_ec;
  stream.socket().shutdown(tcp::socket::shutdown_send, shut_ec);
}

// Connection handling: read request → process → write response
ex::task<void> handle_session(beast::tcp_stream stream) {
  auto req_data = co_await read_some(stream);
  // Simplified: assumes one read contains the full request line (production code should loop)
  http::request_parser<http::string_body> parser;
  parser.eager(true);
  beast::error_code pec;
  parser.put(net::buffer(req_data), pec);
  if (pec) co_return;

  auto req = parser.release();
  auto res = co_await handle_request(std::move(req));
  co_await write_all(stream, http::serialize(res));
}

// Server loop
void run_server(net::io_context& ctx, tcp::acceptor& acceptor) {
  while (!ctx.stopped()) {
    beast::error_code ec;
    tcp::socket sock = acceptor.accept(ec);
    if (ec) break;

    ex::sync_wait(
      handle_session(beast::tcp_stream{std::move(sock)}));
  }
}

int main() {
  net::io_context ctx;
  tcp::acceptor acceptor{ctx, tcp::endpoint{tcp::v4(), 8080}};

  std::jthread server_thread([&] { run_server(ctx, acceptor); });
  server_thread.join();
}

Design Highlights

1. ASIO as Execution Resource

exec::asio::asio_thread_pool wraps net::io_context, serving both as a stdexec scheduler and an ASIO execution context:

cpp
asioex::asio_thread_pool pool{4};    // 4 threads
auto sch = pool.get_scheduler();      // stdexec scheduler
net::io_context& ctx = pool.context(); // ASIO io_context

2. use_sender Bridge Pattern

exec::asio::use_sender converts ASIO async operations (async_accept, async_read, async_write) into stdexec senders. Conversion rules:

ASIO Operation SignatureCorresponding Sender Completion Signature
async_read_some(sock, buf, token)set_value_t(size_t) — on success, error_code is stripped; on failure, converted to set_error
async_write(sock, buf, token)set_value_t(size_t) — same as above
async_accept(acceptor, sock, token)set_value_t(error_code, socket) — accept does not strip error_code

3. Structured Concurrency

Use async_scope to manage all connection tasks:

  • Each connection starts via scope.spawn()
  • Before server exit, scope.on_empty() waits for all connections to complete
  • Exceptions are caught via upon_error, isolated from other connections

4. Allocator Support

use_sender follows stdexec’s environment mechanism, reading the allocator from the receiver environment via ASIO’s associated_allocator trait.

Principle

The use_sender completion token specializes associated_allocator in completion_token.hpp:

cpp
// stdexec exec/asio/completion_token.hpp
template <class Receiver>
struct associated_allocator<completion_handler<Receiver>, ...> {
  using type = decltype(get_allocator(get_env(declval<Receiver>())));
  static type get(const completion_handler<Receiver>& h) noexcept {
    return get_allocator(get_env(h.state().r_));
  }
};

This means when ASIO needs to allocate memory for a handler, it calls get_allocator(get_env(receiver)) to obtain the allocator.

Injecting Custom Allocators

Inject via write_env + prop{get_allocator, my_allocator}:

cpp
// Corresponding test: test_beast_server.cpp → "use_sender propagates allocator"
template <class T>
struct tracker_allocator : std::allocator<T> {
  inline static std::atomic<int> alloc_count{0};

  T* allocate(std::size_t n) {
    ++alloc_count;
    return std::allocator<T>::allocate(n);
  }
};

// Inject into pipeline
auto snd = ex::write_env(
  some_sender,
  ex::prop{ex::get_allocator, tracker_allocator<char>{}}
);

Reading the Allocator in a Coroutine

task<T> can read the allocator from the environment via read_env(get_allocator):

cpp
// Corresponding test: test_beast_server.cpp → "custom allocator propagates through task environment"
ex::task<void> check_allocator() {
  auto alloc = co_await ex::read_env(ex::get_allocator);
  // alloc is the allocator from the current receiver environment
}

5. Go-style Value Context

stdexec’s environment mechanism naturally supports request-scoped value passing similar to Go’s context.Context. Core mapping:

Go contextstdexec Environment
context.WithValue(ctx, key, val)write_env(sndr, env{prop{key, val}})
ctx.Value(key)read_env(key)
Automatically propagated along call chainAutomatically propagated downward in pipeline
Custom key typeCustom query CPO type

Defining a Query CPO (Analogous to Go’s context key)

cpp
// Corresponding test: test_beast_server.cpp
struct request_id_t : ex::__query<request_id_t> {
  using ex::__query<request_id_t>::operator();
  static consteval auto query(ex::forwarding_query_t) noexcept -> bool {
    return true;
  }
};
inline constexpr request_id_t request_id{};

Writing Context (Analogous to Go: context.WithValue)

cpp
// Corresponding test: test_beast_server.cpp → "value context: write_env injects custom values"
auto ctx_env = ex::env{
  ex::prop{request_id, std::string{"req-42"}},
  ex::prop{user_role, std::string{"admin"}},
  ex::prop{trace_id, std::string{"trace-xyz"}}
};

auto snd = ex::write_env(
  // Pipeline internals can read these via read_env
  ex::when_all(
    ex::read_env(request_id),
    ex::read_env(user_role),
    ex::read_env(trace_id)
  ) | ex::then([](std::string rid, std::string role, std::string tid) {
      // Use context values
      return make_response(rid, role, tid);
    }),
  std::move(ctx_env)    // Inject environment, analogous to context.WithValue
);

Reading Context (Analogous to Go: ctx.Value)

cpp
// Corresponding test: test_beast_server.cpp → "value context: propagates through let_value"
// Read in let_value/pipeline:
ex::let_value([](int) {
  return ex::read_env(request_id);  // ← analogous to ctx.Value("request_id")
})

Nested Overrides

Inner write_env (closest to read_env) overrides outer, analogous to Go’s nearest-write-wins:

cpp
ex::write_env(                                      // Outer
  ex::write_env(                                     // Inner
    ex::read_env(request_id),                        // → reads "inner-id"
    ex::env{ex::prop{request_id, "inner-id"}}
  ),
  ex::env{ex::prop{request_id, "outer-id"}}
);

Integration in Beast Server

cpp
// Inject trace_id / request_id per connection
ex::task<void> handle_session_with_context(beast::tcp_stream stream) {
  auto ctx_env = ex::env{
    ex::prop{trace_id, generate_trace_id()},
    ex::prop{request_id, generate_request_id()}
  };

  auto req_data = co_await read_some(stream);
  // Processing environment carries trace_id / request_id
  co_await ex::write_env(
    process_request(std::move(req_data)),
    std::move(ctx_env)
  );
}

6. Timeout and Cancellation

Approach 1: Cancel Pending Operations via socket.cancel()

When an async operation initiated by use_sender is still pending, it can be cancelled via socket.cancel():

cpp
// Corresponding test: test_beast_server.cpp → "socket.cancel() stops pending async_read_some"
std::array<char, 256> buf;
auto read_sender = sock.async_read_some(net::buffer(buf), asioex::use_sender);

// While sync_wait holds read_sender on another thread:
sock.cancel();  // → async_read_some is cancelled → use_sender emits set_error

// sync_wait returns nullopt (set_error converted to exception, sync_wait swallows it)

Key points:

  • socket.cancel() cancels all pending async operations on that socket
  • use_sender converts the operation_aborted error code to set_error
  • sync_wait converts set_error to an exception, returning nullopt to the caller

Approach 2: Timeout via inplace_stop_source + steady_timer

Combine asio::steady_timer with inplace_stop_source for timeout-based cancellation:

cpp
// Corresponding test: test_beast_server.cpp → "socket.cancel() stops pending async_read_some"
// Background thread: initiate async read with use_sender
std::thread reader([&] {
  auto result = ex::sync_wait(std::move(read_sender));
  // Timeout → socket.cancel() → set_error → nullopt
  REQUIRE_FALSE(result.has_value());
});

// Main thread: cancel after timeout
std::this_thread::sleep_for(timeout);
server_sock.cancel();

// Drive io_context to process cancellation callback
ctx.run();
reader.join();

Approach 3: Inject Stop Token into Sender Pipeline

Use write_env to inject an inplace_stop_token into the pipeline environment:

cpp
stdexec::inplace_stop_source stop_src;

auto server_task = run_server(ctx, acceptor)
  | ex::write_env(
      ex::prop{ex::get_stop_token, stop_src.get_token()});

// Trigger graceful shutdown from another thread
// stop_src.request_stop();
// ctx.stop();
